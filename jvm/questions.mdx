### 1、如何排查 OOM（面试口述版）

我一般会分三步说。

**第一步，先看报错里写的是哪块内存爆了。**  
通常来说，绝大多数 OOM 问题发生在**堆**和**方法区（元空间）**；程序计数器、虚拟机栈、本地方法栈一般不会 OOM——
栈溢出通常是 StackOverflowError，PC 和本地方法栈体量很小。所以报错里常见的就是 Java heap space、Metaspace、
再就是 Direct buffer memory等。先根据报错把范围缩到“是堆还是元空间还是直接内存”，再往下走。

**第二步，尽量拿到堆 dump。**  
线上建议提前在 JVM 参数里加上 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump.hprof`，
这样一旦 OOM 会生成一个.hprof 文件。如果没配，事后可以用 `jmap -dump:format=b,file=xxx.hprof &lt;pid&gt;` 打一份，
但 jmap 会 STW（Stop-The-World，即暂停所有应用线程），线上要慎用。 拿到 .hprof 之后，用 **JProfiler** 或 **VisualVM** 打开都行：JProfiler 功能更全，很多团队在用；
VisualVM 是 JDK 自带的，免费。打开后看“对象占用排行”和“谁在引用这些对象”，也就是看 Dominator Tree、GC Roots 引用链，
找出是哪些对象把内存撑爆的、是谁持有着没放。

**第三步，根据分析结果下结论。**  

结论分两类，每类里再区分不同情况，对应不同解法。

**一、内存泄漏**  
定义：对象用完了本该被 GC 回收，但被某处长期引用着不释放，堆里越积越多直到 OOM。  
- **典型情况**：静态 Map/List 一直往里塞没清理；ThreadLocal 用完没 remove；本地缓存没有容量上限或过期；监听器、回调注册了没反注册；非静态内部类持有外部类导致大对象无法回收。  
- **对应方案**：从 dump 里看是谁在引用这些对象（GC Roots 引用链），回到代码找“为什么没释放”，改引用关系、在合适时机 remove/clear、或给缓存加容量/过期策略。**不能靠调大堆解决**，调大了只是延后 OOM，对象会一直涨。

**二、容量或峰值问题（不是泄漏，但内存不够）**  
这里要再分几种情况，对症处理：

| 情况 | 怎么判断 | 对应方案 |
|------|----------|----------|
| **堆/元空间本身设小了** | 业务量、类数量本来就大，平时负载下堆/元空间就经常接近满，GC 日志里老年代或 Metaspace 长期高位。 | **评估后调大堆**（-Xmx、-Xms）**或元空间**（-XX:MaxMetaspaceSize）。机器内存允许、且确认没有泄漏时，可以调大。 |
| **瞬时流量高** | OOM 前一刻 QPS 突增、或接口变慢导致请求堆积，短时间内大量请求同时存活，对象暴增。平时 QPS 正常时没问题。 | **一般不优先调大堆**。优先**限流**（保护单机不被打爆）、**扩容实例**（多机分担）、或配合**熔断/降级**。若只是偶发尖峰、机器内存有余量，可以适当调大堆做缓冲，但根本还是要限流和容量规划。 |
| **单次请求创建对象过多** | dump 里看到某次请求对应的对象特别多（如一次查了超大结果集、大列表没分页、一次性加载大量数据到内存）。 | **不要一味调大堆**。优先**改业务**：分页、流式处理、按需加载、减少单次创建对象数量。业务优化后若仍接近上限，再考虑适当调大堆。 |
| **元空间不足（类太多）** | 报 Metaspace OOM，dump 或类加载监控里看到类数量很多（如大量动态代理、反射生成类、多 classloader 重复加载）。 | 能减少类加载的先**优化**（复用类、收拢 classloader）；必要时**调大 -XX:MaxMetaspaceSize**。 |

总结：**先看是不是泄漏**（有长期不该有的引用就修代码）；**不是泄漏再看是哪种容量/峰值**——常态就吃满就调大堆/元空间；瞬时尖峰就限流+扩容为主；单次请求对象太多就改业务逻辑为主，再视情况调堆。

最后，光看 dump 容易误判，要结合 **GC 日志**（Full GC 是否频繁、老年代是否一直满）和 **当时 QPS、接口耗时**（有无流量尖峰、请求是否堆积）一起看，才能区分是泄漏、常态容量不足，还是瞬时/单次请求导致的峰值。

**追问：线上 OOM 了又没有 dump 文件怎么办？（面试口述版）**  
- **进程已经挂了**：没法再对当时现场打 dump，只能事后分析。  
- **进程还在**（例如 OOM 后 JVM 没退出、或重启了想对同机其他实例防患）：可以尝试 `jmap -dump:format=b,file=xxx.hprof <pid>` 打一份当前堆，但 **jmap 会 STW**，线上要评估是否敢用、或选低峰/从节点打。  
- **没有 dump 时只能靠其他线索**：  
  1）**GC 日志**：看 OOM 前 Full GC 是否很频繁、老年代/元空间是否一直居高不下，判断是“慢慢涨上去的泄漏”还是“瞬间撑爆”。  
  2）**监控 / APM**：看堆内存曲线、QPS、接口耗时，是否有流量尖峰、请求堆积、某接口突然变慢导致对象堆积。  
  3）**最近发布与配置**：是否刚上线新功能、新缓存、调小过堆或改过 GC 参数。  
  4）**代码与业务**：结合报错栈（若有）和业务逻辑，排查是否大对象、大查询、无界集合、ThreadLocal/静态集合未释放等。  
- **事后必做**：给该应用补上 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...`，下次 OOM 自动留 dump，避免再抓瞎。

**面试一句话：** 进程挂了就没了；进程还在可谨慎用 jmap 打 dump（有 STW）。没有 dump 就靠 GC 日志、监控、APM、最近发布和代码推断；事后一定配上 OOM 时自动 dump，下次才好分析。

栈溢出是**虚拟机栈**空间不够用：每次方法调用会压一个栈帧（局部变量、返回地址等），递归太深或单次调用链太长，栈帧把栈占满就抛 StackOverflowError。

**常见原因与对应方案：**

| 原因 | 说明 | 解决思路 |
|------|------|----------|
| **递归层数过深** | 递归没有收敛条件或数据量大使递归层数超过栈深度。 | **优先改代码**：改成循环/迭代实现，或减少单次递归“层数”；若业务上递归深度确实大且无法避免，再考虑调大栈。 |
| **栈深度默认偏小** | 每个线程的栈大小由 `-Xss` 控制（如默认约 1MB），线程多或方法里局部变量多时容易占满。 | **适当调大 -Xss**（如 `-Xss2m`），注意栈变大后单线程占用内存增加，线程数多时总内存会上去。 |
| **方法内局部变量过多/过大** | 单个栈帧很大（如大数组、大对象在方法内声明），同样深度下更容易占满栈。 | 减少方法内过大的局部变量；能放堆里的放堆里；或拆成更小的方法。 |
| **相互递归 / 循环调用** | A 调 B、B 调 A（或长调用链），没有出口，栈帧只增不减。 | 检查调用链，避免无出口的相互调用；加递归边界或改为非递归实现。 |

**排查步骤（口述）：** 1）看异常栈：哪一行、哪个方法在重复出现，就是递归或循环调用的入口。2）看是否递归/长调用链导致，能改代码就改成迭代或减深度。3）确认无法减深度时再调大 `-Xss`，并评估线程数 × 栈大小对总内存的影响。

**面试一句话：** 栈溢出先看调用栈找递归或长调用链，优先改代码（改递归为循环、减深度、减栈帧大小），必要时再调大 `-Xss`。

### 3、JVM 由哪些部分组成？运行时内存区有哪些？

JVM 主要包括：

**1）类加载子系统（ClassLoader）** 把 .class 文件加载进内存，变成 JVM 能用的“类信息”。

**2）运行时数据区（Runtime Data Area）** 程序运行时用到的内存区域。

**3）执行引擎（Execution Engine）** 真正执行字节码：

- 解释执行（逐条解释）
- JIT 编译（热点代码编译成机器码更快）
- GC（垃圾回收也属于 JVM 运行时的重要机制）

**4）本地方法接口 JNI + 本地方法库** 让 Java 能调用 C/C++ 的代码（比如一些底层库）。

**运行时数据区：“3 个线程私有 + 2 个线程共享”**

- **线程私有：**
  - 1）程序计数器（PC）：记录当前线程执行到哪一行字节码了。线程切换回来能接着跑。
  - 2）虚拟机栈（Java Stack）：每次调用方法都会压一个栈帧，里面放局部变量和返回信息。常见错误：StackOverflowError（递归太深）。
  - 3）本地方法栈（Native Stack）：给 JNI 这种调用 C/C++ 的本地方法用的栈，了解即可。

- **线程共享：**
  - 4）堆（Heap）：存对象的地方，new 出来的对象基本都在堆里。常见错误：OutOfMemoryError: Java heap space。
  - 5）方法区（Method Area）：存“类相关的信息”：类元数据、常量池、静态变量等。JDK8 以后方法区的实现主要是元空间 Metaspace（用本地内存）。

### 4、堆的详细结构、新生代/老年代/方法区、YGC 与 FGC

**标题小结：** 堆 = 新生代（Eden + 两个 Survivor）+ 老年代；方法区（JDK8 用元空间实现，不在堆内）。YGC 只收新生代，FGC 收整堆（+ 元空间），触发时机不同。

结构示意（堆 = 新生代 + 老年代；元空间在堆外）：

![JVM 堆与内存结构：新生代 Eden/S0/S1、老年代、元空间；YGC 与 FGC 触发条件](/assets/Generated_image.png)

**1）堆里具体长什么样**

- **堆**在逻辑上分两块：**新生代**（Young Gen）和**老年代**（Old / Tenured Gen）。对象优先在新生代分配，熬过多次 GC 的会进老年代。
- **新生代**再细分为：
  - **伊甸园区**（Eden）：新对象先在这里分配，满了就触发一次 Young GC。
  - **两个幸存者区**（Survivor，常叫 S0、S1 或 From/To）：Eden 里存活的对象会复制到其中一个 Survivor，两块轮流做“来源”和“目标”，多次在 Survivor 间复制仍存活的对象会晋升到老年代。
      默认比例一般是 Eden : S0 : S1 = 8 : 1 : 1（-XX:SurvivorRatio=8）。
- **老年代**：存长期存活的对象，空间比新生代大，回收频率低，但一旦回收（Full GC）耗时更长。
- **方法区（元空间）**：JDK 8 里方法区用**元空间**实现，在**本地内存**里，**不在堆里**。只是面试常和“堆、GC”一起问，所以放这里一起记：堆管对象，方法区管类元数据等。

**方法区、持久代、元空间是一回事吗？**  
- **方法区**是 JVM 规范里的**逻辑名字**，规定“要存类元信息、常量、静态变量等”，**一直都有**。运行时数据区里说的“两个线程共享的：堆 + 方法区”，这里的**方法区就是规范里的这块区域**，和上面说的是同一个概念。  
- **持久代（PermGen）**：JDK 7 及以前，HotSpot 把方法区**实现在堆里**，并取名叫“永久代”。所以那时：方法区（规范）= 持久代（实现），且在堆中，受 `-XX:MaxPermSize` 等限制，容易 PermGen OOM。  
- **元空间（Metaspace）**：JDK 8 起，HotSpot **去掉了堆里的持久代**，改用**本地内存**里的元空间实现方法区。所以现在：方法区（规范）= 元空间（实现），**不在堆里**，用 `-XX:MaxMetaspaceSize` 等控制。  
- **一句话**：方法区是“叫什么、存什么”；持久代是“1.8 以前在堆里怎么实现”；元空间是“1.8 以后在堆外怎么实现”。规范上的方法区只有一个，实现从“堆里的持久代”换成了“堆外的元空间”。

**2）YGC 和 FGC 分别是什么、什么时候触发**

- **YGC（Young GC / Minor GC）**  
  - **是什么**：只回收**新生代**（Eden + Survivor）的垃圾，一般用复制算法，速度较快。  
  - **触发时机**：新生代不够用时，通常是 **Eden 区满了**再分配新对象时触发。  
  - 特点：频率高、停顿短（相对 FGC）。

- **FGC（Full GC / Major GC）**  
  - **是什么**：对**整堆**做回收，有的实现还会同时回收**方法区（元空间）**，停顿时间明显更长。  
  - **触发时机**：老年代满了、方法区（元空间）不足、主动调用 System.gc()（不保证立刻执行）、或某些 GC 策略下根据晋升失败等条件触发。  
  - 特点：频率低、停顿长，线上要尽量避免频繁 Full GC。

**一句话**：堆 = 新生代（Eden + 2 个 Survivor）+ 老年代；YGC 只收新生代、Eden 满时触发；FGC 收整堆（及元空间）、老年代/元空间满或策略触发，要少发生。

### 5、对象创建时内存怎么分配？何时回收？如何判断对象已死？（面试口述版）

**1）创建一个对象，内存是怎么分配的？**  
- **普通对象**：在**堆**上分配。绝大多数先尝试在**新生代 Eden 区**分配；若开启了 **TLAB**（Thread Local Allocation Buffer），会先在当前线程的 TLAB 里分配一小块，减少多线程争用。  
- **大对象**：若对象很大（如很长的数组），可能**直接进老年代**，避免在 Eden 和两个 Survivor 之间来回复制；具体阈值由 `-XX:PretenureSizeThreshold` 等控制（部分 GC 下生效）。  
- **流程简述**：类加载检查 → 在堆上划分内存（指针碰撞或空闲列表，取决于堆是否规整）→ 初始化零值 → 设置对象头（哈希码、分代年龄、锁信息等）→ 执行 `<init>`。  
**面试一句话**：对象在堆里分配，优先 Eden，可能用 TLAB；大对象可直进老年代；分配方式有指针碰撞或空闲列表。

**2）什么时候进行对象回收？**  
- **Young GC**：当**新生代（主要是 Eden）空间不足**时，再分配新对象会触发 YGC，只回收新生代。  
- **Full GC**：当**老年代满**、**元空间不足**、或显式调用 `System.gc()`（不保证立即执行）、以及某些策略下晋升失败等，会触发 FGC，回收整堆（及元空间）。  
- 所以“什么时候回收”可以答：**需要分配内存却发现不够时**会触发 GC（YGC 或 FGC），或由 JVM 根据策略自动选择时机。  
**面试一句话**：Eden 不够触发 YGC，老年代/元空间不够或策略触发 FGC；本质是“要分配但空间不足时”或 JVM 自动调度。

**3）如何判断对象已死？**  
- **可达性分析（主流）**：从一组 **GC Roots** 出发，向下遍历引用链，能走到的对象是“存活”，走不到的就是“已死”，可回收。  
- **GC Roots 常见包括**：虚拟机栈里局部变量表引用的对象、方法区静态变量引用的对象、方法区常量引用的对象、本地方法栈 JNI 引用的对象、被 synchronized 锁住的对象等。  
- **引用计数法（Java 不用）**：给对象记“被引用次数”，为 0 就回收。Java 不用是因为**循环引用**无法回收（A 引用 B，B 引用 A，计数都不为 0 但已经没有被外界引用）。  
**面试一句话**：用可达性分析，从 GC Roots 往下走，走不到的就是死对象；引用计数法有循环引用问题，Java 不用。

### 6、双亲委派机制（面试口述版）

当面试官问“你了解 JVM 的双亲委派吗”，可以这样答：

**是什么：**  
类加载器在加载一个类时，不会自己先加载，而是先把请求**委派给父加载器**；父加载器再委派给它的父加载器，一直到顶层的 **Bootstrap 类加载器**。如果父加载器能加载，就由父加载；只有当父加载器找不到时，子加载器才会尝试自己加载。

**层次关系（从高到低）：**  
- **Bootstrap ClassLoader**：加载 JDK 核心类（如 `java.lang.*`），用 C++ 实现，没有 Java 里的“父”。  
- **Extension ClassLoader**：加载 `ext` 目录下的扩展类。  
- **Application ClassLoader**：加载 classpath 下的应用类，我们写的类一般由它加载。  
- **自定义 ClassLoader**：可继承 `ClassLoader` 实现，父加载器一般为 Application。

**为什么要双亲委派：**  
1）**避免重复加载**：同一个类只会被同一个加载器加载一次。  
2）**保护核心类库**：像 `java.lang.String` 等由 Bootstrap 加载，避免被应用层替换或篡改，保证安全。

一句话总结：先问爹、爹不行再自己来；保证核心类统一由顶层加载，既防重复又保安全。

### 7、方法区在 JDK 1.7 和 1.8 的区别

JDK7 以前“类的信息”放在 JVM 里的一个固定区域（永久代），JDK8 把这个固定区域删了，改成用操作系统的本地内存（元空间），不容易因为类太多把 JVM 撑爆。

- 1.7：方法区主要落在永久代（PermGen），受 JVM 堆内存参数影响，容易 PermGen OOM。  
- 1.8：永久代移除，改为元空间（Metaspace），使用本地内存，参数主要是 -XX:MaxMetaspaceSize。

### 8、Java 8 与之前版本 JVM 的主要更新

面试时问到“Java 8 和之前 JVM 有啥变化”，可以按下面几条说；和上一问重叠的“永久代→元空间”简要带过即可。

**1）永久代移除，改为元空间（JDK 8 最大变化）**  
- 之前：类元数据、静态变量、常量池等放在**永久代（PermGen）**，在堆上划一块，受 `-XX:MaxPermSize` 限制，类/常量一多就 PermGen OOM。  
- Java 8：永久代去掉，改为**元空间（Metaspace）**，用**本地内存**存类元数据，不再占堆，由 `-XX:MaxMetaspaceSize` 控制上限。好处是减少堆压力、不易因类多就 OOM，但要注意 Metaspace 会吃 OS 内存。

**2）方法区、永久代、元空间——先分清这三个词**  
- **方法区**（Method Area）：JVM 规范里的**逻辑概念**，指“存类元数据、运行时常量池、静态变量等”的那块区域，规范只说了“要有这么一块”，没说具体用哪块内存实现。  
- **永久代**（PermGen）和 **元空间**（Metaspace）：是 **HotSpot 对方法区的两种实现**。JDK 7 及以前，方法区用永久代实现（在堆里划一块）；JDK 8 把永久代去掉，方法区改用**元空间**实现（用本地内存）。  
所以：**方法区（元空间）** = 同一件事的两种说法——方法区是“名字”，元空间是“JDK 8 里方法区的实现”，不是两个东西。类比：方法区像“接口”，永久代/元空间像“实现类”；JDK 8 里方法区由元空间实现，所以说“方法区在元空间”或“方法区（元空间）”都是一回事。

**3）字符串常量池 vs 运行时常量池（JDK 7 已改，8 延续）**  
- **运行时常量池**：在**方法区**里（JDK 8 就是元空间里），存类文件常量、符号引用等，没有挪到堆。  
- **字符串常量池**：JDK 6 在永久代；JDK 7 起挪到**堆**，Java 8 延续。所以 Java 8 里“方法区（元空间）里的常量池”指的是**运行时常量池**；“挪到堆”的只是**字符串常量池**。

一句话：Java 8 相对之前最大的 JVM 变化是**永久代→元空间**；再带上“方法区用元空间实现”“字符串常量池在 7 就进堆了、8 延续”，就够面试用了。

### 9、JVM 如何做调优？常用的调优参数有哪些？（面试口述版）

**先明确有哪些常见问题，再针对每类问题说怎么调。**

**一、常见问题有哪些？**

| 问题 | 表现 |
|------|------|
| **OOM（堆）** | 报 Java heap space，堆用满。可能是泄漏（持续涨不回落），也可能是容量/峰值（业务大或瞬时对象多）。 |
| **OOM（元空间）** | 报 Metaspace，类太多、反射/动态代理多。 |
| **OOM（直接内存等）** | Direct buffer memory 等，NIO/Netty 用多。 |
| **GC 停顿大、频繁** | YGC/FGC 次数多，或单次停顿时间长，接口 P99 抖、卡顿。 |
| **吞吐低** | GC 占 CPU 比例高，业务算力被 GC 吃掉。 |
| **栈溢出** | StackOverflowError，递归太深或栈帧过大。 |
| **内存泄漏** | 堆/元空间持续上涨不回落，最终 OOM，调大堆只能延后。 |

**二、针对每类问题如何调优？**

| 问题 | 怎么调优 |
|------|----------|
| **OOM 堆** | **先区分泄漏还是容量**：看 dump、GC 日志、内存曲线。泄漏 → 找引用链、修代码（去静态/ThreadLocal/缓存无界等），**不能靠调大堆**。容量/峰值 → 评估后调大 -Xmx/-Xms；若是瞬时尖峰则优先限流、扩容、改业务（分页/流式），再视情况调堆。 |
| **OOM 元空间** | 优化类加载（少动态生成类、收拢 classloader）；必要时**调大 -XX:MaxMetaspaceSize**。 |
| **OOM 直接内存** | 调大 -XX:MaxDirectMemorySize；检查堆外是否有泄漏、未释放。 |
| **GC 停顿大/频繁** | **选低延迟 GC**：如 -XX:+UseG1GC 配 -XX:MaxGCPauseMillis=200，或 -XX:+UseZGC。**调堆结构**：年轻代过大则 YGC 慢，过小则对象容易进老年代、FGC 多，可调 -Xmn、SurvivorRatio。堆整体偏小也会导致 GC 频繁，可适当调大 -Xmx。 |
| **吞吐低** | 选高吞吐 GC（如 -XX:+UseParallelGC）；调大堆减少 GC 次数；调 GC 线程数（Parallel 相关参数）。减少单次请求对象量、避免大对象。 |
| **栈溢出** | **优先改代码**：递归改迭代、减递归深度、减栈帧大小。不行再**调大 -Xss**（注意线程数×栈大小）。 |
| **内存泄漏** | **必须修代码**：dump 分析 GC Roots 引用链，找到谁持有着不释放，改引用、remove/clear、加过期或容量上限。调参只能延后 OOM，不能根治。 |

**流程一句话：** 先通过日志、监控、dump 确定是上面哪类问题，再按表里“怎么调优”去做；调完观察，不行再迭代。  
**背诵口诀：** 先问题（OOM/停顿/吞吐/栈溢出/泄漏），再对症（泄漏修代码、容量调堆、停顿选 GC 调堆、栈溢出改代码或 -Xss）。

---

**常用的调优参数有哪些？（按类记）**

**1）堆内存**  
- `-Xms`、`-Xmx`：堆初始大小和最大大小，一般设成一样避免运行中扩容抖动，如 `-Xms4g -Xmx4g`。  
- `-Xmn`：年轻代大小（有的版本用 `-XX:NewSize` / `-XX:MaxNewSize`）。  
- `-XX:SurvivorRatio=8`：Eden 与一个 Survivor 的比例，默认 8:1:1（两个 Survivor）。  
- `-XX:NewRatio`：老年代与年轻代的比值，默认 2（老年代约为年轻代 2 倍）。

**2）元空间（JDK8+）**  
- `-XX:MetaspaceSize`、`-XX:MaxMetaspaceSize`：元空间初始/上限。类多、反射多的应用可适当调大，避免 Metaspace OOM。

**3）GC 选择与停顿**  
- `-XX:+UseG1GC`：启用 G1，一般场景常用，可配合 `-XX:MaxGCPauseMillis=200` 设期望最大停顿时间。  
- 低延迟场景：`-XX:+UseZGC`（JDK15+ 生产可用）或 `-XX:+UseShenandoahGC`。  
- 吞吐优先：`-XX:+UseParallelGC`（并行收集器）。

**4）OOM 时自动 dump**  
- `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump.hprof`：OOM 时自动打堆快照，方便事后分析。

**5）GC 日志（便于排查）**  
- JDK9+：`-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=100m`。  
- JDK8：`-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log`。

**6）其它常见**  
- `-Xss`：线程栈大小，默认一般够用，线程极多时可适当调小。  
- `-XX:+UseStringDeduplication`（G1）：字符串去重，减少重复字符串占用。  
- `-XX:AutoBoxCacheMax`：Integer 等包装类缓存上限（默认 -128～127），极少需要调。

一句话：堆用 -Xms/-Xmx/-Xmn，元空间用 MetaspaceSize/MaxMetaspaceSize，GC 用 G1 或 ZGC，再配上 OOM dump 和 GC 日志，就覆盖大部分面试场景了。

### 10、GC 算法（标记-清除、标记-整理、复制、分代收集）

常见有三种基础算法，再结合分代收集在新生代、老年代里搭配使用。

**1）标记-清除（Mark-Sweep）**  
- **过程**：先从头遍历，把存活对象“标记”出来；再统一把未标记的对象“清除”掉，释放内存。  
- **缺点**：产生内存碎片，大对象可能找不到连续空间；清除时可能产生停顿。  
- 老年代常用“标记-清除”或在其基础上改进的“标记-整理”。

**2）标记-整理（Mark-Compact）**  
- **过程**：先标记存活对象，再把存活对象往一端“挪动”，让它们紧凑排列，最后清理边界外的空间。  
- **优点**：没有碎片，分配大对象时更稳定。  
- **缺点**：移动对象有成本，且要更新引用，一般停顿比“只清除”要长。  
- 老年代里对停顿敏感时，有的回收器会用“标记-清除”少移动；对碎片敏感时用“标记-整理”。

**3）复制算法（Copying）**  
- **过程**：把内存分成两块（如 Survivor 的 From / To），只在一块上分配；GC 时把存活对象复制到另一块，再一次性清空当前块。  
- **优点**：实现简单、没有碎片、回收效率高。  
- **缺点**：要浪费一半空间（只能有一半用来放对象）。  
- **适用**：对象“朝生夕死”多的区域最合适，所以**新生代**常用复制算法（Eden + 两个 Survivor 就是一块用、一块备用）。

**4）分代收集（Generational）**  
- **思路**：不同年龄的对象用不同策略。新生代里大部分对象很快死掉，用**复制算法**，只复制少量存活对象；老年代里对象存活久，用**标记-清除**或**标记-整理**，避免频繁复制。  
- **对应关系**：新生代（Eden + Survivor）→ 复制算法；老年代 → 标记-清除或标记-整理（具体看用的是哪种回收器，如 CMS 偏标记-清除，Parallel 老年代用标记-整理）。

一句话：三种基础算法——标记-清除（有碎片）、标记-整理（无碎片、要移动）、复制（无碎片、占一半空间）；分代下新生代用复制、老年代用标记-清除或标记-整理。

### 11、JMM（Java 内存模型）是什么？

**JMM 是什么？**  
JMM，即Java 内存模型。它是 Java 的**并发下的内存规范**，规范里抽象出主内存、工作内存，以及两者之间的**8 种原子操作**
（read、load、use、assign、store、write、lock、unlock）的**执行顺序与约束** 。实现多线程下的**可见性**和**有序性**。

**常说的几个点（面试口述用）**  
- **主内存与工作内存**：JMM 里抽象出“主内存”（可理解为堆里共享变量）和每个线程各自的“工作内存”（可理解为缓存/寄存器里的副本）。线程读变量从主内存读到工作内存，写回去再写回主内存；不同线程之间要通过主内存来“交换”可见性。  
- **可见性**：一个线程改了共享变量，另一个线程不一定马上看到（可能一直在用自己工作内存里的旧值）。synchronized、volatile、final 等能提供一定可见性保证。  
- **有序性**：编译器、CPU 可能对指令重排序，单线程没问题，多线程下可能出问题。JMM 用 **happens-before** 规则规定哪些操作必须“先于”哪些；volatile、synchronized、锁的释放/获取都会参与 happens-before，从而限制重排序、保证可见性。

**volatile、synchronized 如何实现可见性和有序性（说清楚版）**

- **volatile**  
  - **可见性**：写 volatile 时，会在写后插入**写屏障**（如 store-load 屏障），把工作内存的修改**刷回主内存**；读 volatile 时，会在读前插入**读屏障**，从主内存读（或使本地缓存失效）。这样其他线程再读该变量时能看到最新写。  
  - **有序性**：volatile 读/写两侧会加**内存屏障**，规定：不能把 volatile 写之前的普通写重排到 volatile 写之后，不能把 volatile 读之后的普通读重排到 volatile 读之前；且 volatile 写与 volatile 读之间形成 **happens-before**，写对读可见。所以“禁止指令重排序”是通过屏障实现的。  

- **synchronized**  
  - **可见性**：**解锁（unlock）** 会带“释放”语义，把当前线程在同步块里对共享变量的修改**刷回主内存**；**加锁（lock）** 会带“获取”语义，相当于从主内存重新加载，能看到上一个线程解锁前写的内容。JMM 规定：**同一把锁的 unlock happens-before 后续的 lock**，所以块内的修改对下一个拿到锁的线程可见。  
  - **有序性**：因为“unlock 先于 下一个 lock”，编译器/CPU 不能把同步块**内部**的写重排到 **unlock 之后**，也不能把**块外**的读重排到 **lock 之前**，相当于以加锁/解锁为边界限制了重排序，保证块内逻辑对下一个线程来说“像按顺序执行的一样”。
