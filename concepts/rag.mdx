## 面试背诵版（放最前，方便直接说）

 RAG 翻译过来就是检索、增强、生成。  
 主要分两个阶段：  
 第一是**离线建库**：把文本分段，用 Embedding 把每段转成向量（一维数组），存进向量库。  
 第二是**在线问答**：用户每次发来的内容都会走 RAG——先用问题在向量库里做 top-k 检索，拿到最相关的几段文本，把这些段和用户问题拼成一条 prompt，再交给模型生成回答。所以检索、增强、生成这三步都发生在每次问答的时候。

---

## RAG 叫什么？

**RAG** 是 **Retrieval-Augmented Generation** 的缩写，中文常叫 **检索增强生成**。  
三个词对应的是**每次问答时**的三步，不是“建库 + 问答”两阶段：**Retrieval（检索）** = 从向量库查出和问题最相关的几段原文；**Augmented（增强）** = 把这几段原文拼进 Prompt，增强输入；**Generation（生成）** = 大模型根据增强后的 Prompt 生成回答。所以“先检索、再增强、再生成”发生在**问答阶段**里。

---

## RAG 干了什么？

**两阶段**：

- **建库（离线，做一次或定期）**：把你的文档或数据变成“可被检索的知识库”。流程是：文档 → 切块（chunk）→ 用 Embedding 转成向量 → 存入向量库。  
  **MySQL 数据怎么变成知识库**：用 SQL 查出需要参与检索的列，把每一行转成一段描述性文本（例如“订单号 1001，金额 99 元，状态：已发货”），再对这些文本做切块（或整段）→ Embedding → 写入向量库；每条给一个稳定 id（如 `订单_1001`），方便后面按行更新。  
  **MySQL 变了怎么同步**：向量库和 MySQL 是两套存储，需要你主动同步。**全量**：定时或按需从 MySQL 拉全量数据 → 转文本、切块、Embedding → 清空或覆盖向量库。**增量**常见两种方式：（1）**改库时异步改 RAG**——业务在写 MySQL 的同时或之后（如发 MQ、调刷新接口），异步把本条或本批变更转成文本、算 Embedding，在向量库里按 id 更新（若向量库支持按 id 直接修改则直接改，否则先删该 id 再插入）；（2）**基于 binlog**——解析 MySQL binlog 得到哪些行变了，再根据变更内容去更新向量库里对应的 id（同样：支持则直接改，否则先删再加）。
- **问答（在线，每次用户提问）**：用户输入问题后，先把问题转成向量，在向量库里检索出 Top-K 段最相关的原文，把这几段和用户问题拼成一条 Prompt，再交给大模型生成回答。

也就是说：**每次用户输入都会做一次检索和增强**——用当前问题查知识库，把查到的片段和问题一起塞进同一条 Prompt，再发给大模型；模型看到的是“参考资料 + 问题”，所以能按你给的资料来答。

**拼接**：把「检索到的几段原文」和「用户问题」填进一个固定模板，拼成一段长文本，作为发给大模型的一条消息（如 `user` 的 `content`）。本质就是：**长文本 = 检索结果 + 问题 + 说明性指令**。

**实现形态**：RAG 是这套做法，可以写成脚本（本地调一次得到答案），也可以做成服务（如 Flask/FastAPI 提供 `POST /ask`，请求先走检索+增强再返回答案）。

## 核心概念

理解 RAG 时会反复遇到下面几个词，用一句话说明：

| 词 | 一句话 |
|----|--------|
| **向量** | 程序里就是一维数组，如 `[0.1, -0.3, 0.8, ...]`，一段文字对应一个向量；用“数字离得近不近”表示“意思像不像”，便于检索。 |
| **Embedding** | 把文字变成向量的过程（模型/接口）。语义相近的句子向量近、远的向量远；多语言模型下“你好”和“Hello”是两串数但距离近。 |
| **检索 Top-K** | 问题转向量 Q，和库里每个向量算相似度（如余弦相似度），按相似度排序取前 K 个，对应的原文即“最相关的几块”；一般交给向量库算。 |
| **向量库** | 存向量、支持按相似度查。**Chroma** 单机本地、免运维，入门够用；**Milvus** 分布式海量；**pgvector** 是 Postgres 扩展，向量和业务数据同库。 |

## 支持的数据格式与更新方式

**格式**：RAG 只认纯文本。PDF、Word、Markdown、TXT、HTML 需先解析出正文；CSV、Excel、数据库行可转成“列1: xx, 列2: yy”等描述文本；MySQL 可 SELECT 后每行转一段话，用 `表名_主键` 当向量 id 便于按行更新。

**更新**：向量库与数据源是分开的，数据变了要主动同步。**全量**：定时从数据源拉全量 → 转文本、切块、Embedding → 清空/覆盖向量库。**增量**两种常见方式：（1）改库时异步改 RAG（写 MySQL 后发 MQ 或调接口，异步更新向量库）；（2）基于 binlog 解析出变更，再更新向量库里对应的 id。单条更新时：向量库若支持按 id 直接修改则直接改，否则先删该 id 再插入。

## 现成开源 RAG 系统

不想从零写可用现成方案（多为开源、可自托管）：

| 名字 | 特点 | 适合 |
|------|------|------|
| **Dify** | 内置 RAG、可视化工作流、多模型、Docker 一键起 | 开箱即用、有 Web 界面、上传文档即问答 |
| **FastGPT** | 类似 Dify，中文友好 | 国内团队、要现成界面 |
| **LlamaIndex** / **LangChain** | 库，提供 RAG 链、Loader、向量库，需写代码组装 | 在应用里嵌入 RAG、要灵活定制 |
| **RAGFlow** | 针对复杂文档（表格、多格式）解析与检索 | 文档结构复杂、对解析质量要求高 |

## 小结

| 步骤 | 做什么 |
|------|--------|
| 建库 | 文档/数据 → 转文本 → 切块 → Embedding → 存向量库 |
| 问答 | 问题 → Embedding → 检索 Top-K → 拼 Prompt（参考资料+问题）→ 调大模型生成回答 |

**检索负责找资料，大模型根据资料生成回答**，两者通过拼好的 Prompt 组装在一起。
