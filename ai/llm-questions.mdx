## 一、激活函数

### (1) 神经网络中，激活函数层的作用是什么？

**一句背**：**线性叠再多还是线性，加了激活才有非线性。**

- **位置**：接在 **Wx + b** 这种线性层后面。
- **作用**：把结果**掰弯**——引入**非线性**，多层才能拟合复杂曲线和决策边界。
- **反过来说**：没有激活函数，多层网络数学上等价于**一层线性**，表达能力很弱。

---

### (2) Sigmoid 和 ReLU 的优缺点分别是什么？

**对比背**：**Sigmoid 像 S 形压进 0～1，两头平、梯度容易没；ReLU 负的变 0、正的不动，快但可能“死”。**

| | **Sigmoid** | **ReLU**（max(0,z)） |
|---|-------------|----------------------|
| **优点** | 输出 **0～1**，可当软概率；光滑可导 | **算得快**；正半轴梯度为 **1**，缓解两头饱和 |
| **缺点** | **两头饱和 → 梯度消失**；非零中心；有 **exp** 略贵 | **负半轴恒 0 → 神经元死亡**；输出非零中心 |

---

### (3) 根据上述缺点，可以如何改进激活函数？

**口诀**：**Sigmoid 愁消失，换 ReLU 族、加 Norm、上残差；ReLU 怕死神经元，泄漏、ELU、GELU 给负轴留活路。**

- **针对 Sigmoid**：多用 **ReLU 族**；**BatchNorm / LayerNorm** 稳分布；**残差** 帮梯度往回走；也可换 **Tanh**（仍要防饱和）。
- **针对 ReLU**：**Leaky ReLU / PReLU**（负轴小斜率）、**ELU**、**GELU**（Transformer 里常见）。
- **考场补一句**：按任务做消融，在**效果、稳定、算力**之间折中。

## 二、政务系统拟引入 DeepSeek

### (1) 设计一个多级审查系统以避免敏感信息泄露。

**一句背**：**进要分级、传要脱敏、跑要专网、出要扫描、疑要人工、全程可审计。**

**六级（背序号 + 关键词）**

1. **进**：内容**分级分类**（公开 / 内部 / 敏感 / 涉密）——**涉密不进公网大模型**。
2. **净**：**脱敏 + 最小必要**——PII、证件号等识别后替换/截断，只传该传的片段。
3. **跑**：**私有化或政务专网** + **隔离** + **API 网关**（鉴权、限流）。
4. **控**：**系统 Prompt / 策略** 禁止某类输出；高风险意图用**规则或分类器**先拦。
5. **出**：**DLP** 扫输出；命中则**阻断 / 替换 / 转人工**。
6. **人**：高敏或低置信 → **人工复核** 通过再落库/对外。
7. **账**：**全链路日志审计**（谁、何时、上下文、生成内容），满足留存与抽检。

---

### (2) 在这个场景下，采用微调和 Prompt Engineering 哪种方案更合适？说明理由。

**结论先背**：**政务场景多数先 Prompt（+RAG+护栏），合规数据到位再考虑微调。**

**理由四条（对称记）**

| 维度 | **Prompt Engineering 为主** | **微调（SFT/LoRA）** |
|---|---------------------------|----------------------|
| **速度** | 改提示、改检索，**快、好回滚** | 要数据、训练、评测，**周期长** |
| **数据** | 不强制大规模标注语料 | 要**脱敏、授权、可审计**的领域数据，**政务门槛高** |
| **合规** | 规则、检索片段、Prompt **好写进制度** | 权重变化 **难逐项解释**，审计成本高 |
| **适用** | 通用问答、检索增强、快速试点 | **文风、术语、格式**强一致，且**数据已审批** |

**收尾一句**：微调若要做，也必须和**审查、脱敏、专网**叠在一起，不能单押模型。

## 三、大模型规模与 GPU

### (1) 大模型的「大」体现在哪里？GPU 的什么特点决定了其更适合用于大模型的推理和训练？

**「大」四字记**：**参数量大、数据量大、算力消耗大、上下文（KV）占显存大。**

**GPU 为何合适——背四个词**

1. **并行**：矩阵乘、注意力 = 大量**独立小运算** → 贴合 GPU **众核**。
2. **带宽**：权重和激活**频繁大块读写** → GPU（尤其 **HBM**）**显存带宽高**。
3. **专用算力**：**Tensor Core** 等 → **混合精度**矩阵算得快。
4. **扩展**：训练常用**多卡多机**（数据并行、张量并行、流水线并行）。

**一句串**：**大模型吃并行和带宽，GPU 专干这个。**

---

### (2) 模型名称后常常会带一个 xxB，比如 llama3-70B，这里的 70B 是什么意思？

**背**：**B = Billion（十亿），70B ≈ 700 亿参数。**

- 指模型**可训练权重的规模**（量级；具体统计口径以官方为准）。
- **越大**：通常能力上限更高，但**显存、算力、延迟、费用**也更高。

---

## 四、Token、幻觉与 RAG

### (1) 大模型中的 Token 是什么？举例说明。

**一句背**：**Token = 模型眼里的“最小积木”，由分词器切出来，不等于人眼里的一个词。**

- **举例（英文）**：`unhappiness` 常被切成多个子词 token，不是整词一个 token。
- **举例（中文）**：可能**一字一个**，也可能**多字合并**，看具体 tokenizer（BPE、SentencePiece 等）。
- **举例（其它）**：标点、数字、空格也常单独或合并成 token。

---

### (2) 输入 Token 和输出 Token 分别是什么？

**对称背**

- **输入 Token**：**你喂给模型的一切**经分词后的序列——系统提示、多轮历史、用户话、**RAG 贴进来的文档片段**等。
- **输出 Token**：模型**自己生成**的 token，**一个一个往外蹦**，直到结束或到上限。

**计费提示**：产品里**输入/输出**可能分开计价或共用窗口上限，以**官方说明**为准。

---

### (3) 大模型的幻觉(Hallucination)是什么？RAG 中的 Retrieval 技术是什么？RAG 除了用于大模型的微调，还可以用来做什么？

**幻觉一句背**：**说得像真的，其实是编的——流畅自信但事实错。**

- **常见表现**：瞎编论文、法条、数据、人名机构等。
- **常提缓解**：**RAG**、工具/检索核验、人工审核、降温度、强约束提示。

**Retrieval 一句背**：**生成前先“翻书”——从知识库/向量库里把相关段落捞出来，再拼进 Prompt 里生成。**

**概念纠偏（必背）**：RAG 主流是**推理时**「检索 + 生成」，**不是**“微调”的代名词；说“除了微调还能干啥”时，先答：**本职是推理增强**。

**RAG 还能干啥（记场景）**：政务/企业**知识问答**、**客服**、**内部文档搜索与摘要**、**代码库问答**、**引用依据的合规答复**、**只更新知识库不重训大模型**。

---

## 五、上下文窗口

下列长度为常见量级，**以厂商最新文档为准**。

| 模型 / 系列 | 典型上下文窗口（约） |
|---|----------------------|
| GPT-4o | 128K tokens 量级 |
| Claude 3.5 Sonnet | 200K 量级 |
| Gemini 1.5 Pro | 1M+（长上下文版本） |
| Llama 3（常见开源权重） | 8K～128K（视变体与微调而定） |
| DeepSeek-V3 等国产商用 API | 以官方为准（常见 64K～128K 量级产品） |

---

### (1) 什么是上下文窗口？

**一句背**：**一次前向里，模型最多能一起“看见”的 token 总长（系统+历史+检索+用户+已生成等，是否合并计费看产品）。**

- **超出怎么办**：截断、摘要、滑动窗口等。

---

### (2) 上下文窗口长度过长或过短会对推理速度和推理质量有什么影响？解释原因。

**短窗口**

| | 影响 | **原因（背逻辑）** |
|---|------|-------------------|
| **质量** | 易**丢前文**、**胡编**风险升 | 长文档、多轮对话**挤不出去的就被忘掉**，信息不全 |
| **速度** | 通常**更快、更省** | 算子随序列变短，**KV Cache** 也小 |

**长窗口**

| | 影响 | **原因（背逻辑）** |
|---|------|-------------------|
| **速度** | **变慢、占显存** | 注意力等代价随长度涨（整体算力、**KV** 都涨） |
| **质量** | 信息多但可能**抓不住重点** | **注意力分散**、噪声多；检索差时**垃圾进、垃圾出** |

**收尾一句**：工程上靠 **RAG 精选片段、截断、摘要、重排序** 把“放进窗口的东西”管好。

## 一、激活函数

### (1) 神经网络中，激活函数层的作用是什么？

**背诵版**：激活函数把线性结果变成非线性，让多层网络能学复杂关系。

**关键词**：非线性；`Wx + b` 后；表达能力

**记忆口诀**：没激活=直线，有激活=弯曲。

**极短补充**：没有激活，多层等价于单层线性模型。

---

### (2) Sigmoid 和 ReLU 的优缺点分别是什么？

**背诵版**：Sigmoid 输出 0～1 的 S 形结果但两头易饱和导致梯度消失；ReLU 计算快但负半轴可能“死”。

**关键词**：Sigmoid=S 形；饱和消失；ReLU=快死

**Sigmoid 一句话**：优=软概率（0～1）；缺=两头饱和梯度小。

**ReLU 一句话**：优=正区梯度稳定、计算快；缺=负区输出 0，可能长期不更新。

**记忆口诀**：Sigmoid=压进壳，ReLU=过门变硬。

---

### (3) 根据上述缺点，可以如何改进激活函数？

**背诵版**：改进=“防消失 + 救死亡 + 稳训练”。

**关键词**：ReLU 族；Norm/残差；Leaky/GELU

- 防消失：更少饱和/更稳定（如 Tanh/GELU）+ `BatchNorm / LayerNorm` + 残差结构。
- 救死亡：Leaky ReLU / PReLU / ELU / GELU，让负半轴仍有梯度。
- 稳训练：结合合适初始化、学习率与训练技巧。

## 二、政务系统拟引入 DeepSeek

### (1) 设计一个多级审查系统以避免敏感信息泄露。

**背诵版**：分级进入、脱敏最小、专网运行、策略拦截、输出扫描、人审兜底、全程审计。

**关键词**：分级；脱敏；DLP+人工

```text

净：脱敏/最小化
跑：专网/私有化+网关
控：策略 Prompt/规则

疑：人工复核（高风险）
账：日志审计（留痕）
```

- 进：涉密不出网。
- 净：PII/敏感字段替换或截断。
- 跑：隔离部署与统一鉴权限流。
- 控：禁止高风险输出模式。
- 出：输出做 DLP + 语义/规则检测。
- 疑：低置信或高敏走人工。
- 账：全链路可追溯。

### (2) 在这个场景下，采用微调和 Prompt Engineering 哪种方案更合适？说明理由。

**背诵版**：政务场景先用 Prompt Engineering（+RAG+护栏），只有在数据合规且需要强一致文风/格式时才考虑微调。

**关键词**：先快后稳；合规数据；护栏+RAG

- 先做 Prompt：改提示词/策略/检索，迭代快、好回滚。
- 再考虑微调：微调依赖审批通过的脱敏领域数据，周期与审计成本更高。
- 即使微调：也要继续配套审查、脱敏与专网护栏。

**记忆口诀**：护栏先行，微调后补；不合规不训。

**关键词**：Billion；参数规模；更强也更贵

## 四、Token、幻觉与 RAG

### (1) 大模型中的Token是什么？举例说明。

**背诵版**：Token 是模型读写文本的最小单位，由 tokenizer 切分；“一个字/词”不一定等于一个 token。

**关键词**：最小单位；tokenizer；子词/合并

- 英文例子：`unhappiness` 常被切成多个子词 token。
- 中文例子：可能一字一 token，也可能多字合并（看具体分词策略）。

---

### (2) 输入Token 和输出Token 分别是什么？

**背诵版**：输入 token=你喂给模型的 prompt/历史/检索证据；输出 token=模型新生成的 token 序列（自回归逐 token 产出）。

**关键词**：输入=提示；输出=生成；自回归

---

### (3) 大模型的幻觉(Hallucination)是什么？RAG (Retrieval Augmented Generation) 中的 
Retrieval 技术是什么？RAG除了用于大模型的微调，还可以用来做什么？

**背诵版**：幻觉=编造事实；Retrieval=先检索证据再生成；RAG=检索增强，用“证据锚定”答案。

**关键词**：幻觉=事实错；Retrieval=搜证据；RAG=证据锚定

- 幻觉：说得像真的但内容不真实（编数据/编引用）。
- Retrieval：从知识库/向量库检索相关片段，拼进提示再生成。
- 不等同微调：RAG 常见含义是“推理时检索增强”，不是“训练时微调”的同义词。
- 还能做：企业/政务知识问答、内部文档搜索与摘要、客服/工单辅助、引用依据的合规答复、降低知识更新成本。

## 五、上下文窗口

### (1) 什么是上下文窗口？

**背诵版**：上下文窗口=一次前向里模型能同时“看见并计算”的最大 token 总长度（超出会截断/摘要）。

**关键词**：一次可见；token 上限；截断/摘要

---

### (2) 上下文窗口长度过长或过短会对推理速度和推理质量有什么影响？解释原因。

**背诵版**：短窗口=忘关键信息更易胡编；长窗口=更慢更耗显存，并可能注意力分散被噪声带偏。

**关键词**：短=丢；长=慢+噪；KV 显存

- 短窗口：长文/早期信息挤不进去 -> 上下文不全 -> 质量下降。
- 长窗口：注意力与 KV cache 随长度增长 -> 速度下降；噪声增多、注意力分散 -> 质量可能不升反降。

**工程解法**：RAG 精选+重排序+截断/摘要，控制“进窗口的有效信息”。

**一句背**：**进要分级、传要脱敏、跑要专网、出要扫描、疑要人工、全程可审计。**

**六级（背序号 + 关键词）**

1. **进**：内容**分级分类**（公开 / 内部 / 敏感 / 涉密）——**涉密不进公网大模型**。
2. **净**：**脱敏 + 最小必要**——PII、证件号等识别后替换/截断，只传该传的片段。
3. **跑**：**私有化或政务专网** + **隔离** + **API 网关**（鉴权、限流）。
4. **控**：**系统 Prompt / 策略** 禁止某类输出；高风险意图用**规则或分类器**先拦。
5. **出**：**DLP** 扫输出；命中则**阻断 / 替换 / 转人工**。
6. **人**：高敏或低置信 → **人工复核** 通过再落库/对外。
7. **账**：**全链路日志审计**（谁、何时、上下文、生成内容），满足留存与抽检。

---

### (2) 在这个场景下，采用微调和 Prompt Engineering 哪种方案更合适？说明理由。

**结论先背**：**政务场景多数先 Prompt（+RAG+护栏），合规数据到位再考虑微调。**

**理由四条（对称记）**

| 维度 | **Prompt Engineering 为主** | **微调（SFT/LoRA）** |
|------|---------------------------|----------------------|
| **速度** | 改提示、改检索，**快、好回滚** | 要数据、训练、评测，**周期长** |
| **数据** | 不强制大规模标注语料 | 要**脱敏、授权、可审计**的领域数据，**政务门槛高** |
| **合规** | 规则、检索片段、Prompt **好写进制度** | 权重变化 **难逐项解释**，审计成本高 |
| **适用** | 通用问答、检索增强、快速试点 | **文风、术语、格式**强一致，且**数据已审批** |

**收尾一句**：微调若要做，也必须和**审查、脱敏、专网**叠在一起，不能单押模型。

## 三、大模型规模与 GPU

### (1) 大模型的「大」体现在哪里？GPU 的什么特点决定了其更适合用于大模型的推理和训练？

**「大」四字记**：**参数量大、数据量大、算力消耗大、上下文（KV）占显存大。**

**GPU 为何合适——背四个词**

1. **并行**：矩阵乘、注意力 = 大量**独立小运算** → 贴合 GPU **众核**。
2. **带宽**：权重和激活**频繁大块读写** → GPU（尤其 **HBM**）**显存带宽高**。
3. **专用算力**：**Tensor Core** 等 → **混合精度**矩阵算得快。
4. **扩展**：训练常用**多卡多机**（数据并行、张量并行、流水线并行）。

**一句串**：**大模型吃并行和带宽，GPU 专干这个。**

---

### (2) 模型名称后常常会带一个 xxB，比如 llama3-70B，这里的 70B 是什么意思？

**背**：**B = Billion（十亿），70B ≈ 700 亿参数。**

- 指模型**可训练权重的规模**（量级；具体统计口径以官方为准）。
- **越大**：通常能力上限更高，但**显存、算力、延迟、费用**也更高。

## 四、Token、幻觉与 RAG

### (1) 大模型中的 Token 是什么？举例说明。

**一句背**：**Token = 模型眼里的“最小积木”，由分词器切出来，不等于人眼里的一个词。**

- **举例（英文）**：`unhappiness` 常被切成多段子词，不是整词一个 token。
- **举例（中文）**：可能**一字一个**，也可能**多字合并**，看具体 tokenizer（BPE、SentencePiece 等）。
- **举例（其它）**：标点、数字、空格也常单独或合并成 token。

---

### (2) 输入 Token 和输出 Token 分别是什么？

**对称背**

- **输入 Token**：**你喂给模型的一切**经分词后的序列——系统提示、多轮历史、用户话、**RAG 贴进来的文档片段**等。
- **输出 Token**：模型**自己生成**的 token，**一个一个往外蹦**，直到结束或到上限。

**计费提示**：产品里**输入/输出**可能分开计价或共用窗口上限，以**官方说明**为准。

---

### (3) 大模型的幻觉(Hallucination)是什么？RAG 中的 Retrieval 技术是什么？RAG 除了用于大模型的微调，还可以用来做什么？

**幻觉一句背**：**说得像真的，其实是编的——流畅自信但事实错。**

- **常见表现**：瞎编论文、法条、数据、人名机构等。
- **常提缓解**：**RAG**、工具/检索核验、人工审核、降温度、强约束提示。

**Retrieval 一句背**：**生成前先“翻书”——从知识库/向量库里把相关段落捞出来，再拼进 Prompt 里生成。**

**概念纠偏（必背）**：RAG 主流是**推理时**「检索 + 生成」，**不是**“微调”的代名词；说“除了微调还能干啥”时，先答：**本职是推理增强**。

**RAG 还能干啥（记场景）**：政务/企业**知识问答**、**客服**、**内部文档搜索与摘要**、**代码库问答**、**引用依据的合规答复**、**只更新知识库不重训大模型**。

## 五、上下文窗口

下列长度为常见量级，**以厂商最新文档为准**。

| 模型 / 系列 | 典型上下文窗口（约） |
|-------------|----------------------|
| GPT-4o | 128K tokens 量级 |
| Claude 3.5 Sonnet | 200K 量级 |
| Gemini 1.5 Pro | 1M+（长上下文版本） |
| Llama 3（常见开源权重） | 8K～128K（视变体与微调而定） |
| DeepSeek-V3 等国产商用 API | 以官方为准（常见 64K～128K 量级产品） |

### (1) 什么是上下文窗口？

**一句背**：**一次前向里，模型最多能一起“看见”的 token 总长（系统+历史+检索+用户+已生成等，是否合并计费看产品）。**

- **超出怎么办**：截断、摘要、滑动窗口等。

---

### (2) 上下文窗口长度过长或过短会对推理速度和推理质量有什么影响？解释原因。

**短窗口**

| | 影响 | **原因（背逻辑）** |
|---|------|-------------------|
| **质量** | 易**丢前文**、**胡编**风险升 | 长文档、多轮对话**挤不出去的就被忘掉**，信息不全 |
| **速度** | 通常**更快、更省** | 算子随序列变短，**KV Cache** 也小 |

**长窗口**

| | 影响 | **原因（背逻辑）** |
|---|------|-------------------|
| **速度** | **变慢、占显存** | 注意力等代价随长度涨（整体算力、**KV** 都涨） |
| **质量** | 信息多但可能**抓不住重点** | **注意力分散**、噪声多；检索差时**垃圾进、垃圾出** |

**收尾一句**：工程上靠 **RAG 精选片段、截断、摘要、重排序** 把“放进窗口的东西”管好。
