## OpenClaw 是什么

**OpenClaw**（大家常叫「龙虾」）是一个**开源、本地优先的 AI 智能体**，定位不是聊天机器人，而是**能真正执行任务的数字助手**。

**一句话总结**：通过接入微信、飞书、钉钉、企业微信等**第三方社交/办公平台**，你用语音或文字与**本地（或自己服务器）部署的智能体**对话，智能体在你这台机器或服务器上**实际执行**你规划的任务——例如收邮件、整理文件、跑命令、查代码、做自动化——并把结果通过同一平台反馈给你。数据留在本机或自建环境，不经过第三方，适合既要「能干活」又在意隐私和可控性的个人或团队。

## 技术特点（和别家比有啥不一样）

| 点 | 说明 |
|----|------|
| **本地/自托管** | 跑在自己电脑或服务器，数据不出本机，可完全自管 |
| **模型灵活** | 可接 Claude、GPT、豆包、智谱等云端模型，也可接本地模型 |
| **技能扩展** | 有大量预置技能，也支持自己写技能扩展能力 |
| **低延迟** | 通过异步请求池等技术，模型调用延迟可压到约 380ms 量级 |

## 怎么用

- **安装**：支持 macOS、Windows、Linux，官方提供一行命令安装脚本，也会自动装 Node 等依赖。
- **初始化**：安装后执行 `openclaw onboard` 做首次引导，按提示配好模型和要用的通道（如企业微信、飞书等）。
- **入口**：可以在终端用 CLI，也可以用桌面助手（如 macOS 菜单栏）配合使用；若接了企业微信/飞书/钉钉等，直接在对应 App 里对话即可让 OpenClaw 干活。

**官方**：[OpenClaw 官网](https://openclaws.io/zh/) · [中文文档](https://openclaw.cc/) · [GitHub](https://github.com/openclaw/openclaw)

## 原理概览：两层架构 + 六步循环（面试口述版）

可以把 OpenClaw 的工作方式理解成：**一层是“收消息和发消息的网关”，一层是“真正干活的智能体运行时”，两层之间通过一个“思考–行动–再思考”的六步循环在工作。**

**一、两层架构**

1）**网关层（Gateway）**  
- 负责接入各种对话渠道：Telegram、WhatsApp、企业微信、飞书、钉钉、Slack 等。  
- 做的事情包括：  
  - 把不同平台的消息**统一格式化**（比如都转成统一的内部 Message 结构）；  
  - 做**鉴权**（这个人/群有没有权限触发某些任务）；  
  - 做**路由分发**：把这条消息分派给对应的 Agent/技能去处理。

2）**运行时层（Agent Runtime）**  
- 真正执行任务的地方，核心是一个或多个 “Agent 智能体”。  
- 负责：  
  - 组装上下文（用户输入、历史对话、系统提示、Skills 文档等）；  
  - 调用大模型（Claude/GPT/豆包等）得到决策；  
  - 根据决策调用各种工具（浏览器、文件系统、API 等）；  
  - 把执行结果再返给模型，让它继续推理、规划下一步。

可以一句话记：**网关负责“收/发消息和接入渠道”，运行时负责“理解需求、调用大模型和工具、完成任务”。**

**二、六步循环（Agentic Loop）**

OpenClaw 和普通“问一句答一句”的聊天机器人不一样，它内部跑的是一个**持续的“思考–行动–再思考”的循环**。可以按这六步理解：

1）**组装上下文**：把这次用户输入 + 历史对话 + 系统提示（规则）+ 相关技能说明文档（Skills）整合成一段上下文。  
2）**调用大模型**：把上下文发给模型，让模型先“想一想接下来要干嘛”。  
3）**解析响应**：  
   - 如果模型直接给出“最终回答”（比如只是解释个概念），循环可以结束；  
   - 如果模型认为需要执行某个操作（如“去浏览器搜一下”、“去读一个文件”、“发一封邮件”），就变成一个**工具调用计划**，进入下一步。  
4）**执行工具**：运行对应的工具（例如调用浏览器 Skill 抓网页、读写本地文件、执行命令、调外部 API 等）。  
5）**结果回填**：把工具执行结果作为新信息写回上下文（就像人执行完一步操作后，会对“当前局面”有新的认识）。  
6）**返回循环**：带着更新后的上下文，再次回到第 2 步调用模型，让模型在“新局面”下继续思考，直到任务完成或达到最大迭代次数。

**面试一句话**：OpenClaw 不是单轮问答，而是网关层接入多渠道，运行时层通过“组装上下文→模型决策→工具执行→结果回填”的六步循环，不停迭代，直到把一个实际任务干完。

**三、面试常见追问：工具失败、上下文太长怎么办？**

- **Q1：工具调用失败了怎么办？**  
  - 工具的错误信息也会作为“工具执行结果”回填到上下文里，模型能看到“上一步失败的原因”，再决定是**重试同一工具、换一个工具，还是调整计划**。  
  - 系统侧会设定**最大迭代次数/超时**作为兜底，防止模型在错误状态下无限循环调用工具。

- **Q2：任务很复杂，上下文长度不够用怎么办？**  
  - OpenClaw 会通过多种机制管理上下文：  
    - **按需加载 Skills**：只在需要某类能力时才把对应的技能文档加载进上下文，避免一次性塞太多规则。  
    - **对话压缩（Compaction）**：把长对话/中间步骤“总结浓缩”，保留关键信息，丢掉冗余细节；  
    - **记忆刷新（Memory Flush）**：把阶段性成果写入更长期的记忆或状态，必要时再按需取回。  
  - 一句话：通过“选择性带入 + 总结压缩 + 状态持久化”，在模型上下文长度的约束下，尽量保证关键信息不丢。

## 别人都拿他干什么了（一句话带过）

整体来看，大家用 OpenClaw 做的事情可以概括为几类：**个人效率**（每日简报、待办和日历、邮件分类提醒、个人知识库、本地文件整理）、**团队协作**（项目状态查询、会议纪要和待办、群里的数字员工）、**开发与运维**（自动审查 PR、诊断报错、夜间巡检和备份）、**内容与资讯/企业金融**（把长文档转播客、做每日赛事/资讯简报、财务报销自动化、投研与风控智能体）、以及各种**浏览器自动化和群内 FAQ/活动报名**等场景。简单说，就是「凡是你平时在电脑上重复做的工作流」，大概率都能迁到 OpenClaw 里交给它按流程自动跑一遍。
