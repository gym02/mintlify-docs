## 四、数据库（MySQL/ES）（续）

### 18、MySQL 中的锁有哪些（面试口述版）

按 MySQL/InnoDB 常见的说法，你可以从 3 个层面讲：

**1）按锁粒度：**

- 表锁：锁整张表，开销小但并发差。MyIsam 就是通过表锁。
- 行锁：锁某些行，并发好；InnoDB 的行锁是基于索引实现的。InnoDB 就主要以行锁为主。

**2）按读写：**

- 共享锁（S 锁）：读锁，多个事务可以同时读。
- 排他锁（X 锁）：写锁，写时别人不能读/写。

**3）InnoDB 为了解决幻读的锁（RR 隔离级别常见）：**

- 间隙锁（Gap Lock）：锁住两个索引值之间没有实际记录的空间（比如 id=1 和 id=5 之间的 gap）。只对 id=1 和 id=5 两条锁住，其它事务仍可能往 (1,5) 插入新记录，这就可能导致 A 再次查询时结果突然多出来新行 → 幻读。为了阻止这种情况，InnoDB 通过间隙锁会锁住 (1,5) 这个区间的“间隙”，让别人不能插入新记录。这样下一次相同查询就不会多出幻影记录。
- 临键锁（Next-Key Lock）：记录锁（即最基本的行锁）+ 间隙锁，既锁记录也锁间隙。在间隙锁的基础上，锁住了左开右闭区间的一个区间 (1,5]，用行锁保证边界值不被修改。

**补充：**

- 意向锁：表级的“声明”，告诉 InnoDB 该事务打算稍后对表的某些行加哪种锁（S 或 X）。
- MDL（元数据锁）：对表结构加锁，任何对表结构的修改（如 ALTER TABLE/DROP TABLE）都需要 MDL 锁。这样可以避免 DDL（数据定义）语句和 DML（数据修改）同时冲突。

一句话总结：表锁/行锁；S 锁/X 锁；Gap/Next-Key；再补意向锁和 MDL。

### 19、MySQL 主从复制怎么实现的？（结构化答法）

MySQL 主从复制就是：主库把变更写到 binlog，从库拉取到 relay log，再在从库重放执行。

**① 前提条件**

- 主库开启 binlog（所有数据修改都会写 binlog）
- 从库能连主库，并有权限读取 binlog

**② 核心组件**

- binlog（主库）：记录所有数据变更事件（DML + DDL），是复制的“源头日志”
- relay log（从库）：从库把拉到的 binlog 先落到本地，给后续重放用
- 复制线程：负责“拉取”和“执行”

**③ 复制过程（更好理解版）**

记法：主库“记账”（binlog），从库“搬运+落盘”（relay log），再“照着做”（重放）。

1）主库提交事务，把变更写进 binlog（binlog 在主库）  
2）从库 IO 线程连主库发起请求：我要从哪个位置开始拉 binlog  
3）主库开 binlog dump 线程，把 binlog 事件流发给从库（这是对“拉取请求”的响应）  
4）从库 IO 线程收到后，写到本地 relay log（relay log 在从库）  
5）从库 SQL 线程读 relay log 并执行，把数据追平，与主库保持一致

补一句：IO 线程负责拉取，SQL 线程负责执行，执行慢不会影响继续拉取。

relay log 的价值就是：先把主库的变更可靠地落到从库本地，再慢慢按顺序执行。防止主库故障了，更改也丢失了。

| 分类 | 主要作用 | 典型语句 |
|------|----------|----------|
| **DDL** | 定义/修改 表/结构 | `CREATE`, `ALTER`, `DROP` |
| **DML** | 操作表中的数据 | `INSERT`, `UPDATE`, `DELETE` |
| **DQL** | 查询数据 | `SELECT` |
| **DCL** | 控制权限 | `GRANT`, `REVOKE` |
| **TCL** | 事务管理 | `COMMIT`, `ROLLBACK`, `SAVEPOINT` |

### 20、Elasticsearch 倒排索引（Inverted Index）

正排索引：更像“文档 ID → 词”  
倒排索引里：是“词 → 一堆文档 ID”

Elasticsearch 的查询逻辑主要分四步：

1）**分词**：分词器把查询语句拆成词项（比如“Java 开发教程”拆成“Java”、“开发”、“教程”）。  
2）**查倒排索引**：分别找每个词项的倒排列表，取交集得到命中的文档。  
3）**相关性评分**：按 TF-IDF 计算每个文档的得分（词频越高、越稀有越相关），然后排序。  
4）**返回结果**：返回排序后的文档，可能带高亮。

### 21、数据库连接池（是什么 + 常问面试题）

**连接池是什么？**  
应用里维护**一批已建好的连接**，用的时候从池里借、用完还回去，避免每次请求都新建/关闭连接。建连、关连有开销（TCP、认证等），高并发下复用连接可提性能、控连接数，避免打满 MySQL。

**常问与答法（记要点）**  
- **为什么用？** 建/关连接贵；高并发易打满 `max_connections`；池子复用、控总数。  
- **核心参数**：最大连接数、获取连接超时、空闲/最大存活时间；取连接时可做检测（如 `SELECT 1`）避免用断开的连接。  
- **连接泄漏**：连接没还（没 close、异常没还）。表现池满、拿连接超时；查代码是否都 close，Druid 可开泄漏检测看堆栈。  
- **常见实现**：HikariCP（Spring Boot 默认）、Druid（带监控/防泄漏）；按需选。  
- **池大小**：不是越大越好，结合 QPS、`max_connections`、机器数估，再压测调。

### 22、MySQL 的锁有哪些？（面试口述版）

可以从**粒度、读写、以及 RR 下防幻读**三方面说：

**1）按锁粒度**  
- **表锁**：锁整张表，开销小、并发差。MyISAM 默认表锁。  
- **行锁**：锁具体行，并发好。InnoDB 支持行锁，且**行锁是加在索引上的**，如果更新条件没走索引，可能退化成表锁。

**2）按读写（共享/排他）**  
- **共享锁（S 锁）**：读锁，多个事务可以同时持有，用于 `SELECT ... LOCK IN SHARE MODE` 等。  
- **排他锁（X 锁）**：写锁，同一行只能一个事务持有，用于 UPDATE、DELETE、`SELECT ... FOR UPDATE`。

**3）InnoDB 防幻读（RR 下常见）**  
- **间隙锁（Gap Lock）**：锁“索引值之间的空隙”，不让别人在空隙里插入，从而避免幻读。  
- **临键锁（Next-Key Lock）**：行锁 + 间隙锁，锁住“当前行 + 到下一行之前的间隙”，左开右闭区间。

**4）其他**  
- **意向锁**：表级“声明”，表示即将对某些行加 S 或 X，方便表锁和行锁协调。  
- **MDL（元数据锁）**：对表结构加锁，ALTER TABLE 等 DDL 会占 MDL，和 DML 互斥，所以大表改结构要注意阻塞。

**面试一句话：** 粒度上有表锁和行锁；读写上有 S 锁和 X 锁；RR 下用间隙锁、临键锁防幻读；再补意向锁和 MDL。

### 23、MySQL 主从复制原理？读写分离、分库分表？（面试口述版）

- **主从复制**：主库把变更写入 **binlog**；从库的 **IO 线程**连主库拉取 binlog，写到本地的 **relay log**；从库的 **SQL 线程**读 relay log 并重放，使从库数据和主库一致。一句话：主库记账（binlog），从库拉取落盘（relay log），再重放执行。  
- **读写分离**：写请求走主库，读请求走从库（可多个），减轻主库压力、提高读能力。要注意主从延迟，对强一致读仍要走主库或等延迟。  
- **分库分表**：**垂直**按业务或表拆分（不同库不同表）；**水平**按行拆分（同一表结构数据分到多库多表，如按 user_id 取模）。解决单库容量、单表行数过多、写压力过大等问题。要解决全局唯一 ID、跨库事务、跨库查询等。

面试一句话：主从 = binlog → relay log → 重放；读写分离写主读从；分库分表有垂直和水平，解决容量和性能瓶颈。

### 24、分库分表后怎么知道用哪个库？怎么切换数据源？（面试口述版）

核心是**按分片键做路由**：根据请求里带的分片键（如 user_id、order_id）算出该访问哪个库、哪张表。

**1）怎么算库和表？**  
常见做法：对分片键**取模**。例如 4 个库、每库 4 张表：`库下标 = user_id % 4`，`表下标 = (user_id / 4) % 4`（或统一 `user_id % 16` 再映射到库+表）。也有按范围、按时间、按一致性哈希的，看业务选。

**分片键是雪花算法生成的 ID，怎么算？**  
雪花算法生成的是 **Long 型数字**，不是递增的 1、2、3，但**照样可以取模**：`库下标 = snowflake_id % 库数`，或 `snowflake_id % 总表数` 再映射到库+表。雪花 ID 的高位是时间戳、中间是机器/序列，整体分布比较均匀，取模后各库各表的数据量也会比较均衡。和自增 ID 的区别：自增取模容易让“新数据”集中到某几个库（因为自增连续）；雪花取模不会让新数据总落在同一个库，分布更均匀，路由算法本身不需要改。

**2）怎么切换数据源？**  
应用不会写死连一个库，而是“多数据源 + 按需选一个”：

- **方式一：应用内封装多数据源**  
  在代码里配置多个 DataSource（如 Spring 里配多个 DataSource），根据**当前请求的分片键**（从参数、ThreadLocal 或上下文里取）算出用哪个库，在切面（AOP）或工具里把“当前要用的数据源”设置到 ThreadLocal，MyBatis/JPA 执行时从 ThreadLocal 取数据源再执行。这样一次请求内，所有 SQL 都走同一个库。

- **方式二：用分库分表中间件**  
  用 **ShardingSphere-JDBC**（内嵌在应用里）或 **ShardingSphere-Proxy**（独立代理）。在配置里写好分片规则（如 user_id 取模 4 库、每库 4 张表），应用只连一个“逻辑数据源”；中间件在执行 SQL 时根据 SQL 里的分片键值自动算库、算表，再路由到对应的真实库表。对业务代码来说就像在查一张表，不用自己切换数据源。

**面试一句话：** 分库分表后靠**分片键 + 路由规则**（如取模）算出库和表；切换数据源要么在应用里自己封装多数据源、按分片键选库（ThreadLocal + AOP），要么用 ShardingSphere 等中间件自动路由。

### 25、缓存和数据库一致性怎么保证？Redis 和 MySQL 怎么配合？（面试口述版）

- **一致性**：常用 **Cache Aside**。读：先查缓存，命中就返回，未命中再查 DB 并回写缓存。写：**先更新数据库，再删除缓存**（而不是先更新缓存），这样下次读会回源 DB 拿到最新值再填缓存。避免“先更新缓存再更新 DB”时并发导致缓存和 DB 不一致。可配合重试、消息队列补偿删缓存。  
- **Redis 和 MySQL 配合**：热点读放 Redis，减轻 MySQL 压力；写仍落 MySQL，再按上面策略更新/删除缓存。可配合过期时间、缓存穿透/击穿/雪崩的防护（布隆过滤器、互斥更新、多级缓存等）。

面试一句话：Cache Aside 写时先更 DB 再删缓存；Redis 扛读、MySQL 扛写，一致性靠更新顺序和删缓存保证。
