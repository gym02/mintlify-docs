## Skills、Prompt、MCP、Function Calling 有什么区别？

（以 Cursor / 现代 AI 编程助手为例，方便面试或自用时说清这几类能力。）

1）Prompt（提示 / 自然语言指令）

- **是什么**：你写给模型的一段**自然语言**，告诉它“做什么、按什么要求做”。例如在对话框里输入“帮我把这段代码改成异步”“用 Java 写一个单例”。
- **特点**：一次性的、随用随写；模型根据当前对话上下文 + 这段 prompt 生成回复。没有固定结构，不涉及“自动调用外部能力”。
- **类比**：你口头跟同事说“帮我把这个需求实现一下”，属于**输入指令**。

2）Skills（技能 / 技能文档）

- **是什么**：预置的**结构化说明文档**（如 SKILL.md），描述某类任务该怎么做、用哪些步骤、遵守什么规范。Agent 在执行这类任务时会**先读技能文档**，再按里面的流程和规则来行动。
- **特点**：可复用、可版本管理；把“最佳实践”和“领域知识”固化下来，让 Agent 在写代码、做评审、改配置等场景下行为一致、少出错。
- **与 Prompt 的区别**：Prompt 是**当次对话里你临时说的话**；Skills 是**事先写好的、可被多次引用的操作手册**。Prompt 更灵活、即兴；Skills 更稳定、可积累。

3）MCP（Model Context Protocol，模型上下文协议）

- **是什么**：一套**开放协议**，让 AI 能连接并调用**外部工具/数据源**（如本地文件、数据库、浏览器、API）。这些能力以“MCP 服务器”的形式提供，模型通过统一接口去“用”这些工具，拿到结果再继续推理。
- **特点**：扩展的是模型的**“手和脚”**——能读文件、执行命令、查数据库、操作浏览器等，而不只靠“想”和“生成文本”。
- **与 Prompt/Skills 的区别**：Prompt 和 Skills 都是“给模型看的文字”；MCP 是“给模型用的外部能力”。没有 MCP，模型只能输出文本；有了 MCP，模型可以驱动真实工具并拿到结构化结果。

4）Function Calling（函数调用）

- **是什么**：模型在回复时不仅输出自然语言，还可以输出**结构化的“调用请求”**（函数名 + 参数），由**宿主环境**（你的应用、IDE、API 服务）真正执行该函数，把执行结果再塞回给模型，模型再根据结果继续回答或决策。
- **特点**：模型负责“决定调什么、传什么参数”；执行和权限在你这边，适合查数据库、发请求、改配置等需要**安全边界**的场景。
- **与 MCP 的关系**：MCP 是**协议 + 生态**（如何定义工具、如何连接多种服务器）；Function Calling 是**模型侧的能力**（如何表达“我要调用哪个函数、传什么参”）。很多 MCP 工具在实现上会用到“模型发起调用 → 宿主执行”的机制，和 Function Calling 是同一类思路。

一句话对照
| 概念 | 一句话 |
|------|--------|
| **Prompt** | 你临时输入的自然语言指令，告诉模型“这次要做什么”。 |
| **Skills** | 事先写好的任务流程与规范文档，模型在做某类任务时按它来执行。 |
| **MCP** | 让模型能连接并调用外部工具/数据源的协议，扩展“能做的事”。 |
| **Function Calling** | 模型输出“调用哪个函数、传什么参”，由宿主环境真正执行并回传结果。 |

## RAG（检索增强生成）

RAG = 先检索知识库得到相关片段，再把这些片段和用户问题一起给大模型生成回答，减少幻觉、可更新知识。**概念说明与用 ChatGPT API 从零搭建、组装使用的具体步骤见 [RAG 是什么与如何搭建](/concepts/rag)。**

## LangGraph（有状态编排 / Agent 工作流）

- **是什么**：用于构建**有状态、可编排**的 Agent 工作流的框架。把流程建模成**图**：节点（Node）= 可执行单元（调 LLM、工具、数据处理），边（Edge）= 节点间的流向（顺序、分支、循环），状态（State）= 节点间共享的上下文。相比线性链式调用，支持分支、循环、重入，更适合复杂多步与 Multi-Agent。
- **和 LangChain 的区别与联系**：  
  **联系**：LangGraph 出自 LangChain 生态，底层可以复用 LangChain 的模型封装、Tool、Chain 等组件；两者常一起用（用 LangChain 接模型和工具，用 LangGraph 做流程编排）。  
  **区别**：LangChain 侧重**链式调用**（一步接一步的 Pipeline）和**组件接口**（LLM、Prompt、Retriever 等）；LangGraph 侧重**图编排**和**显式控制流**（分支、循环、多节点、有状态），把“下一步走哪”写成图而不是固定链。不强制依赖 LangChain，可单独用 LangGraph 做图，底层换别的模型/工具库也行。

**面试常问：**

1）必须用 LangChain 才能用 LangGraph 吗？  
**答：** 不需要，LangGraph 可独立使用。

2）LangGraph 和其他 Agent 框架有啥区别？  
**答：** 用图显式表达控制流，支持分支、循环、重入，不限制成单一黑盒链路，适合定制复杂任务和 Multi-Agent。

3）LangGraph 会影响应用性能吗？  
**答：** 不会额外增加运行时开销，且支持流式。

4）LangGraph 开源吗？  
**答：** 是，MIT 许可，可免费使用。

5）Multi-Agent 怎么设计？  
**答：** 每个 Agent 做成图里的节点，职责单一、工具集合固定；用一个 **Supervisor** 节点做路由，根据当前状态决定下一步走哪个 Agent；状态在节点间共享，可设最大步数/超时与失败重试，保证稳定和成本可控。
