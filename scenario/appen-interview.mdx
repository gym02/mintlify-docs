### 一、前端相关（简单问问）

#### 1. 浏览器从输入 URL 到页面展示，发生了什么？

**回答结构建议：** 你可以把过程分成 4～5 个关键阶段，用「大白话 + 关键词」的方式讲清楚。

**示例回答：**

1. **DNS 解析和连接建立**
   - 浏览器先通过 DNS 把域名解析成 IP。
   - 然后通过 TCP（三次握手）建立连接，通常是走 HTTPS（TLS 握手）。
2. **发起 HTTP 请求**
   - 浏览器根据地址栏信息构造 HTTP 请求（方法、URL、Header、Body 等）。
   - 把请求发给服务器，比如 Nginx，再转发到后端服务（如 Spring Boot）。
3. **服务器处理并返回响应**
   - 后端根据路由和业务逻辑，访问数据库或其他服务。
   - 把结果封装成 HTTP 响应（状态码、响应头、HTML/JSON 等）返回。
4. **浏览器解析与渲染**
   - 解析 HTML 构建 DOM 树，解析 CSS 构建 CSSOM，合成渲染树。
   - 执行 JavaScript，可能会修改 DOM / CSSOM，再触发重排和重绘。
   - 加载图片、字体等静态资源，最终把页面渲染出来。
5. **后续交互**
   - 用户的点击、输入等事件被 JavaScript 监听，发起 Ajax / Fetch 请求更新页面。
   - 视图层通过框架（React / Vue 等）和状态管理进行更新。

**简洁总结：** 输入 URL → DNS 解析 → 建立 TCP/HTTPS 连接 → 发送 HTTP 请求 → 后端处理并返回 → 浏览器解析渲染 → JS 继续驱动交互。

#### 2. `flex: 1` 代表什么？常见用法？

**关键点：**

- `flex` 是 `flex-grow flex-shrink flex-basis` 的缩写。
- `flex: 1` 通常等价于：`flex: 1 1 0%`。

**示例回答：**

- 在一个 `display: flex` 的父容器中，子元素设置 `flex: 1`，通常表示：
  - **可以按比例扩展占据剩余空间**（`flex-grow: 1`）。
  - **在空间不足时允许缩小**（`flex-shrink: 1`）。
  - **初始宽度为 0，由剩余空间按比例分配**（`flex-basis: 0%`）。

**典型用法：**

```css
.container {
  display: flex;
}

.item {
  flex: 1; /* 每个 .item 平分容器剩余空间 */
}
```

- 多个子元素都设为 `flex: 1`，就会**平均分配可用宽度**。
- 如果一个是 `flex: 1`，另一个是 `flex: 2`，则宽度比例为 1:2。

#### 3. 前端如何做接口错误处理和全局异常捕获？

**回答结构建议：** 分为「单次请求」和「全局处理」两层：

1. **单次请求级别**
   - 使用 `try / catch` 捕获 `async/await` 中的异常。
   - 对 HTTP 状态码（如 400、401、403、500）进行判断，按不同情况给出用户提示。
2. **全局请求拦截器**
   - 如果用 Axios，可以配置 request / response 拦截器：
     - 在 request 拦截器中统一加 Token、TraceId 等。
     - 在 response 拦截器中统一处理非 2xx 状态码、后端业务错误码。
   - 例如 401 统一跳转登录页，500 统一弹出「系统异常，请稍后再试」。
3. **前端运行时异常捕获**
   - 使用 `window.onerror`、`window.onunhandledrejection` 捕获未处理的错误。
   - 在 React 里可以通过错误边界组件进行局部兜底。
   - 可以结合前端监控（如 Sentry、自建埋点）上报错误信息。

**加分点：** 结合你实际项目说一句，例如「在项目中，我用 Axios 拦截器 + 统一 Toast 组件来做全局错误处理」。

### 二、Java 后端相关

#### 4. 一个 HTTP 请求在 Spring Boot 里是怎么被处理的？

**回答结构建议：** 从「入口」讲到「出口」，把主要组件串起来。

**示例回答：**

1. **入口：Web 容器接收请求**
   - 一般使用内嵌的 Tomcat / Jetty。
   - 它接收到 HTTP 请求后，交给 Spring MVC 处理。
2. **Spring MVC 分发请求**
   - `DispatcherServlet` 是前端控制器。
   - 根据请求路径和方法，从 HandlerMapping 找到对应的 Controller 方法。
3. **业务处理**
   - 调用对应的 Controller 方法。
   - Controller 再调用 Service / Repository，进行业务逻辑处理和数据库访问。
4. **返回结果与序列化**
   - 方法返回对象或 `ResponseEntity`。
   - 使用 `HttpMessageConverter`（如 Jackson）把对象序列化成 JSON。
   - Spring 把响应写回给容器，再返回到浏览器。
5. **过滤器 / 拦截器（可选）**
   - 请求处理前后，可以经过 Filter / HandlerInterceptor 做：
     - 鉴权、日志、链路追踪等横切逻辑。

#### 5. 使用 Session 和 JWT 实现登录，各自优缺点？

**Session 方案：**

- **原理：**
  - 用户登录成功后，服务器创建 Session，保存用户信息。
  - 把 SessionID 写入 Cookie（或 URL），之后每次请求带上这个 ID。
  - 服务器通过 Session 存储验证用户身份。
- **优点：**
  - 后端容易控制会话失效，服务端随时能删除 Session。
  - 存在服务器端，相对容易做权限控制和会话管理。
- **缺点：**
  - Session 状态在服务器内存或集中缓存（如 Redis）中，有**状态**。
  - 做水平扩展时，需要 Session 共享或粘性会话，部署较复杂。

**JWT 方案：**

- **原理：**
  - 登录成功后，后端签发 JWT（Header + Payload + Signature）。
  - 客户端保存 Token（Cookie 或 LocalStorage）。
  - 每次请求带上 Authorization Header，后端通过签名验证合法性。
- **优点：**
  - 不依赖服务端 Session，可实现**无状态**服务，水平扩展友好。
  - 前后端分离、跨域时使用方便。
- **缺点：**
  - Token 一旦签发，不易单点失效（需要黑名单等机制）。
  - Payload 是明文（Base64），敏感信息需要避免放入或再加密。
  - Token 变长，频繁携带在 Header 会稍微增加带宽开销。

**总结一句：** 传统单体应用、用户量不大时用 Session 简单直接；前后端分离、微服务场景更适合 JWT 或 OAuth2 这类无状态方案。

#### 6. 描述一次你解决过的 Bug 或性能问题（答案模版）

你可以用 STAR 法则来回答（Situation、Task、Action、Result）：

1. **背景和现象（Situation）**
   - 某接口在高并发场景下响应很慢，监控发现 P95 响应时间 > 2s。
   - 日志里没有明显错误，但数据库 CPU 和 QPS 很高。
2. **任务（Task）**
   - 找出性能瓶颈，保证在高峰期也能稳定在 200ms 左右。
3. **行动（Action）**
   - 先用日志和监控定位是某个查询接口慢。
   - 打开 SQL 日志，发现某条 SQL 没有走索引，是「模糊匹配 + 排序」。
   - 通过 `EXPLAIN` 查看执行计划，确认是全表扫描。
   - 调整查询条件，改用前缀匹配 + 合理分页。
   - 给排序字段 + 过滤条件建立联合索引。
   - 通过 Redis 做简单缓存，避免重复查询。
4. **结果（Result）**
   - 优化后接口平均耗时从 1s 降到 100ms 左右。
   - 监控指标恢复正常，业务方反馈使用体验明显改善。

> 把上面这个模版中的「某接口」「某 SQL」替换成你真实项目中的例子即可。

### 三、DevOps：Jenkins / GitLab CI/CD（从 0 开始版）

这一节按「先搞懂概念 → 再说怎么用」来写，适合你完全没实际用过，从头准备面试。

#### 1. 先搞清这几个大词

- **DevOps**：Development + Operations，一句话就是：用一整套自动化流程，让「写代码、测代码、上线、运维」的人协同起来，**快速而稳定地发版**。
- **CI（持续集成）**：代码一提交，就自动触发「拉代码 → 构建 → 跑测试 → 代码检查」这一串动作，尽早发现问题。
- **CD（持续交付/部署）**：把构建好的包/镜像，**自动部署**到测试/预发/生产环境，最好还能一键回滚。
- **Pipeline（流水线）**：就是把上面这些动作排成的一条「脚本流程」，比如：
  - Stage1：build（编译打包）
  - Stage2：test（测试）
  - Stage3：deploy（部署）
  这条线在 Jenkins 里用 `Jenkinsfile` 描述，在 GitLab 里用 `.gitlab-ci.yml` 描述。

> 面试可以先丢一句总括：  
> 「我理解的 DevOps / CI/CD，就是通过 Pipeline 把构建、测试、部署这些步骤自动化起来，让发版可重复、可回滚。」

#### 2. Jenkins 里的几个关键概念

- **Jenkins 本身**：一台独立部署的 CI/CD 服务器，本质是「帮你在服务器上按顺序/并行跑脚本」。
- **Job**：早期的配置方式，你在 Jenkins Web 界面里点点点配置「要执行哪些命令」。
- **Pipeline**：
  - 更现代的用法，用一个 `Jenkinsfile`（Groovy DSL）把流水线写成代码；
  - 好处是：能放进代码仓库里版本管理，团队协作更方便。
- **Stage**：
  - Pipeline 里的「大步骤」，比如 `Build`、`Test`、`Deploy`；
  - 一般在可视化界面里，会以几个大方块展示每个 Stage 的状态。
- **Step（或 Steps）**：
  - 每个 Stage 里具体的「小动作」，比如执行一条 Shell 命令、发通知等。
- **Agent / Node**：
  - 真正执行构建任务的机器，可以是多台；
  - 有些用来跑 Java 构建，有些用来跑前端，有些连着 Docker / K8s。
- **触发方式（Trigger）**：
  - 代码 push / MR 触发；
  - 定时任务（比如每天凌晨构建一次）；
  - 手工点击「Build Now」。

> 一句话记忆：**在 Jenkins 里，就是用 Pipeline 把一堆 Step 排成几个 Stage，让 Agent 帮你自动跑完这整条线。**

#### 3. GitLab CI/CD 里的关键概念

- **`.gitlab-ci.yml`**：
  - 放在项目根目录的配置文件，用来定义这条流水线长什么样；
  - 类似 Jenkins 的 `Jenkinsfile`，但语法是 YAML。
- **stages**：
  - 在 `.gitlab-ci.yml` 里先声明有哪些阶段，比如：
    ```yaml
    stages:
      - build
      - test
      - deploy
    ```
  - 同一个 stage 里的 job 可以并行，不同 stage 之间默认按顺序执行。
- **jobs**：
  - 每个 job 是具体执行的一段脚本，比如：
    ```yaml
    build-backend:
      stage: build
      script:
        - mvn clean package
    ```
  - 你可以有很多 job，分别构建后端、前端、运行不同测试等。
- **Runner**：
  - 真正执行 job 的机器（或容器），类似 Jenkins 的 Agent；
  - 可以是共享 Runner，也可以是某个项目/组的专用 Runner。
- **variables（变量）**：
  - 在 GitLab 的 CI/CD settings 里配置的环境变量；
  - 比如 `DB_URL`、`ENVIRONMENT=test/prod`，可以在 job 的脚本里读取；
  - 用来区分测试环境、预发、生产等配置。
- **artifacts / cache**（知道大概干嘛就行）：
  - artifacts：某个 job 产生的「构建产物」，可以传给后续 job，或者提供下载；
  - cache：依赖缓存（例如 Maven 仓库、npm cache），让多次构建更快。
- **environments**：
  - 给某些 job 标记是部署到 `test` / `staging` / `production`；
  - GitLab 会记录最近一次在某个环境部署的是哪个 commit。

> 一句话记忆：**GitLab CI/CD = 在 `.gitlab-ci.yml` 里配置 stages + jobs，然后由 Runner 按顺序自动执行，实现从构建到部署的一条流水线。**

#### 7. 描述一条你熟悉的 CI/CD 流水线

**示例回答：**

1. **触发方式**
   - 开发者提交代码到 GitLab 指定分支（如 `develop` / `master`）。
   - Push / Merge Request 会触发 CI 流水线。
2. **CI 阶段（持续集成）**
   - 代码拉取：Runner/Agent 拉取对应分支代码。
   - 依赖安装与构建：如 `mvn clean package`、`npm install && npm run build`。
   - 静态检查与单元测试：如 `mvn test`、`eslint`、`jest`。
   - 构建镜像：用 Dockerfile 构建应用镜像，并推送到镜像仓库。
3. **CD 阶段（持续交付 / 部署）**
   - 测试环境部署：
     - 拉取最新镜像，更新到测试环境（K8s / Docker / 物理机）。
     - 运行数据库迁移脚本，执行简单冒烟测试。
   - 生产环境部署（可选手动审批）：
     - 通过 Jenkins / GitLab CI 的手动审批步骤后才执行。
     - 使用滚动更新 / 蓝绿发布降低风险。
4. **回滚策略**
   - 保留上一版本镜像和配置。
   - 部署失败或线上有严重问题时，一键回滚到上一个稳定版本。

#### 8. 如何区分测试环境和生产环境、管理构建参数？

**在 Jenkins 中：**

- 使用「构建参数」：
  - 例如 `ENV=test/prod`，在流水线脚本中根据环境变量去选择不同配置。
- 在 Pipeline（Jenkinsfile）中：
  - 通过 `when { environment name: 'ENV', value: 'prod' }` 控制不同 Stage。
  - 使用不同的 kubeconfig 或 Ansible Inventory。

**在 GitLab CI 中：**

- 使用不同 `stage` 和 `environment`：
  - 如 `deploy-test`、`deploy-prod`。
- 通过变量（CI/CD Variables）注入不同的数据库地址、API 地址等。
- 对生产环境的 job 添加 `when: manual` 实现手动审批。

#### 9. 流水线失败但本地构建成功时，怎么排查？

**示例回答：**

1. **环境差异**
   - 查看 CI 环境的 JDK / Node / Maven / OS 版本是否与本地一致。
   - 检查是否有私有依赖在 CI 环境拉取不到（网络、凭证等问题）。
2. **配置和凭证**
   - CI 中没有正确配置环境变量（如数据库密码、Token）。
   - 权限不足导致某些命令在 CI 上失败（如 Docker 构建、文件写入权限）。
3. **并行与顺序问题**
   - 本地构建通常是按顺序跑，而 CI 可能会并行执行多个 Job。
   - 某些脚本可能隐含依赖（必须先创建某目录或配置文件）。
4. **日志和重现**
   - 首先仔细看流水线日志，找到出错命令和具体错误信息。
   - 在本地模拟 CI 环境（Docker 容器或同配置机器）重现问题。

### 四、算法：两数之和

#### 10. 题目：给定数组和目标值，找出两个数和为目标值的下标

**题目描述（简化版）：**

> 给定一个整数数组 `nums` 和一个整数目标值 `target`，请你在该数组中找出和为目标值 `target` 的**那两个整数**，并返回它们的数组下标。  
> 假设每种输入只会对应一个答案，不能重复使用同一个元素。

---

#### 解法一：暴力解法

- 用两层循环，枚举所有两个数的组合：
  - 外层下标 `i`，内层下标 `j > i`。
  - 如果 `nums[i] + nums[j] == target`，返回 `[i, j]`。
- **时间复杂度**: \(O(n^2)\)，实现简单，但在面试中一般作为「先想到的基础方案」。

---

#### 解法二：哈希表优化（推荐）

- 利用「空间换时间」：
  - 遍历数组时，维护一个 `Map<数值, 下标>`。
  - 对于当前元素 `nums[i]`：
    - 计算差值 `need = target - nums[i]`。
    - 如果 `map` 中已经有 `need`，说明找到了两数：`map.get(need)` 和 `i`。
    - 否则，把当前 `(nums[i], i)` 放入 `map` 中，继续遍历。

- **时间复杂度**: \(O(n)\)
- **空间复杂度**: \(O(n)\)

**Java 示例代码：**

```java
public int[] twoSum(int[] nums, int target) {
    Map<Integer, Integer> map = new HashMap<>();
    for (int i = 0; i < nums.length; i++) {
        int need = target - nums[i];
        if (map.containsKey(need)) {
            return new int[]{map.get(need), i};
        }
        map.put(nums[i], i);
    }
    return new int[]{-1, -1}; // 理论上不会到这里
}
```

**面试表达建议：**

> 「我会先提到 O(n^2) 的暴力解法，说明我能想到最直接的方法。然后我会主动说可以优化，用哈希表把时间复杂度降到 O(n) 并给出思路和伪代码。」

### 五、AI 相关

#### 11. 你如何理解机器学习和深度学习的区别？

**示例回答：**

- **机器学习**：
  - 更广义的概念，包括线性回归、逻辑回归、决策树、SVM、随机森林等。
  - 一般需要人工做特征工程（提取特征、选择特征）。
- **深度学习**：
  - 是机器学习的一个子集，核心是「深层神经网络」。
  - 通过多层网络自动学习复杂特征，尤其适合图像、语音、自然语言处理等数据。
  - 依赖大规模数据和高算力（GPU）。

**总结一句：** 深度学习是机器学习的一种，更侧重用多层神经网络自动学习特征；机器学习则包括更多传统算法，通常需要手工做特征工程。

#### 12. 你在 AI 项目中可以发挥哪些作用？

结合澳鹏科技做 AI 训练数据、标注、工具链的特点，你可以从以下几个方向回答：

- **数据与平台侧：**
  - 设计和开发数据标注平台的前后端功能（任务分配、标注界面、质检流程）。
  - 开发数据清洗、简单统计的脚本，配合算法同学准备训练数据。
- **工程化侧：**
  - 封装模型推理服务，把模型部署为 HTTP / gRPC API，供业务系统调用。
  - 搭建简单的监控和日志体系，记录模型调用情况和效果（如响应时间、成功率）。
- **集成与体验：**
  - 把 AI 能力集成到业务系统前端，例如智能推荐、智能搜索、文本审核等组件。
