## 九、场景题（系统设计口述版）

### 1、设计登录系统/登录页面（面试讲解稿：1-2 分钟口述版）

我会从“功能实现、安全防范、高并发优化”三个层次来设计。

**首先是基本功能：**  
- 核心逻辑：支持多标识（手机/邮箱/用户名）+ 密码登录。  
- 会话管理：后端验证通过后，发放 Token（如 JWT）或创建 Session，前端存入 Cookie 或 LocalStorage，后续请求带上标识。

**补充知识点：什么是 JWT？**  
- Token：就是一个身份令牌（字符串），证明登录过。  
- JWT（JSON Web Token）：是一种特殊的 Token，它把用户信息（如 userId）直接加密在字符串里。  
- 优点：服务器不需要存 Session（无状态），直接解密就能知道你是谁，非常适合分布式架构。  
- 安全：靠服务器的“密钥”签名，别人改不了内容。

**其次是关键细节（点到为止）：**  
- 传输与存储安全：全站 HTTPS 必选；密码严禁明文，注册和登录比对必须使用加盐哈希（如 Bcrypt）。  
- 交互校验：前端做防抖和格式校验，后端做统一错误提示（不暴露是账号错还是密码错），防止撞库。

**补充知识点：什么是加盐哈希（Bcrypt）？**  
- 用途：专门用于【数据库存储密码】，不是加密 Token。  
- 哈希（Hash）：把密码变成不可逆的乱码，即使数据库泄露也看不到原密码。  
- 加盐（Salt）：每个用户分配一个随机字符串（盐），即使两人密码相同，存的乱码也不同。  
- Bcrypt：后端最常用的加密库，它会自动处理加盐，且计算速度故意设计得很慢，防止黑客暴力破解。

**补充知识点：前端防抖（Debounce）**  
- 含义：你尽管点，我只执行最后一次。  
- 原理：设置一个计时器，如果用户在计时结束前又点了，就清空重连，直到用户停下超过设定时间才发请求。  
- 目的：防止用户因为网络慢或着急连点，导致向后端发送一堆重复的请求。

**最后是体现技术功底的优化（针对高并发和稳定性）：**  
- 性能优化：在数据库 identifier 字段（用户名/手机号）建唯一索引；将会话数据（Token/Session）存入 Redis，利用内存读写应对高频的登录校验请求。  
- 流量限流与防爆破：接入 Redis 计数器做接口限流，针对同 IP 或同账号频繁失败的情况进行强制验证码拦截或临时封禁。  
- 扩展性：如果考虑到分布式环境，我会采用 JWT 或 Redis 共享 Session，保证水平扩展时登录状态的一致性。

### 2、MQ 消息确认机制？（面试口述版）

MQ 要保证消息不丢、不重复，离不开**生产者确认**和**消费者确认**两套机制；面试常问“怎么保证消息可靠投递”“ack 怎么用”。

**生产者确认（消息是否成功到达 Broker）：**

- **RabbitMQ**：开启 **Publisher Confirm**。生产者发完消息后，Broker 会异步回调 confirm 或 nack；收到 confirm 才认为写入成功，否则可重发。配合持久化（队列、消息都持久化），避免 Broker 宕机丢消息。
- **Kafka**：生产者根据 **acks** 配置。`acks=1` 表示 leader 写成功就返回；`acks=all`（或 -1）表示 leader + 所有 replica 都落盘才返回，更可靠但延迟更高。

**消费者确认（消息是否被业务成功处理）：**

- **RabbitMQ**：  
  - **自动 ack**：消息一推给消费者就从队列删除，消费者处理失败也会丢消息，一般不用。  
  - **手动 ack**：消费者处理完业务后调用 `basicAck`，Broker 才删消息；处理失败可 `basicNack` / `basicReject`，消息可重新入队或进死信队列。  
  - 配合 **prefetch（QoS）** 限制未 ack 的数量，避免消费者压垮或消息堆积在客户端。
- **Kafka**：通过 **offset 提交** 表示“已消费到哪里”。`enable.auto.commit=true` 时自动按间隔提交，可能重复消费；生产上常用**手动提交**（先处理再 commit），保证至少处理一次。要防重复需业务幂等或配合事务/幂等表。

**常见语义简要对比：**

- **至少一次（at least once）**：可能重复，不会丢。消费者先处理再 ack/commit，失败则重试。  
- **至多一次（at most once）**：可能丢，不重复。发完/收到就认为成功，不重试。  
- **恰好一次（exactly once）**：不丢不重。需要生产者幂等 + 消费者幂等 + 存储支持（如 Kafka 事务、幂等表），实现成本高。

**面试可一句话概括：** 生产者侧用 MQ 的 confirm/acks 保证消息到 Broker；消费者侧用手动 ack 或手动提交 offset，先业务成功再确认，配合持久化和重试，实现至少一次可靠消费；要防重复就做幂等。

### 3、RabbitMQ 消息积压如何处理？

当 RabbitMQ 消息出现积压时，我会从三个主要方面进行处理：

**提升消费者处理能力：**  
- 横向扩容：最直接有效的方式是增加消费者实例数量，让更多消费者并行处理消息，提高整体吞吐量。  
- 优化消费逻辑：对消费者业务逻辑进行优化，如将复杂耗时操作改为异步处理（快速 ACK），或采用批量处理机制，以提高单个消费者的处理效率。  
- 合理限流 (QoS)：通过 `basic.qos` 控制消费者一次性拉取的消息数量，避免单个消费者因处理过多消息而崩溃，保持处理的稳定性。

**优化 RabbitMQ 队列配置：**  
- 限制队列长度：通过 `x-max-length` (消息数) 或 `x-max-length-bytes` (总大小) 配置，避免队列无限膨胀，超出限制后可选择丢弃最老消息或发送到死信队列。  
- 消息 TTL (Time To Live)：为消息或队列设置过期时间，让超期未消费的消息自动清除或进入死信队列，避免无效消息长时间占用资源。  
- 死信队列 (DLQ)：配合 TTL 或消息拒绝机制，将过期、被拒或超长的消息路由到死信队列（DLQ），进行统一管理、分析和后续处理（如告警、重试），避免其影响主业务队列。  
- 优先级队列：针对核心业务消息，设置高优先级，确保在积压时能优先被处理，保障业务核心流程。

**系统级流量控制与保护：**  
- 消息分级：根据消息重要性，将不同等级的消息投递到不同的队列，甚至独立的 MQ 集群，确保核心业务通道的畅通和优先级。  
- 上游限流：在消息生产方进行流量控制（如令牌桶、漏桶算法），防止突发或过高的流量直接冲垮 MQ 或下游服务，从源头控制风险。  
- 下游熔断降级：当消息处理的下游服务出现故障或处理能力明显下降时，消费者可以暂停消费，或将消息路由到降级处理逻辑（如持久化到数据库稍后重试），避免故障蔓延导致整个系统雪崩。  
- 监控与报警：建立完善的监控体系，实时监测队列长度、消费者状态、消息生产消费速率等关键指标，并配置及时报警，以便在积压初期迅速发现问题并介入处理。

### 4、诊所收费统计 SQL 笔试题

**题目：给定四张表如下：**

clinic 存储诊所信息（假设数据上万条）  

- id 数据表主键（诊所 ID）  
- name 诊所名字  

employee 存储雇员信息（假设数据上万条）  

- id 数据表主键（雇员 ID）  
- name 雇员名字  
- type 雇员类型，1：医生，2：收费员  

clinic_employee 存储诊所和雇员之间关联，一个雇员可以在多个诊所就职  

- id 数据表主键  
- clinic_id 诊所 id  
- employee_id 雇员 id  

charge 存储收费信息（假设数据上亿条）  

- id 数据表主键  
- clinic_id 诊所 id  
- employ_id 雇员 id  
- amount 收费金额  
- created_time 收费时间，格式 `2020-01-01 00:00:00`  

请给出下述能实现需求对应 SQL：  

1. 按天给出所有收费员收费金额，按收费金额降序输出。  
2. 按天给出每个诊所每个收费员收费金额，按收费金额降序输出。  
3. 按天给出每个诊所每个雇员收费金额，并给出雇员类型，按收费金额降序输出。  
4. 按天给出每个诊所收费金额最大的雇员。  

**参考答案：**

**1）按天给出所有收费员收费金额，按收费金额降序输出**

理解为「**按天 + 收费员** 维度统计」，不区分诊所，总金额按降序排列：

```sql
SELECT
  DATE(c.created_time)      AS charge_date,
  e.id                      AS employee_id,
  e.name                    AS employee_name,
  SUM(c.amount)             AS total_amount
FROM charge c
JOIN employee e ON c.employ_id = e.id
WHERE e.type = 2                     -- 只统计收费员
GROUP BY DATE(c.created_time), e.id, e.name
ORDER BY total_amount DESC;          -- 按收费金额降序
```

如需「同一天内先按日期再按金额排序」，可以 `ORDER BY charge_date, total_amount DESC`。

**2）按天给出每个诊所每个收费员收费金额，按收费金额降序输出**

维度变为「按天 + 诊所 + 收费员」：

```sql
SELECT
  DATE(c.created_time)      AS charge_date,
  cl.id                     AS clinic_id,
  cl.name                   AS clinic_name,
  e.id                      AS employee_id,
  e.name                    AS employee_name,
  SUM(c.amount)             AS total_amount
FROM charge c
JOIN clinic   cl ON c.clinic_id = cl.id
JOIN employee e  ON c.employ_id = e.id
WHERE e.type = 2
GROUP BY
  DATE(c.created_time),
  cl.id, cl.name,
  e.id, e.name
ORDER BY total_amount DESC;          -- 按金额降序
```

这里没有强制要求一定使用 `clinic_employee` 表，因为 `charge` 已经携带 `clinic_id` 和 `employ_id`；如果需要校验合法关系，也可以额外 `JOIN clinic_employee`。

**3）按天给出每个诊所每个雇员收费金额，并给出雇员类型，按收费金额降序输出**

在上一问的基础上去掉「只统计收费员」的过滤，并把雇员类型也查出来：

```sql
SELECT
  DATE(c.created_time)      AS charge_date,
  cl.id                     AS clinic_id,
  cl.name                   AS clinic_name,
  e.id                      AS employee_id,
  e.name                    AS employee_name,
  e.type                    AS employee_type,
  SUM(c.amount)             AS total_amount
FROM charge c
JOIN clinic   cl ON c.clinic_id = cl.id
JOIN employee e  ON c.employ_id = e.id
GROUP BY
  DATE(c.created_time),
  cl.id, cl.name,
  e.id, e.name,
  e.type
ORDER BY total_amount DESC;
```

**4）按天给出每个诊所收费金额最大的雇员**

思路：先按「按天 + 诊所 + 雇员」维度汇总收费金额，然后在 `HAVING` 里用相关子查询，要求当前雇员的总金额等于该诊所当天的最大总金额；这一版不用窗口函数，大多数 MySQL 版本都支持。

```sql
SELECT
  DATE(c.created_time) AS charge_date,
  c.clinic_id,
  c.employ_id          AS employee_id,
  SUM(c.amount)        AS total_amount
FROM charge c
GROUP BY DATE(c.created_time), c.clinic_id, c.employ_id
HAVING total_amount = (
  SELECT MAX(sub_sum)
  FROM (
    SELECT
      SUM(c2.amount) AS sub_sum
    FROM charge c2
    WHERE DATE(c2.created_time) = DATE(c.created_time)
      AND c2.clinic_id          = c.clinic_id
    GROUP BY c2.employ_id
  ) x
)
ORDER BY charge_date, total_amount DESC;
```

#### 1、给定一个字符串以及拼接符，输出反序的字符串

**题目：**

给定一个字符串以及拼接符，输出反序的字符串。比如输入 `"prod,gray,pre"`，分割符号是逗号，输出 `"pre,gray,prod"`。  
（要求：至少用 Java 基本类型进行一种实现；另外可以使用其他类库的方法实现）

```java
/*
 * @param needReverseValue 需要反序的字符串
 * @param delimiter        分割符号
 */
public static String reverseDelimitedString(String needReverseValue, String delimiter){

}
```

**参考答案一：仅用基础 API 手动切分 + 拼接（不依赖高级工具类）**

思路：

- 从左到右扫描原始字符串，遇到分隔符就把前面的子串截出来放到列表里；
- 扫描结束后，把列表从后往前拼接，中间再补上分隔符。

```java
public static String reverseDelimitedString(String needReverseValue, String delimiter) {
    if (needReverseValue == null || needReverseValue.isEmpty()) {
        return needReverseValue;
    }
    if (delimiter == null || delimiter.isEmpty()) {
        // 没有明确分隔符，就把整个串当成一个整体
        return needReverseValue;
    }

    java.util.List<String> parts = new java.util.ArrayList<>();
    int len = needReverseValue.length();
    int dLen = delimiter.length();
    int start = 0;

    while (start <= len) {
        int idx = needReverseValue.indexOf(delimiter, start);
        if (idx == -1) {
            // 最后一个子串
            parts.add(needReverseValue.substring(start));
            break;
        } else {
            parts.add(needReverseValue.substring(start, idx));
            start = idx + dLen;
        }
    }

    StringBuilder sb = new StringBuilder();
    for (int i = parts.size() - 1; i >= 0; i--) {
        sb.append(parts.get(i));
        if (i > 0) {
            sb.append(delimiter);
        }
    }
    return sb.toString();
}
```

**参考答案二：使用常见类库方法（`split` + `Collections.reverse`）**

```java
public static String reverseDelimitedStringLib(String needReverseValue, String delimiter) {
    if (needReverseValue == null || needReverseValue.isEmpty()) {
        return needReverseValue;
    }
    if (delimiter == null || delimiter.isEmpty()) {
        return needReverseValue;
    }

    String[] arr = needReverseValue.split(java.util.regex.Pattern.quote(delimiter));
    java.util.List<String> list = java.util.Arrays.asList(arr);
    java.util.Collections.reverse(list);
    return String.join(delimiter, list);
}
```

**问题 2.1 参考答案：客户端请求 key 设计成 String 是否合理？**

- **如果客户端与服务端都是 Java 或其他强类型语言，且 ID 范围在 `Long` 可表示范围内，直接用 `Long` 更合理。**  
  - `key` 在语义上就是主键 ID，用 `Long` 可以在编译期做类型检查，避免把本来是数字的 ID 和其他字符串（比如 `"abc"`）混在一起。  
  - 省去频繁的 `toString` / `parseLong` 转换，比较时也不会因为格式差异导致匹配失败。  
- **如果协议要兼容 JavaScript / 前端，担心 Long 精度问题而统一用字符串 ID，也可以接受，但需要在协议里明确说明「这是 Long ID 的字符串表示」。**  
- 就本题给出的 Java 定义和 `SvrDbOrderItem.key` 为 `Long` 的前提下，更推荐 **客户端同样使用 `Long key`**，把字符串序列化/反序列化交给 HTTP/JSON 层，而不是在业务模型里用 `String`。

一句话总结：**在这道题给定的前提下，把主键 ID 设计成 `String` 不够合理，更好的做法是客户端和服务端都使用 `Long` 类型，除非有跨语言精度的特殊需求。**

**问题 2.2 参考答案：完善 ClientReqOrderItem 协议，实现高可靠更新**

要让客户端通过一批 `ClientReqOrderItem` **高可靠**地更新后台 `SvrDbOrderItem`，协议设计需要解决几个核心问题：

- 区分「新增 / 修改 / 删除」三类操作；  
- 避免并发覆盖（乐观锁）；  
- 避免重复请求导致的重复写入（幂等）。  

可以将 `ClientReqOrderItem` 设计为（示意）：  

```java
class ClientReqOrderItem {
    // 服务端主键，新增时可以为空或 0
    public Long id;

    // 操作类型：C=CREATE, U=UPDATE, D=DELETE
    public String opType;

    // 业务字段
    public String valueField;

    // 版本号，用于乐观锁，防止并发覆盖
    public Long version;

    // 幂等键：一次批量更新中的唯一请求 ID
    public String requestId;
}
```

**协议及服务端处理要点可以这样设计：**

- **操作语义清晰：**
  - 当 `opType = "C"` 时，`id` 可以为空，由服务端生成新 `SvrDbOrderItem` 的主键。  
  - 当 `opType = "U"` 时，必须携带已有的 `id` 和当前 `version`，服务端做「`WHERE id = ? AND version = ?`」的更新，更新成功后 `version` 自增。  
  - 当 `opType = "D"` 时，携带 `id`（和可选的 `version`），服务端做逻辑删除或物理删除。  
- **乐观锁防止覆盖：**  
  - 更新和删除操作都基于版本号，若 `UPDATE ... WHERE id = ? AND version = ?` 影响行数为 0，则说明数据已被其他请求更新，返回版本冲突错误，让客户端重新拉取数据后再提交。  
- **幂等保证高可靠：**  
  - 客户端为每次批量提交生成全局唯一的 `requestId`，每个 `ClientReqOrderItem` 都携带该值。  
  - 服务端在一张「请求幂等表」中记录已经成功处理过的 `requestId`，遇到重复的 `requestId` 直接返回成功，避免网络重试导致的重复写入。  
- **批量处理与结果反馈：**  
  - 服务端按一批 `ClientReqOrderItem` 事务性地处理：要么整批成功，要么整批回滚，保证客户端和服务端列表最终一致。  
  - 返回结果中携带每个成功创建/更新/删除的条目（包括新生成的 `id` 和最新 `version`），便于客户端同步本地状态。

一句话总结：**通过在 `ClientReqOrderItem` 中增加「操作类型 + 版本号 + 幂等键」等字段，并在服务端采用乐观锁和幂等处理，就可以让客户端通过该协议高可靠地对 `SvrDbOrderItem` 做增删改，避免并发覆盖和重复写入。**

**问题 2.3 参考答案：生成新增 / 修改 / 删除的 SvrDbOrderItem 列表**

基于上面第 2 步中设计好的协议（包含 `id`/`opType`/`valueField`/`version` 等字段），可以这样做差异计算：

- 用一个 `Map<主键, SvrDbOrderItem>` 存当前服务端列表，方便 O(1) 查找和删除；  
- 遍历客户端列表：
  - 对于需要新增的数据，调用 `KeyUtils.genKey()` 生成新主键，加入「新增列表」；  
  - 对于需要更新的数据，从 `svrMap` 里取出旧值，如果内容不同则加入「修改列表」；  
  - 每处理完一个已有的服务端条目，就从 `svrMap` 中移除；  
- 遍历结束后，`svrMap` 中剩下的就是客户端已经不存在、需要删除的条目，加入「删除列表」。

可以先定义一个简单的结果封装类：

```java
class DiffResult {
    public List<SvrDbOrderItem> toAdd = new java.util.ArrayList<>();
    public List<SvrDbOrderItem> toUpdate = new java.util.ArrayList<>();
    public List<SvrDbOrderItem> toDelete = new java.util.ArrayList<>();
}
```

然后实现差异计算逻辑（下面以第 2 步改造后的 `ClientReqOrderItem` 为例，假设其中的 `id` 对应 `SvrDbOrderItem.key`）：

```java
public DiffResult func(List<ClientReqOrderItem> clientList, List<SvrDbOrderItem> svrList) {
    DiffResult result = new DiffResult();

    // 1. 先把服务端列表放入 Map，key = 主键
    java.util.Map<Long, SvrDbOrderItem> svrMap = new java.util.HashMap<>();
    for (SvrDbOrderItem item : svrList) {
        svrMap.put(item.key, item);
    }

    // 2. 遍历客户端请求列表
    for (ClientReqOrderItem req : clientList) {
        if ("C".equals(req.opType)) { // 新增
            SvrDbOrderItem add = new SvrDbOrderItem();
            add.key = KeyUtils.genKey();  // 生成新 key
            add.valueField = req.valueField;
            result.toAdd.add(add);
        } else if ("U".equals(req.opType)) { // 修改
            SvrDbOrderItem old = svrMap.get(req.id);
            if (old != null && !java.util.Objects.equals(old.valueField, req.valueField)) {
                SvrDbOrderItem updated = new SvrDbOrderItem();
                updated.key = old.key;
                updated.valueField = req.valueField;
                result.toUpdate.add(updated);
            }
            // 不论是否修改，既然客户端还保留这条记录，就从 map 中移除，后面就不会被当成“删除”
            svrMap.remove(req.id);
        } else if ("D".equals(req.opType)) { // 删除
            SvrDbOrderItem old = svrMap.remove(req.id);
            if (old != null) {
                result.toDelete.add(old);
            }
        }
    }

    // 3. svrMap 中剩余的是客户端已经不存在、但服务端还在的记录，可视为“删除”
    for (SvrDbOrderItem leftover : svrMap.values()) {
        result.toDelete.add(leftover);
    }

    return result;
}
```

说明：

- 上面的实现以第 2 步中新设计的 `ClientReqOrderItem` 为基础（用 `id` + `opType`），能直接对应到数据库主键，并且天然区分新增 / 修改 / 删除。  
- 如果面试官要求**严格按题目原始签名 `public void func(List<ClientReqOrderItem> clientList, List<SvrDbOrderItem> svrList)`**，可以把 `DiffResult` 拆成 3 个 `List` 作为额外的输出参数传入，核心遍历与对比逻辑保持不变即可。
