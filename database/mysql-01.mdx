## 四、数据库（MySQL/ES）

### 1、事务的基本特性（ACID）与隔离级别（面试口述版）

**1）ACID：**

- 原子性（Atomicity）：一个事务里的操作要么全成功，要么全失败，失败就回滚。
- 一致性（Consistency）：一致性指的是数据库总是从一个一致性的状态转换到另外一个一致性的状态。比如 A 转账给 B 100 块钱，假设中间 sql 执行过程中系统崩溃 A 也不会损失 100 块，因为事务没有提交，修改也就不会保存到数据库。
- 隔离性（Isolation）：一个事务的修改在最终提交前，对其他事务是不可见的。
- 持久性（Durability）：一旦事务提交，所做的修改就会永久保存到数据库中。

**2）并发下常见问题：**

- 脏读：读到别的事务未提交的数据。
- 不可重复读：同一事务两次读同一行，结果不一样（别人改了并提交）。
- 幻读：同一个事务里，两次用同样条件查数据，第二次结果集多了（或少了）一些“新行”，像凭空出现/消失一样。

**3）隔离级别（从弱到强）：**

- 读未提交（READ UNCOMMITTED）：可能脏读/不可重复读/幻读。
- 读已提交（READ COMMITTED）：避免脏读，但可能不可重复读/幻读。
- 可重复读（REPEATABLE READ）：避免脏读/不可重复读；MySQL InnoDB 默认。
- 串行化（SERIALIZABLE）：最严格，基本避免所有并发问题，但并发性能最差。

**4）那你说说什么是幻读？什么是 MVCC？（口述版）**

- 幻读：同一个事务里，两次用同样条件查数据，第二次结果集多了（或少了）一些“新行”，像凭空出现/消失一样。和不可重复读区别：不可重复读是“同一行的值变了”；幻读是“多了/少了整行”。
- MVCC，也叫多版本并发控制，核心目标是，在并发事务里让读操作尽可能不加锁、提高并发性能，同时仍然保证隔离性。底层的实现，是通过数据库给每条记录保留多个版本，让每个事务按自己的“快照”去读数据。InnoDB 里一般是通过隐藏字段 + undo log + Read View 来实现。

一句话总结：幻读=同条件两次查多/少了行；MVCC=多版本+快照读，提高并发。

一句话总结：ACID 记四个词；隔离级别从 RU -> RC -> RR -> Serializable，级别越高越安全但并发越差。

### 2、数据库三范式（面试口述版）

- 第一范式（1NF）：**字段要拆开**，一列只放一种信息，比如姓名和手机号分两列放。
- 第二范式（2NF）：**完全依赖主键**，非主键不能只依赖“联合主键的一部分”，否则就拆表或改成单一主键。
- 第三范式（3NF）：**只依赖主键**，非主键之间不要再互相依赖，比如不要在班级表里既放“老师 ID”又放“老师姓名”，姓名应该去老师表。
- 口诀版：1NF 拆字段，2NF 看“是否完全依赖主键”，3NF 看“是否只依赖主键”；实际设计里在范式和查询性能之间平衡，必要时会适当冗余。

### 3、什么情况下索引可能失效/不走索引

常见原因：

- 联合索引不满足最左前缀。`idx(a, b, c)`；`select * from t where b = 2;`
- 对索引列做函数/运算/类型转换（如字符串列用数字比较）。`select * from t where date(create_time) = '2026-01-24';`
- like 以 % 开头。`select * from user where name like '%mi%';`
- 使用 OR 且部分条件无索引（可能导致全表扫描）。假设 a 有索引，b 没索引；`select * from t where a = 1 or b = 2;`
- 选择性太差/回表成本高，优化器可能选择全表扫描。假设 gender 只有男/女两种值，即使建了索引也可能不走；`select * from user where gender = '男';`
- join 字段字符集/类型不一致可能导致转换，从而影响索引使用。  
  t1.user_id 是 varchar(32)（有索引）  
  t2.user_id 是 int（有索引）  
  可能不走索引：

```sql
select *
from t1
join t2 on t1.user_id = t2.user_id;
```

### 4、如何排查慢 SQL

先定位是哪条 SQL 慢：看慢查询日志/监控，确认具体 SQL、执行次数和耗时。  
再看为什么慢：用 EXPLAIN 看执行计划，重点看有没有走索引、扫描了多少行、有没有排序/临时表。  
最后对症优化：没走索引/索引用得不好 → 补索引/调索引顺序，SQL 写法导致索引用不上 → 改写 SQL，如果是锁等待就查事务和锁。（分页太深就改成游标/延迟关联）

1）先定位：开启慢查询日志/监控，确认慢 SQL、频率、平均/TP99。  
2）看执行计划：EXPLAIN 关注 type、key、rows、Extra（Using filesort/temporary）。  
3）常见优化：

- 建/改索引（覆盖索引、联合索引顺序）。
- 改写 SQL（避免函数、避免 %like、拆分 OR、减少返回列）。
- 控制分页深度（延迟关联/基于游标）。
- 检查锁等待与事务隔离级别。

### 5、常见索引类型及特点

**按用途（你建索引是为了解决什么问题）：**

**1）约束类：**

- 主键索引（PRIMARY KEY）：唯一且非空，一个表只能有一个主键索引；适用场景：唯一标识记录，如用户 ID、订单 ID 等。
- 唯一索引（UNIQUE）：值唯一，允许为空（但最多一个 NULL）；适用场景：需要唯一约束字段的，如邮箱、手机号等。

**2）查询加速类：**

- 普通索引（INDEX）：无唯一性约束，允许重复值；适用场景：频繁查询的字段，如商品名称、分类 ID 等。
- 联合索引（Composite Index）：多个字段组合而成，遵循最左前缀原则；适用场景：多条件组合查询，如 (user_id, create_time)。

**3）特殊检索类：**

- 全文索引（FULLTEXT）：用于文本内容检索，支持分词搜索；适用场景：文章内容、商品描述等长文本字段的搜索。
- 空间索引（SPATIAL）：用于地理空间数据类型（如 POINT、GEOMETRY）；适用场景：地理位置相关查询，如附近的商家、范围查询。

**按底层结构（索引是怎么存的）：**

**B+ 树索引**

- B+ 树是数据库最常用的一种索引数据结构，特别是在 MySQL InnoDB 引擎里默认使用的就是 B+ 树索引。
- 它是一种自平衡的多路搜索树：内部节点只存键，叶子节点存储键（＋数据或指针），而且所有叶子节点通过链表顺序连接，方便范围查询。
- B+ 树的查找、插入、删除复杂度都是对数级别 O(log N)，可以高效处理大量数据。
- B+ 树非常适合于范围查询和排序（例如 BETWEEN、&gt;、&lt; 等），因为数据在叶子节点是按顺序链接的。

**Hash 索引**

- Hash 索引是另一种索引结构，它通过对索引列计算哈希值来快速定位数据。
- Hash 索引对于等值查询（如 =、&lt;=&gt;）非常快，理论上可以达到常数时间查找 O(1)。
- 但是 Hash 索引不能用于范围查询（比如 &lt;、&gt;、BETWEEN），因为哈希结构本身不保存排序关系。
- 在 MySQL 中只有某些存储引擎才支持 Hash（比如 MEMORY 引擎），InnoDB 默认使用的是 B+ 树。

**根据数据的物理存储方式：**

聚簇索引  
非聚簇索引

**索引设计原则：**

- 遵循最左前缀：联合索引按顺序使用字段；
- 选择高选择性字段：区分度高的字段更适合建索引；
- 控制索引数量：过多索引会影响插入/更新性能；
- 考虑覆盖索引：查询字段都在索引中，避免回表；
- 避免频繁更新的字段：更新会导致索引重建。

### 6、聚簇索引 vs 非聚簇索引（InnoDB 口述版）

**聚簇索引（Clustered Index）**

- 聚簇索引是一种索引+数据物理顺序存储合二为一的结构。
- 在 MySQL InnoDB 中，主键索引就是聚簇索引：B+ 树的叶子节点直接存储行数据（整行数据），同时按主键顺序物理存放在磁盘上。
- 因为数据本身按索引顺序排列，所以范围查询效率更高、IO 更少。
- 一个表最多只能有一个聚簇索引（因为数据只能按一个顺序存储）。

✅ 简单理解：聚簇索引把索引和真实数据合并在一起了，找一次索引就直接定位到数据，不需要再去查表。

**非聚簇索引（Non‑Clustered Index）**

- 非聚簇索引是索引和数据分开存放的结构。
- 在 MySQL InnoDB 中，非主键列创建的索引（辅助索引/二级索引）就是非聚簇索引：B+ 树的叶子节点只存储索引键＋对应的主键值（或指向行的指针），查到这个主键值后，需再去聚簇索引里根据主键查一趟数据（这个过程叫“回表”）。
- 一个表可以有多个非聚簇索引。

✅ 简单理解：非聚簇索引是“辅助索引”，不存行数据，查完索引后需要再去聚簇索引（主键）拿数据。

### 7、什么是回表？什么是覆盖索引？（口述版）

- **回表**：当查询用的是一个普通的二级索引（不是主键索引），数据库找到了匹配的主键，再根据主键回到聚簇索引把整行数据取出来，这一步就叫回表。  
  假设有这么一张表：

```sql
CREATE TABLE user (
  id INT PRIMARY KEY,
  name VARCHAR(20),
  age INT,
  INDEX(name)
);
```

- 你查询：`SELECT id FROM user WHERE name = 'Tom';`  
  因为 id 在索引里（在二级索引叶子节点存有主键值），所以不用去主表取数据 ❗ — 不回表。
- 你查询：`SELECT age FROM user WHERE name = 'Tom';`  
  age 不在 name 索引里，只在主表数据里。所以先通过二级索引找到记录对应的主键值，然后再回到主表去取 age 值 — 这就是回表。

📌 所以 回表 = 索引查到主键 → 再根据主键到表里取完整数据。

- **覆盖索引**：一次查询需要的字段都在同一个索引里（通常是二级索引里就包含了 select 的列），直接从索引返回结果，不需要回表。  
  举个例子继续上面的表：  
  `SELECT id, name FROM user WHERE name = 'Tom';`  
  这个查询需要的数据是 id 和 name。如果你有一个索引 (name, id)（顺序放一起）：索引里已经有 name 和 id。查询时数据库就可以只靠这个索引就拿到结果，不需要去主表找别的字段 → 覆盖索引。

### 8、SQL 调优经验（极简记忆版）

**核心口诀**

- 索引优化：高选最左，覆盖少回表
- SQL 写法：避函数，少 %like，拆 OR，限返回
- 执行计划：看 type，查 rows，避 filesort
- 事务锁：短事务，合理隔离，避死锁
- 表结构：分库分表，合理类型，用分区
- 其他：缓存热点，读写分离，预计算

**关键要点（一句话总结）：**

1. 索引：选区分度高的字段，联合索引按最左前缀，用覆盖索引减少回表。
2. SQL：索引列别用函数，少用 `%like`，OR 拆成 UNION，只查必要字段。
3. 执行计划：type 越左越好（const &gt; eq_ref &gt; ref &gt; range &gt; index &gt; ALL），rows 越少越好，避开 filesort/temporary。
4. 事务：缩小事务范围，用读已提交，避免长事务，统一锁顺序。
5. 表结构：大表分库分表（如超过 1000 万行）时，选合适数据类型（INT 存数字），按时间分区，定期清理历史数据。垂直分库分表（基于“列”的拆分），水平分库分表（基于“行”的拆分）
6. 技巧：Redis 缓存热点，主从读写分离，统计数据预计算。

### 9、分页查询怎么实现？深分页为什么慢？怎么优化？（面试口述版）

**基本写法：**  
用 `LIMIT offset, size`，例如查第 2 页、每页 10 条：`SELECT * FROM t ORDER BY id LIMIT 10, 10`。offset 表示跳过多少条，size 表示取多少条。

**深分页为什么慢：**  
offset 很大时（比如 `LIMIT 100000, 10`），MySQL 实际会**先扫描并丢弃前 100000 行**，再返回 10 行。即使有索引，也要沿着索引扫过 100010 行，再丢弃前 100000，代价高、越往后越慢。

**常见优化：**

- **延迟关联（子查询）**：先按索引查出当前页的 id，再根据 id 回表查完整行。例如：
  - `SELECT * FROM t WHERE id IN (SELECT id FROM t ORDER BY id LIMIT 100000, 10);`
  - 内层只走索引、只拿 id，外层再用 id 查，避免对大量行回表。
- **基于上次最大 id（游标分页）**：不用 offset，用上一页最后一条的 id 作为条件。例如第二页：`SELECT * FROM t WHERE id > 上一页最后一条的 id ORDER BY id LIMIT 10`。适合“往下翻”的场景，不能跳页，但每页都很快。
- **业务上限制**：禁止跳到最后几页（如只允许前 N 页）、用搜索引擎（如 ES）做深分页。

**面试一句话：** 分页用 LIMIT offset, size；深分页慢是因为要扫过并丢弃前 offset 行；优化用延迟关联或基于上次最大 id 的游标分页，必要时限制深度或交给 ES。

### 10、MySQL 数据字段类型区别？（面试口述版）

**整数：int vs bigint**  
- **int**：4 字节，有符号约 ±21 亿，无符号约 0～42 亿。一般业务主键、数量够用。  
- **bigint**：8 字节，范围更大。主键要全局唯一、或数量/ID 可能超 int 时用 bigint。

**字符串：varchar vs char**  
- **varchar**：变长，按实际长度存（再加少量长度信息），省空间，适合长度不固定的（姓名、地址等）。有长度上限（如 65535 字节，和字符集有关）。  
- **char**：定长，不足会补空格，适合长度固定的（手机号、固定编码）。查询时通常会自动 trim 尾部空格。

**日期时间：datetime vs timestamp**  
- **datetime**：存“年月日时分秒”，不涉及时区，存什么取什么。范围约 1000–9999 年。  
- **timestamp**：存的是 UTC 时间戳，占 4 字节；会按会话时区转成当地时间显示。范围约 1970–2038 年。跨时区或要省空间用 timestamp，要“所见即所存”用 datetime。

**大文本/二进制：text vs blob**  
- **text**：存大文本（字符），有 text、mediumtext、longtext，区别是最大长度。  
- **blob**：存二进制（图片、文件流等），同样有 blob、mediumblob、longblob。  
- 注意：大字段会拉长行、影响缓冲和性能，能放对象存储就尽量只存 URL。

**小数：decimal vs float/double**  
- **decimal**：定点数，精确存小数，适合金额、精确计算。例如 `decimal(10,2)` 表示总共 10 位、小数 2 位。  
- **float/double**：浮点数，有精度损失，不适合金额。适合科学计算、对精度要求不高的场景。

**面试一句话：** 整数看范围选 int/bigint；字符串变长用 varchar、定长用 char；日期要时区用 timestamp、否则 datetime；金额用 decimal；大文本/二进制用 text/blob 但要慎用、能外存就外存。

### 11、MySQL 的隔离级别？（面试口述版）

**四种隔离级别（从弱到强）：**

| 隔离级别           | 脏读 | 不可重复读 | 幻读 |
|--------------------|------|------------|------|
| 读未提交 RU        | 可能 | 可能       | 可能 |
| 读已提交 RC        | 不会 | 可能       | 可能 |
| 可重复读 RR        | 不会 | 不会       | InnoDB 下通过 MVCC+间隙锁 基本避免 |
| 串行化 Serializable | 不会 | 不会       | 不会 |

- **读未提交（READ UNCOMMITTED）**：能读到别的事务未提交的数据，脏读/不可重复读/幻读都可能发生，一般不用。  
- **读已提交（READ COMMITTED）**：只读已提交的数据，避免脏读；同一事务内两次读同一行可能得到不同结果（不可重复读），也可能多/少行（幻读）。  
- **可重复读（REPEATABLE READ）**：MySQL InnoDB **默认**。同一事务内多次读同一行结果一致，且通过 MVCC + 间隙锁 在 InnoDB 下能避免幻读。  
- **串行化（SERIALIZABLE）**：最严格，读写都加锁，基本不会脏读/不可重复读/幻读，但并发最差。

**三个问题再记一遍：**  
- **脏读**：读到未提交的数据。  
- **不可重复读**：同一事务内同一行两次读结果不一样（被别的事务改并提交了）。  
- **幻读**：同一事务内同条件两次查，结果集行数变了（多了或少了行）。

**面试一句话：** 四种级别 RU / RC / RR / Serializable；InnoDB 默认 RR；脏读/不可重复读/幻读要能区分；级别越高越安全、并发越差。

### 12、什么情况下索引会失效？（口述版）

常见几类：  
- **联合索引不满足最左前缀**：例如索引是 (a, b, c)，只查 `WHERE b = 2` 或 `WHERE c = 3` 用不到这个联合索引，必须从最左列 a 开始用。  
- **对索引列做函数、运算或隐式类型转换**：例如 `WHERE date(create_time) = '2026-01-24'`、`WHERE id + 1 = 10`，或者字符串列和数字比较导致隐式转换，优化器无法用索引。  
- **like 以 `%` 开头**：例如 `WHERE name LIKE '%mi%'`，前面不确定就无法按索引有序扫描，往往退成全表扫描。  
- **OR 一侧无索引**：例如 `WHERE a = 1 OR b = 2`，若 b 无索引，优化器可能选择全表扫描。可改成 UNION 或分别走索引再合并。  
- **选择性太差**：例如性别只有男/女，建了索引优化器也可能认为回表成本高而不用。  
- **join 两表字段类型/字符集不一致**：会导致隐式转换，可能用不上索引。

面试一句话：最左前缀、别对索引列做函数/类型转换、like 别左模糊、OR 注意两侧索引、低选择性慎建索引。

### 13、聚簇索引和非聚簇索引区别？回表、覆盖索引？（口述版）

- **聚簇索引**：在 InnoDB 里，主键索引就是聚簇索引。叶子节点直接存**整行数据**，数据和索引按主键顺序存在一起。一个表只有一个聚簇索引。查主键时一次就能拿到整行，不需要再找别的结构。  
- **非聚簇索引**：即二级索引/辅助索引。叶子节点只存**索引列的值 + 主键值**，不存整行。用二级索引查到主键后，还要再根据主键到聚簇索引里查一次才能拿到完整行，这一步就叫**回表**。  
- **覆盖索引**：如果一次查询需要的列（select 和 where 里用到的）**全部出现在某个索引里**，就不用回表，直接从索引返回结果。例如有索引 (name, id)，查 `SELECT id, name FROM user WHERE name = 'Tom'`，id 和 name 都在索引里，就不回表，性能更好。

面试一句话：聚簇存整行、非聚簇存键+主键要回表；覆盖索引让查询列都在索引里，避免回表。

### 14、B+ 树和 Hash 索引区别？（口述版）

- **B+ 树**：多路平衡搜索树，叶子节点存键（和行数据或主键），且叶子按顺序串成链表。支持**范围查询**（`BETWEEN`、`>`、`<`）和 **ORDER BY**，因为数据有序。InnoDB 默认用 B+ 树，查找/插入/删除都是 O(log N) 级别。  
- **Hash 索引**：对键做哈希，等值查询理论 O(1)，但**不支持范围查询、不支持排序**，因为哈希不保序。MySQL 里只有部分引擎（如 MEMORY）支持，InnoDB 不支持 Hash 索引。

面试一句话：B+ 树支持范围和排序、InnoDB 默认；Hash 等值快但不支持范围，InnoDB 不用。

### 15、ACID 分别是什么？幻读和不可重复读区别？MVCC 是什么？（口述版）

- **ACID**：原子性（事务要么全成功要么全回滚）、一致性（数据从一种一致状态变到另一种一致状态）、隔离性（未提交的修改对其他事务不可见）、持久性（提交后持久落盘）。  
- **不可重复读**：同一事务内，对**同一行**两次读，结果不一样（被别的事务改并提交了）。  
- **幻读**：同一事务内，用**同一条件**两次查，结果集**行数**变了，多了或少了行。区别在于：不可重复读是“同一行值变了”，幻读是“多了/少了行”。  
- **MVCC**：多版本并发控制。通过给记录保留多个版本、每个事务按自己的“快照”（Read View）读，让读不加锁也能保证隔离性，提高并发。InnoDB 用隐藏字段 + undo log + Read View 实现。

面试一句话：ACID 四性各一句；不可重复读是同行值变、幻读是行数变；MVCC 是多版本+快照读提并发。

### 16、慢 SQL 怎么排查？EXPLAIN 看什么？SQL 调优常见手段？（口述版）

- **排查步骤**：先通过慢查询日志或 APM 监控**定位**是哪条 SQL 慢、执行次数和耗时（平均/TP99）；再用 **EXPLAIN** 看执行计划，重点看 **type**（访问方式，const/ref/range/index/ALL 越前越好）、**key**（实际用的索引）、**rows**（预估扫描行数）、**Extra**（是否有 Using filesort、Using temporary 等）；最后根据结果改索引或改 SQL。  
- **调优手段**：建/改索引（联合索引最左前缀、用覆盖索引少回表）；改写 SQL（索引列避免函数、避免 `%` 开头的 like、大 OR 改 UNION、只查需要的列）；控制分页深度（延迟关联、游标分页）；缩短事务、避免长事务占锁；读写分离、热点用缓存。

面试一句话：先定位慢 SQL，再 EXPLAIN 看 type/key/rows/Extra，对症改索引或改 SQL；调优围绕索引、写法、分页、事务、架构。

### 17、三范式是什么？实际会反范式吗？主键用自增还是 UUID？（口述版）

- **三范式**：1NF 列不可再分，一列只存一种含义；2NF 非主键完全依赖主键（不能只依赖联合主键的一部分）；3NF 非主键只依赖主键、不依赖其他非主键。实际设计里会**适当反范式**，用冗余换查询性能（如多表合并、冗余常用字段）。  
- **主键自增 vs UUID**：**自增**顺序写入、对 B+ 树友好、占空间小，但单机或单库内唯一、可能暴露业务量；**UUID** 全局唯一、适合分布式，但无序、插入可能造成页分裂、占空间大。按是否分布式、是否要隐藏递增关系来选；也有用雪花算法等折中方案。

面试一句话：三范式是 1NF 拆列、2NF 完全依赖主键、3NF 只依赖主键；实际会冗余换性能；主键自增简单高效、UUID 全局唯一但占空间且无序。
