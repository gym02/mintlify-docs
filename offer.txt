技术面试

一、Java 基础
1、方法重载（Overload）和方法重写（Override）的区别
- 重载：同一个类中，方法名相同，但“参数列表”不同（参数个数/类型/顺序不同）。返回值不同不构成重载。
- 重写：发生在继承体系中，子类对父类的实例方法提供新的实现。方法名和参数列表必须一致；返回值可协变；访
  问权限不能更严格；抛出异常范围不能更大。

2、自动装箱/拆箱原理与常见问题
- 装箱：基本类型 -> 包装类型，例如 Integer i = 1，本质是 Integer.valueOf(1)（可能走缓存）。
- 拆箱：包装类型 -> 基本类型，例如 int x = i，本质是 i.intValue()。
常见问题：
1）NPE：包装类型为 null 时拆箱会触发空指针。
2）性能：频繁装箱/拆箱会带来额外对象创建与 GC 压力，尤其在循环中。
3）比较陷阱：包装类型用 equals 比值；用 == 比的是引用。Integer 在 [-128,127] 有缓存，可能导致 == 表现“看起来相等”。
（JDK 会把一小段范围内的 Integer 对象提前创建好并复用 ，避免频繁 new 对象。可以通过 JVM 参数 -XX:AutoBoxCacheMax=<size> 来调整缓存的上限，下限固定为 -128。）

3、JDK 1.7 vs 1.8 HashMap 区别
1）数据结构：
- 1.7：数组 + 链表。
- 1.8：数组 + 链表/红黑树. 树化条件：当链表长度 >= 8 且 数组 table 自动扩充到 容量 >= 64 才会树化；当树节点数 <= 6 会退化回链表。
2）插入方式：
- 1.：7头插法（扩容迁移时，在多线程情况下可能形成环）。
多线程下还是使用concurrentHashMap，它用synchronized结合CAS（比较并交换）更细粒度的锁。具体来说就是只对正在操作的链表头节点，或者
红黑树的根节点加锁，其他线程可以同时操作其他的节点。同时，它还通过volatile关键字保证了节点的可见性。即避免线程缓存问题，
保证一个线程改了value时，其他线程也能看到这个修改。
- 1.8：尾插法（降低上述风险）。
3）扩容迁移：
- 1.7：都通过 rehash 后重新计算索引。
- 1.8：通过位运算判断是否重新计算哈希和索引，减少计算。

补充：hashmap 自动扩充过程：hashmap默认容量是16，负债因子是0.75，所以当元素超过12时，就会触发扩容，扩容的时候，新容量是旧容量的两倍。
16变成32。然后会创建一个的新的数组，把旧数组里的元素放到新数组里，用新数组替换旧数组，完成扩容。

4、ArrayList 和 Vector 的区别
- 线程安全：ArrayList 非线程安全；Vector 方法大多是 synchronized，线程安全但性能更差。
- 扩容策略：ArrayList 默认长度是10，通常按 1.5扩 倍容；Vector 默认长度是10，默认 2 倍（或按 capacityIncrement）。
实际开发中：需要线程安全更常用 Collections.synchronizedList 或 CopyOnWriteArrayList（读多写少）。

补充：ArrayList 的扩容机制（以 JDK 1.8 为例）
- 初始容量：无参构造第一次 add 时扩成 10，有参构造则用你指定的容量。
- 触发时机：每次 add 之前会检查容量，若当前 size + 1 > 数组长度，就触发扩容。
- 新容量计算：newCapacity = oldCapacity + (oldCapacity >> 1)，也就是 1.5 倍扩容；如果还不够容纳最小需要容量，会取“需要容量”和 1.5 倍中的更大者。
- 扩容过程：新建一个更大的数组，把旧数组里的元素用 System.arraycopy 复制过去，再用新数组替换旧数组。

5、synchronized 和 ReentrantLock 的区别
用法上：
    - synchronized ：写在方法/代码块上，出了作用域自动释放锁
    - ReentrantLock ：需要 lock() / unlock() ，一般必须写在 finally 里
功能上nReentrantLock synchronized 比 多这些能力：
1） 可中断，线程在等锁时可以被 interrupt() 打断（避免一直死等）。
2） tryLock，定时获取锁 。可以“拿不到就算了”或“等一会儿拿不到就放弃”，更适合做降级/避免死锁。
3） 公平锁 ，可以选择公平（先来先得）或非公平（默认，吞吐更高）。synchronized 只有非公平的感觉。
4） 多个条件队列 Condition。一个锁可以配多个 Condition （比如“队列不空”“队列不满”分开等）。 
synchronized 只有一个隐式条件队列（配合 wait/notify ，粒度更粗）。
底层实现上：
- synchronized ：JVM 帮你管理锁（monitor），你只管写关键字
- ReentrantLock ：Java 层自己实现了一套锁机制（AQS），用一个状态值 + 队列来管理“谁拿到锁、谁在排队”，并用 CAS 做原子更新

- 只要互斥、代码简单：优先 synchronized
- 需要 tryLock/超时/可中断/公平/多个条件队列：用 ReentrantLock

6、什么是 CAS
CAS（Compare And Swap）是一种无锁原子。操作它比较内存里的值，是否和上次看到的值是一样的，若相等则更新为新值，
否则，说明被别人改过了，我这次更新失败，通常会 重试 （自旋）。
作用：主要用来做 无锁并发 ，减少 synchronized 那种阻塞、唤醒带来的开销。
问题：CAS 的问题主要是 ABA、竞争激烈时自旋耗 CPU、以及只能保证单变量原子更新

7、锁的类型有哪些（面试口述版）
按 Java 并发里常见的分类，你可以从 4 个维度讲：
1）按是否阻塞：
- 悲观锁：假设“肯定会冲突”，所以先把资源锁住再操作，别的线程只能阻塞等待（典型如 synchronized、ReentrantLock）。
- 乐观锁：假设“多数时候不会冲突”，不一开始就加锁，只是在更新时用版本号/CAS 检查是否被人改过，若失败就重试（典型如 Atomic 类、基于版本号的乐观锁字段）。

2）按是否可重入：
- 可重入锁：同一线程拿到锁后可以再次进入（synchronized、ReentrantLock）。
- 不可重入锁：拿到锁后再次进入会锁死（了解即可）。

3）按是否公平：
- 公平锁：先来先得（ReentrantLock 可配置）。
- 非公平锁：允许插队，吞吐更高（synchronized 更偏非公平）。

4）按读写场景：
- 互斥锁：读写都互斥（synchronized、ReentrantLock）。
- 读写锁：读共享、写互斥（ReentrantReadWriteLock），适合读多写少。

一句话总结：悲观/乐观、可重入/不可重入、公平/非公平、互斥锁/读写锁。

5）按 JVM 对 synchronized 的优化（了解即可）：
- 偏向锁：基本只有一个线程用锁时，几乎不做竞争开销。
- 轻量级锁：有少量竞争时，先用 CAS + 自旋尝试拿锁。
- 重量级锁：竞争激烈时，线程会阻塞/唤醒，开销最大。
- 自旋锁：不立刻阻塞，先在 CPU 上循环等一会儿（适合持锁时间很短）。

6）并发库里常见的“锁/同步工具”（扩展）：
- StampedLock：读写的增强版锁，支持乐观读。
- Semaphore / CountDownLatch / CyclicBarrier：严格说不是锁，但也是并发控制常用工具。

8、Java 常见集合（面试口述版：讲得懂）
先把“场景”记住：
- List：要顺序、允许重复
- Set：要去重
- Map：要 key-value，通过 key 找 value

1）List（有序、可重复）
- ArrayList：底层数组。
  你可以这样理解：数组下标定位，所以“查快”；但中间插入/删除要搬移，所以“改慢”。
- LinkedList：底层双向链表。
  你可以这样理解：改指针所以“改快”；但要从头/尾一路找，所以“查慢”。
- Vector：线程安全版 ArrayList（方法加 synchronized），但性能一般，现在更多用并发容器替代。

2）Set（不可重复）
- HashSet：底层就是 HashMap。
  你可以这样理解：把元素当成 HashMap 的 key，所以天然去重；无序；查找快。
- LinkedHashSet：在 HashSet 基础上多维护一条链表，所以“按插入顺序有序”。
- TreeSet：底层红黑树，所以“按排序有序”，适合范围查询，但比 HashSet 慢一些。

3）Map（key 不重复）
- HashMap：最常用；无序；查找快；线程不安全；允许 key 为 null。
- LinkedHashMap：在 HashMap 基础上维护顺序（插入顺序/访问顺序），常用来做 LRU。
    LRU（Least Recently Used）：最近最少使用淘汰策略。
    你可以这样理解：缓存满了就把“最久没被访问”的那条数据删掉，保留最近常用的。
    LinkedHashMap 可以用 accessOrder=true 维护访问顺序，再配合 removeEldestEntry 实现 LRU。
- TreeMap：红黑树；key 排序有序；适合范围查询。
- Hashtable：线程安全（synchronized），老旧；现在更多用 ConcurrentHashMap。
- ConcurrentHashMap：并发场景首选（更细粒度的并发控制）。

4）遍历集合的常见方式
- 传统 for 下标遍历：适合 List，按索引随机访问；不适合 Set/Map。
  for (int i = 0; i < list.size(); i++) {
      String s = list.get(i);
      System.out.println(s);
  }
- 增强 for-each：语法最简单，适合大多数“只读遍历”场景。
  for (String s : list) {
      System.out.println(s);
  }
- Iterator：显式拿迭代器，边遍历边删除元素时用它的 remove，避免 ConcurrentModificationException。
  Iterator<String> it = list.iterator();
  while (it.hasNext()) {
      String s = it.next();
      if (s.isEmpty()) {
          it.remove(); // 遍历时安全删除
      }
  }
- forEach / Stream：Java 8 之后更常见，配合 Lambda 做过滤、映射、聚合，更适合集合的“处理流水线”式写法。
  list.stream()
      .filter(s -> s.length() > 3)
      .forEach(System.out::println);

一句话口诀：数组查快；链表改快；要有序用 Linked；要排序用 Tree；要并发用 Concurrent。

9、Java 8 新特性（面试口述版：从 0 讲清楚）
Java 8 最核心的变化是引入了函数式编程的思想，整体目标是让代码更简洁、可读性更好，同时提升对并行处理的支持。
在实际使用中，感受最明显的主要有几个方面。

1） Lambda 表达式
它本质上是对函数式接口的简化写法。经常配合集合操作或线程使用，提高可读性。

2）函数式接口与 @FunctionalInterface
Java 8 明确提出了“函数式接口”的概念：即只包含一个抽象方法的接口，来配合 Lambda 的使用。
比如 Runnable、Callable、Comparator 都是典型的函数式接口。 new Thread(() -> System.out.println("run")).start();

3）方法引用（Lambda 的进一步简化）
它其实是对Lambda 的进一步简化，当我们去“调用一个已有方法”时，可以用 :: 。比如 类名::静态方法、对象::实例方法、类名::实例方法、类名::new。

4） Stream API
让我们可以通过链式调用来完成过滤、聚合等操作，而不需要手动控制循环过程。
Stream 的操作分为中间操作和终止操作，只有在执行终止操作时才会真正计算，这种惰性执行的机制提供了良好的性能。
另外，Stream 还天然支持并行流，在合适的场景下可以比较方便地利用多核 CPU。
List<String> names = users.stream()
    .filter(u -> u.getAge() > 18)   // 这里的 u -> ... 是 Lambda
    .map(User::getName)             // 这里的 User::getName 是方法引用（也是 Lambda 的一种写法）
    .collect(Collectors.toList());


5） Optional 类
Optional 主要是为了解决空指针异常的问题，它通过显式地表达“值可能不存在”这一语义，迫使开发者在使用之前进行处理。
在项目中，Optional 通常用于方法的返回值，可以减少大量的 null 判断，让代码意图更加清晰，不过一般不太建议作为成员变量或方法参数来使用。

6）新的日期时间 API（java.time 包）
相比之前的 Date 和 Calendar，java.time 包，是线程安全的，而且在语义上更加清晰。

7）接口默认方法（default method）
Java 8 允许接口写 default 方法（带实现）。
意义：接口升级加新方法时，不想让所有实现类都改。
标准库里 Collection 新增的 forEach、stream 等就是典型例子。

8）CompletableFuture（并发能力增强）
相比传统 Future你提交任务后，只能“等/轮询”拿结果，编排能力弱。
CompletableFuture 提供了更好用的异步编排能力，它可以把“完成后做什么、异常怎么处理、多个任务怎么组合”都写成一条链。

10、== 和 equals 区别？hashCode 必须满足什么约定？（面试口述版）
1）== 和 equals 区别
- ==：
  - 基本类型：比“值”。
  - 引用类型：比“引用地址”（是不是同一个对象）。
- equals：默认（Object.equals）也是比地址；但很多类（String、Integer、BigDecimal 等）重写了 equals，用来比“内容”。

2）hashCode 的约定（必须背这 3 条）
- 如果 a.equals(b) 为 true，那么 a.hashCode() 必须等于 b.hashCode()。
- 如果 a.hashCode() 不等于 b.hashCode()，那么 a.equals(b) 一定为 false。
- 如果 a.equals(b) 为 false，hashCode 可以相等（哈希冲突），但会影响性能。

3）面试常追问：重写 equals 为啥必须重写 hashCode
- 如果不重写 hashCode，会用 Object.hashCode()，它通常跟“对象标识/内存地址”相关。new User(1) 和 new User(1) 即使内容一样，默认 hashCode 也大概率不同。
  哈希容器先用 hashCode 定位桶，再用 equals 判断相等；equals 相等但 hashCode 不等，会进不同桶，导致去重失败/查不到。

11、final / finally / finalize 区别？（面试口述版）
- final：关键字。
  - 修饰类：类不能被继承。
  - 修饰方法：方法不能被重写。
  - 修饰变量：变量只能赋值一次（引用不可变，但对象内容可能可变）。

- finally：异常处理里的代码块。
  - 不管是否抛异常，通常都会执行（常用来释放资源：close/unlock）。
  - 例外：JVM 直接退出（System.exit）、进程崩溃等情况可能不执行。

- finalize：Object 的方法（已不推荐/逐步废弃）。
  - GC 之前可能会调用，用来做“最后清理”，但不保证及时执行，也不保证一定执行。
  - 现在更推荐 try-with-resources / Cleaner / 显式 close。

12、static 关键字：静态变量/静态方法/静态代码块加载时机？（面试口述版）
static 是类级别的，全类共享一份。第一次用到类时会触发类初始化，static 变量和 static 代码块只执行一次。

什么时候会触发类初始化（常见场景，背这几个就够）：
- new 这个类的对象
- 访问/修改这个类的静态变量（final 常量除外）
- 调用这个类的静态方法
- 反射使用这个类（Class.forName / 反射创建对象等）
- 初始化子类时，会先初始化父类

子类初始化/实例化的完整顺序（面试常问，背这个）：
1）父类：静态变量/静态代码块（只执行一次）
2）子类：静态变量/静态代码块（只执行一次）
3）父类：实例变量/实例代码块（每 new 一次执行一次）
4）父类：构造方法
5）子类：实例变量/实例代码块
6）子类：构造方法
一句话口诀：父静态 → 子静态 → 父实例 → 父构造 → 子实例 → 子构造。

static 方法属于类，不走对象多态，所以不能 override。

13、接口 vs 抽象类区别？什么时候用接口，什么时候用抽象类？（面试口述版）
1）接口（interface）
- 更像“能力/规范”：强调你能做什么。
- 支持多实现（一个类可以实现多个接口）。
- Java 8 以后接口可以有 default 方法，但核心还是“规范”。

2）抽象类（abstract class）
- 更像“模板/父类”：强调你是什么。
- 只能单继承（extends 一个抽象类）。
- 可以有成员变量、构造方法、抽象方法 + 已实现方法，适合复用公共代码。

3）怎么选（面试一句话）
- 需要“多继承能力/解耦/定义规范” → 用接口。
- 需要“复用公共代码/统一模板/有状态字段” → 用抽象类。

14、深拷贝 vs 浅拷贝？Cloneable 怎么用？为什么不推荐？（面试口述版）
1）浅拷贝（Shallow Copy）
- 只复制对象本身；对象里的引用字段仍指向同一份子对象。

2）深拷贝（Deep Copy）
- 不仅复制对象本身，还会把引用的子对象也复制一份，互不影响。

3）Cloneable 怎么用（了解即可）
- 实现 Cloneable 接口，并重写 clone()（通常调用 super.clone()）。
- 注意：super.clone() 默认是浅拷贝；要深拷贝需要手动 clone 子对象。

4）为什么不推荐 clone
- 语义不清晰（浅拷贝/深拷贝容易踩坑）。
- 需要处理 CloneNotSupportedException。
- 容易破坏封装（直接绕过构造方法）。
- 实际更常用：拷贝构造器、静态工厂、序列化/JSON（看场景）。

15、反射是什么？优缺点？常见使用场景（面试口述版）
1）反射是什么
- 程序在运行时，可以获取类的结构信息，并动态地创建对象、调用方法、访问属性的一种机制。
  比如通过 Class.forName() 拿到类的元数据。

2）优点
- 灵活：灵活性强，可以做通用框架、解耦（不依赖具体类）。

3）缺点
- 性能开销较大：反射调用比直接调用慢、高频场景不适合大量使用
- 破坏封装性：可以访问 private 成员

4）常见场景
- Spring：IOC、依赖注入、AOP
- 注解：运行时读取注解做路由/校验/映射。
- 动态代理：JDK Proxy 基于反射调用 InvocationHandler。

一句话总结：反射=运行时操作类结构；优点是灵活解耦；缺点是慢且不安全；框架用它做 IOC/注解/代理。

16、异常体系（面试口述版）
1）Error vs Exception
- Error：JVM 层面的严重问题，通常程序无法/不应该处理（如 OOM、StackOverflowError）。
- Exception：程序运行中可预期的异常情况，通常需要处理或抛出。

2）RuntimeException vs Checked Exception
- RuntimeException（运行时异常）：不强制 try-catch/throws，通常是“代码问题/参数问题”（NPE、IndexOutOfBounds、IllegalArgument）。
- Checked Exception（受检异常）：编译器强制处理（try-catch 或 throws），通常是“外部环境问题”（IO、网络、数据库等）。

3）try-catch-finally 执行顺序？finally 一定会执行吗？
- 正常流程：try →（有异常走 catch）→ finally。
- finally 通常都会执行，用来释放资源（close/unlock）。
- 例外情况（面试要说）：
  - JVM 直接退出：System.exit()
  - 进程崩溃/被 kill
  - 极端情况：finally 里死循环/阻塞导致看起来“没执行完”

4）自定义异常怎么设计？什么时候抛业务异常？
- 设计建议：
  - 业务异常一般继承 RuntimeException（不强制层层 throws）。
  - 带上错误码/错误信息/上下文（比如订单号、用户 id），便于定位。
  - 区分“业务可预期异常”（如库存不足）和“系统异常”（如 DB 挂了）。
- 什么时候抛：
  - 业务校验不通过：参数非法、权限不足、状态不允许、库存不足等。
  - 需要让上层统一捕获并返回明确错误码/提示时。

一句话总结：Error 别硬扛；Exception 要处理；运行时异常多是代码问题；受检异常多是外部问题；finally 负责收尾；业务异常用 RuntimeException + 错误码。

17、volatile 作用？（面试口述版）
volatile 是一个轻量级并发关键字，用来保证“可见性”和“有序性”。
- 保证可见性：保证一个线程的修改，会立即刷新到主内存中，对其他线程立即可见。
- 保证有序性（禁止指令重排序）：JVM 和 CPU 可能会对指令进行重排序，比如 a = new Object();
  理想顺序下，其实至少三步：分配内存、初始化对象、把对象引用赋值给 a。但在多线程下可能导致逻辑错误，比如第二步和第三部的顺序对调了。

3）volatile 不能保证什么
- 不保证原子性：i++ 这种复合操作仍然不是原子操作。

4）常见使用场景
- 状态标记/开关：比如 stop 标志位。
- DCL 单例里的 volatile（防止 new 对象时指令重排导致拿到半初始化对象）。

18、ThreadLocal 原理？为什么会内存泄漏？怎么避免？（面试口述版）
1）ThreadLocal 是什么
- ThreadLocal 是“线程本地变量”：同一个 ThreadLocal，在不同线程里各有一份值，互不影响。

2）底层原理
- 每个线程 Thread 内部都有一个 ThreadLocalMap。
- ThreadLocalMap 的 key 是 ThreadLocal（弱引用），value 是你存的对象。

3）为什么会内存泄漏
- key 是弱引用：ThreadLocal 没有强引用时，key 可能被 GC 掉，变成 null。
- 但 value 是强引用：如果线程一直不结束（线程池线程），value 可能一直挂在 ThreadLocalMap 里，导致内存泄漏。

4）怎么避免
- 用完一定 remove：try/finally 里 ThreadLocal.remove()。
- 线程池场景尤其要 remove（线程复用，最容易泄漏/串数据）。

19、线程池和 ThreadLocal 搭配一起用会有什么问题？（面试口述版）
- 核心原因：线程池会复用线程，但 ThreadLocal 的值是“挂在线程上的”，不会自动清理。
- 常见问题：
  - 串数据：上一个任务没清理，下一个任务可能读到上一个请求的用户信息/租户ID/traceId。
  - 内存泄漏：线程长期不结束，ThreadLocalMap 里的 value 可能长期占用内存（尤其是大对象）。
  - 上下文传递坑：InheritableThreadLocal 只对“新建线程”生效，在线程池里常常不生效或拿到旧值。
- 解决方案（背这句）：**用完必清理（try/finally remove），别放大对象；需要跨线程传递就用任务包装/TTL 统一传递与清理。**

20、synchronized 底层原理（面试口述版）
1）synchronized 锁的是什么
- synchronized 锁的是“对象监视器 monitor”。
- 对象头（Mark Word）里会记录锁状态，必要时指向 Monitor。

2）怎么实现互斥
- 进入 synchronized：尝试获取 monitor。
- 获取不到：在重量级锁下会阻塞挂起，等持锁线程释放后再唤醒。

3）锁升级（JDK 对 synchronized 的优化）
- 无锁 → 偏向锁 → 轻量级锁（CAS + 自旋）→ 重量级锁（阻塞/唤醒）。
- 目的：在“无竞争/少竞争”时尽量不阻塞，提高性能。

4）常见追问
- synchronized 是可重入锁。
- synchronized 释放锁：退出同步代码块/方法时自动释放。

一句话总结：volatile 管可见性/有序性；ThreadLocal 管线程隔离但要 remove；synchronized 通过 monitor 实现互斥并有锁升级优化。

21、多线程基础（面试口述版）
1）线程和进程的区别
- 可以这样回答（口述版）：
  进程就是程序的一次运行实例，有自己独立的内存空间和资源，不同进程之间默认不共享内存，是资源分配单位。
  线程是进程里的执行单元，一个进程可以有多个线程，它们共享同一块内存，是 CPU 调度单位。


2）线程怎么创建（常见 3 种）
- 继承 Thread：new MyThread().start()。
- 实现 Runnable：new Thread(runnable).start()。
- 线程池 + Callable/Future：提交任务拿 Future（生产更常用）。

3）线程状态（6 个）
- NEW / RUNNABLE / BLOCKED / WAITING / TIMED_WAITING / TERMINATED。

4）常见方法（面试常问）
- start vs run：start 才会真正启动新线程；run 只是普通方法调用。
- sleep：让当前线程睡一会儿，不释放锁。
- wait/notify：必须在 synchronized 内使用；wait 会释放锁；notify/notifyAll 唤醒等待线程。
- join：等待另一个线程执行完。

22、线程池核心参数（ThreadPoolExecutor 7 参数，面试口述版）
线程池的核心是 ThreadPoolExecutor：

ThreadPoolExecutor 3 个最重要的参数（面试可以这样说）：
- corePoolSize（核心线程数）
  线程池里常驻的线程数量，上线后基本会一直存在。
  当运行的线程 < corePoolSize 时：有新任务来，就直接新建线程执行，不进队列。
  可以理解为：“平时保持这么多工人常驻干活”。

- maximumPoolSize（最大线程数）
  线程池里能开的最大线程总数（核心 + 非核心）。
  当队列满了，还来任务时，如果当前线程数 < maximumPoolSize，就会再开新线程（非核心线程）处理。
  可以理解为：“最多能临时拉这么多工人一起上”。

- workQueue（任务队列）
  放“等待执行任务”的队列。
  流程是：
  - 先看当前运行线程数是否 < corePoolSize，是就新建核心线程执行；
  - 否则，任务进队列等着；
  - 队列也满了，再尝试开非核心线程（直到 maximumPoolSize）；
  - 实在放不下，就走拒绝策略。
  常见实现：ArrayBlockingQueue（有界数组）、LinkedBlockingQueue（链表，有界/无界）、SynchronousQueue（不存任务，直接交给线程）。

ThreadPoolExecutor 其他常见参数：
- keepAliveTime（非核心线程存活时间）
  当线程池里线程数大于 corePoolSize（也就是有非核心线程）时，这些非核心线程如果空闲了，不会立刻销毁，而是会“等一会儿”。
  等的这段时间就是 keepAliveTime；超过这个时间还没任务，就把这些非核心线程回收掉。
  所以它影响的是：高峰期结束后，临时加的线程多久收回。

- unit（时间单位）
  就是 keepAliveTime 的时间单位，比如 TimeUnit.SECONDS、MILLISECONDS 等。

- threadFactory（线程工厂）
  负责创建新线程的策略。
  可以用它来：
  - 给线程起统一、有含义的名字（方便排查问题）；
  - 设置是否是守护线程、优先级等。

- handler（拒绝策略）
  当前线程数已经到 maximumPoolSize，队列也满了，新任务没地方放时，就会触发这个策略。
  常见几种：
  - AbortPolicy：直接抛异常（默认）。
  - CallerRunsPolicy：由提交任务的线程自己执行（起到削峰作用）。
  - DiscardPolicy：悄悄丢掉新任务。
  - DiscardOldestPolicy：丢掉队列里最老的任务，把新任务加进去。

一句话记线程池执行流程（面试口述版）：
- 先用核心线程处理任务；
- 核心线程都忙不过来，任务进队列排队；
- 队列也满了，就再开非核心线程，一直开到 maximumPoolSize；
- 线程也满了、队列也满了，就触发拒绝策略（handler）。

23、Object 类的常见方法有哪些？（面试口述版）
- getClass()：获取对象的运行时类（Class），反射、类型判断时用。
- equals(Object obj)：默认比较“引用是否同一个”；重写后用来比“内容是否相等”。
- hashCode()：默认和对象地址相关；重写 equals 时一般要一起重写，满足哈希容器的约定。
- toString()：默认返回 类名@十六进制hashCode；重写后便于日志、调试打印。
- clone()：浅拷贝，需实现 Cloneable；子类要重写并注意深拷贝问题，实际用得不多。
- wait() / notify() / notifyAll()：和 synchronized 配合，做线程间等待与唤醒。
- finalize()：GC 前可能调用，已不推荐使用，了解即可。

24、ArrayList 与 LinkedList 区别？（面试口述版）
- 底层结构：ArrayList 是动态数组，元素在内存中连续存放；LinkedList 是双向链表，节点之间通过指针前后相连。
- 随机访问：ArrayList 通过下标随机访问是 O(1)，适合按索引读写；LinkedList 按下标访问需要从头/尾走一圈，是 O(n)。
- 插入/删除：在中间位置频繁插入/删除时，ArrayList 需要搬移后面的元素，平均 O(n)；LinkedList 只要改前后指针，找到节点后是 O(1)。
- 空间与缓存：ArrayList 连续存储，指针开销小，对 CPU 缓存友好，遍历性能好；LinkedList 每个节点多两个指针，内存开销大，缓存友好性差。
- 使用场景总结：随机访问多、遍历多、插入删除不算频繁，用 ArrayList；极少数真的需要在中间频繁插删、且不怎么按下标访问时，才考虑 LinkedList（实际项目里用得相对少）。

二、JVM
1、如何排查 OOM（面试口述版，记 3 步）
可以这样答：**先看类型、再抓现场、最后定原因**。

第 1 步：先看 OOM 类型（是什么内存不够）
- 常见几种：Java heap space、Metaspace、GC overhead limit exceeded、Direct buffer memory 等。
- 先搞清楚是堆不够，还是元空间/直接内存等不够，心里有个大方向。

第 2 步：抓现场（拿 dump）
- 启动参数打开自动 dump：-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=...。
- 或事后用 jmap 导出堆：jmap -dump:format=b,file=xxx.hprof <pid>。

第 3 步：分析 dump + 判断原因
- 用 MAT / VisualVM 看 Dominator Tree、Top Consumers、GC Roots，看看是哪些对象占了大头、被谁引用着。
- 典型两类结论：
  - 内存泄漏：对象本该释放但被长期引用（静态集合、无限制缓存、ThreadLocal 未清理等），要改代码。
  - 内存真的不够/峰值太高：对象确实需要，考虑优化数据结构、分批处理、限流/分页，或在评估后适当调大堆。
- 最后结合日志和监控（GC 日志、接口 QPS、响应时间、缓存命中率等）交叉验证，避免只盯 dump。

2、JVM 由哪些部分组成？运行时内存区有哪些？
JVM 主要包括：
1） 类加载子系统（ClassLoader）  把 .class 文件加载进内存，变成 JVM 能用的“类信息”。
2） 运行时数据区（Runtime Data Area）程序运行时用到的内存区域。
3） 执行引擎（Execution Engine） 真正执行字节码:
- 解释执行（逐条解释）
- JIT 编译（热点代码编译成机器码更快）
- GC（垃圾回收也属于 JVM 运行时的重要机制）
4） 本地方法接口 JNI + 本地方法库.让 Java 能调用 C/C++ 的代码（比如一些底层库）。

运行时数据区：“3 个线程私有 + 2 个线程共享”
- 线程私有：
    1） 程序计数器（PC）. 记录当前线程执行到哪一行字节码了。线程切换回来能接着跑。
    2） 虚拟机栈（Java Stack）。每次调用方法都会压一个栈帧，里面放局部变量和返回信息。 常见错误： StackOverflowError （递归太深）。
    3） 本地方法栈（Native Stack） 给 JNI 这种调用 C/C++ 的本地方法用的栈，了解即可。
- 线程共享：
    4） 堆（Heap）。存对象的地方，new 出来的对象基本都在堆里。 常见错误： OutOfMemoryError: Java heap space 。
    5） 方法区（Method Area）存“类相关的信息”：类元数据、常量池、静态变量等。
        JDK8 以后方法区的实现主要是 元空间 Metaspace （用本地内存）。


3、方法区是什么？
方法区是线程共享的内存区域，用于存放类元数据（类结构、运行时常量池、字段/方法信息、静态变量等）。

4、方法区在 JDK 1.7 和 1.8 的区别
JDK7 以前“类的信息”放在 JVM 里的一个固定区域（永久代），
JDK8 把这个固定区域删了，改成用操作系统的本地内存（元空间），不容易因为类太多把 JVM 撑爆。

- 1.7：方法区主要落在永久代（PermGen），受 JVM 堆内存参数影响，容易 PermGen OOM。
- 1.8：永久代移除，改为元空间（Metaspace），使用本地内存，参数主要是 -XX:MaxMetaspaceSize。

三、Spring
1、Spring IOC 底层原理（简述）
Spring IOC 对象不再由你 new ，而是 Spring 负责创建、组装（注入依赖）、并保存起来，你需要时直接从容器拿。
大概流程是： 扫描 -> 创建 -> 注入 -> 放进容器 。
展开成 4 步：
1） 扫描 ：启动时扫描@Component/@Service/@Controller/@Configuration 等注解，把每个 Bean 解析成定义信息（BeanDefinition）。 
2） 创建 ：需要用到时（或启动时）根据定义去 new 对象（反射/构造器）。 
3） 注入 ：创建完对象后，Spring 会把它依赖的其他 Bean 填进去。
4） 缓存 ：如果是单例 Bean，Spring 会把它放进容器的缓存里，后面直接取用。

Spring 启动时会扫描配置/注解，解析成 BeanDefinition；
然后通过 BeanFactory 创建 Bean：实例化（反射/构造器）-> 属性填充（依赖注入）-> Aware 回调 -> BeanPostProcessor 
前后置处理（AOP 代理常在这里生成）-> 初始化（@PostConstruct/InitializingBean）-> 放入单例池（singletonObjects 等缓存）供后续获取。

补充：@Autowired 和 @Resource 的区别（面试口述版）
- @Autowired（Spring 提供）：默认按类型装配（byType），如果有多个同类型 Bean，再结合 @Qualifier 或参数名按名称细化；支持构造器注入、字段注入、setter 注入，required=false 时找不到 Bean 也不会报错。
- @Resource（JSR-250 标准）：默认按名称装配（byName，先看 name，再看字段名），找不到再按类型；更偏向规范注解，在多框架环境下可移植性更好。
- 简单记：**优先 @Autowired（配合 @Qualifier），需要按名称精确指定或做更通用写法时用 @Resource。**

2、Spring MVC 执行流程
第一步，找入口：请求先到DispatcherServlet，它是整个流程的核心，负责调度。
第二步，找方法：HandlerMapping 根据 URL 找到对应的 Controller 方法
               HandlerAdapter 再去执行这个方法，得到ModelAndView 或返回值。
第三步，做收尾：拦截器的 afterCompletion 方法做最后执行，将响应返回。

请求进入 DispatcherServlet：
1）HandlerMapping 根据 URL 找到 Controller 方法 及拦截器链。
2）HandlerAdapter 调用目标方法。
3）返回 ModelAndView 或返回值：
- 视图：ViewResolver 解析视图并渲染。
- JSON：@ResponseBody / @RestController 走 HttpMessageConverter 序列化。
4）拦截器 afterCompletion，最终响应返回。

3、Spring AOP 是什么？底层原理是什么？（更好背版）
- 什么是 AOP：可以简单理解为“面向切面编程”，就是**给方法套一层外壳**，在不改原方法代码的前提下，把日志、事务、权限这些“横切关注点”统一抽出来，在方法执行的前后/异常时统一处理。
- 在 Spring 里，AOP 的本质是：Spring 不直接把原始 Bean 给你，而是给你一个“代理对象”，你调的其实是代理；代理在合适的时机再去调用目标方法，中间把各种增强逻辑织进去。

- 底层原理：主要是“动态代理 + 方法拦截”
  - JDK 动态代理：基于接口，只能代理实现了接口的类；Spring 默认优先用这个（代理的是接口）。
  - CGLIB 代理：通过生成子类、重写方法来实现代理；适合没有接口的普通类（代理的是类本身）。
  - 方法调用链：调用代理方法 -> 进入一堆切面（前置/后置/环绕等） -> 最后再执行目标方法 -> 再执行收尾逻辑。

4、事务是什么？传播行为和隔离级别（面试口语版）
事务就是把一组操作当成一个整体：要么都成功提交，要么出错就一起回滚，保证数据一致性（ACID）。

Spring 事务传播行为（常问 3 个就够用）：
- REQUIRED（默认）：外面有事务就加入；外面没有就新开一个。
- REQUIRES_NEW：不管外面有没有，都新开一个事务；外层事务先挂起。
- SUPPORTS：外面有事务就加入；外面没有事务就“以非事务方式”执行。

隔离级别（解决并发读写问题）：
- 读未提交
- 读已提交
- 可重复读
- 串行化

一句话总结：传播行为决定“多个方法事务怎么嵌套/怎么开”；隔离级别决定“并发下能看到什么数据”。

5、Spring Cloud Alibaba 常用组件和功能（面试口语版）
- Nacos：注册中心 + 配置中心（服务注册发现、动态配置、灰度/多环境配置）。
- Gateway：API 网关（统一路由、鉴权、限流、日志、跨域等）。
- Seata：分布式事务（AT/TCC 等模式，解决跨服务一致性问题）。
- OpenFeign：声明式 HTTP 调用（像调本地方法一样调远程服务）。
- Sentinel：限流/熔断/降级（保护服务不被发突流量打挂）。
- RocketMQ：消息队列（异步解耦、削峰填谷、最终一致性）。

一句话总结：Nacos 管注册和配置，Sentinel 管保护，Feign 管调用，Gateway 管入口，Seata 管分布式事务，MQ 管异步。

6、Spring 循环依赖如何解决？（面试口述版）
循环依赖是在 Spring 容器中，两个或多个 Bean 互相依赖对方，导致无法完成实例化。Spring 通过三级缓存机制来解决循环依赖。
一级缓存：用于存放已经完全初始化的 Bean。
二级缓存：用于存放部分初始化的 Bean（也叫早期对象）。当 A 依赖 B，B 还没有完全初始化时，Spring 会先将 B 放入二级缓存，允许 A 从二级缓存中获取到部分初始化的 B。
三级缓存：保存的是 beanDefinition，这是 Spring 用来描述 Bean 定义的元数据，帮助 Spring 完成 Bean 的创建过程。

7、MyBatis 和 MyBatis-Plus 的区别？（面试口述版）
- MyBatis：自己写 SQL，灵活但 CRUD 要多写代码。
- MyBatis-Plus：帮你生成/封装常见 CRUD，少写 SQL，开发更快。
- 口诀：**MyBatis 写得多更灵活，MyBatis-Plus 写得少更高效，底层都是 MyBatis。**

四、数据库（MySQL/ES）
1、事务的基本特性（ACID）与隔离级别（面试口述版）
1）ACID：
- 原子性（Atomicity）：一个事务里的操作要么全成功，要么全失败，失败就回滚。
- 一致性（Consistency）：⼀致性指的是数据库总是从⼀个⼀致性的状态转换到另外⼀个⼀致性的状态。⽐如A转账给B100块钱，
                       假设中间sql执⾏过程中系统崩溃A也不会损失100块，因为事务没有提交，修改也就不会保存到数据库。
- 隔离性（Isolation）：⼀个事务的修改在最终提交前，对其他事务是不可⻅的。
- 持久性（Durability）：⼀旦事务提交，所做的修改就会永久保存到数据库中。

2）并发下常见问题：
- 脏读：读到别的事务未提交的数据。
- 不可重复读：同一事务两次读同一行，结果不一样（别人改了并提交）。
- 幻读：同一个事务里，两次用同样条件查数据，第二次结果集多了（或少了）一些“新行”，像凭空出现/消失一样。

3）隔离级别（从弱到强）：
- 读未提交（READ UNCOMMITTED）：可能脏读/不可重复读/幻读。
- 读已提交（READ COMMITTED）：避免脏读，但可能不可重复读/幻读。
- 可重复读（REPEATABLE READ）：避免脏读/不可重复读；MySQL InnoDB 默认。
- 串行化（SERIALIZABLE）：最严格，基本避免所有并发问题，但并发性能最差。

4）那你说说什么是幻读？什么是 MVCC？（口述版）
- 幻读：同一个事务里，两次用同样条件查数据，第二次结果集多了（或少了）一些“新行”，像凭空出现/消失一样。
  和不可重复读区别：不可重复读是“同一行的值变了”；幻读是“多了/少了整行”。

- MVCC，也叫 多版本并发控制，核心目标是，在并发事务里让读操作尽可能不加锁、提高并发性能，同时仍然保证隔离性。
  底层的实现，是通过数据库给每条记录保留多个版本，让每个事务按自己的“快照”去读数据。InnoDB 里一般是通过隐藏字段 + undo log + Read View 来实现。

一句话总结：幻读=同条件两次查多/少了行；MVCC=多版本+快照读，提高并发。

一句话总结：ACID 记四个词；隔离级别从 RU -> RC -> RR -> Serializable，级别越高越安全但并发越差。

2、数据库三范式（面试口述版）
- 第一范式（1NF）：**字段要拆开**，一列只放一种信息，比如姓名和手机号分两列放。
- 第二范式（2NF）：**完全依赖主键**，非主键不能只依赖“联合主键的一部分”，否则就拆表或改成单一主键。
- 第三范式（3NF）：**只依赖主键**，非主键之间不要再互相依赖，比如不要在班级表里既放“老师 ID”又放“老师姓名”，姓名应该去老师表。
- 口诀版：1NF 拆字段，2NF 看“是否完全依赖主键”，3NF 看“是否只依赖主键”；实际设计里在范式和查询性能之间平衡，必要时会适当冗余。

3、什么情况下索引可能失效/不走索引
常见原因：
- 联合索引不满足最左前缀。  idx(a, b, c) ; select * from t where b = 2;
- 对索引列做函数/运算/类型转换（如字符串列用数字比较）。select * from t where date(create_time) = '2026-01-24';
- like 以 % 开头。 select * from user where name like '%mi%';
- 使用 OR 且部分条件无索引（可能导致全表扫描）。  假设 a 有索引， b 没索引; select * from t where a = 1 or b = 2;
- 选择性太差/回表成本高，优化器可能选择全表扫描。 假设 gender 只有男/女两种值，即使建了索引也可能不走; select * from user where gender = '男';
- join 字段字符集/类型不一致可能导致转换，从而影响索引使用。
t1.user_id 是 varchar(32) （有索引）
t2.user_id 是 int （有索引）
可能不走索引：select *
from t1
join t2 on t1.user_id = t2.user_id;

最后对症优化 ：没走索引 / 索引用得不好 → 补索引 / 调索引顺序
            ，SQL 写法导致索引用不上 → 改写 SQL
            ，如果是锁等待就查事务和锁。
            （，分页太深就改成游标/延迟关联）

1）先定位：开启慢查询日志/监控，确认慢 SQL、频率、平均/TP99。
2）看执行计划：EXPLAIN 关注 type、key、rows、Extra（Using filesort/temporary）。

5、常见索引类型及特点
按用途（你建索引是为了解决什么问题）：
1）约束类：
- 主键索引（PRIMARY KEY）：唯一且非空，一个表只能有一个主键索引；适用场景：唯一标识记录，如用户ID、订单ID等。
- 唯一索引（UNIQUE）：值唯一，允许为空（但最多一个NULL）；适用场景：需要唯一约束字段的，如邮箱、手机号等。

2）查询加速类：
- 普通索引（INDEX）：无唯一性约束，允许重复值；适用场景：频繁查询的字段，如商品名称、分类ID等。
- 联合索引（Composite Index）：多个字段组合而成，遵循最左前缀原则；适用场景：多条件组合查询，如 (user_id, create_time)。

3）特殊检索类：
- 全文索引（FULLTEXT）：用于文本内容检索，支持分词搜索；适用场景：文章内容、商品描述等长文本字段的搜索。
- 空间索引（SPATIAL）：用于地理空间数据类型（如 POINT、GEOMETRY）；适用场景：地理位置相关查询，如附近的商家、范围查询。

按底层结构（索引是怎么存的）：
B+ 树索引
- B+ 树是数据库最常用的一种索引数据结构，特别是在 MySQL InnoDB 引擎里默认使用的就是 B+ 树索引。
- 它是一种自平衡的多路搜索树：内部节点只存键，叶子节点存储键（＋数据或指针），而且所有叶子节点通过链表顺序连接，方便范围查询。
- B+ 树的查找、插入、删除复杂度都是对数级别 O(log N)，可以高效处理大量数据。
- B+ 树非常适合于 范围查询和排序（例如 BETWEEN、>, < 等），因为数据在叶子节点是按顺序链接的。
Hash 索引
- Hash 索引是另一种索引结构，它通过对索引列计算 哈希值 来快速定位数据。
- Hash 索引对于 等值查询（如 =、<=>）非常快，理论上可以达到常数时间查找 O(1)。
- 但是 Hash 索引不能用于范围查询（比如 <, >, BETWEEN），因为哈希结构本身不保存排序关系。
- 在 MySQL 中只有某些存储引擎才支持 Hash（比如 MEMORY 引擎），InnoDB 默认使用的是 B+ 树。

根据数据的物理存储方式：
聚簇索引
非聚簇索引

索引设计原则：
- 遵循最左前缀：联合索引按顺序使用字段；
- 选择高选择性字段：区分度高的字段更适合建索引；
- 控制索引数量：过多索引会影响插入/更新性能；
- 考虑覆盖索引：查询字段都在索引中，避免回表；
- 避免频繁更新的字段：更新会导致索引重建。

4、聚簇索引 vs 非聚簇索引（InnoDB 口述版）

聚簇索引（Clustered Index）
- 聚簇索引是一种 索引+数据物理顺序存储合二为一 的结构。
- 在 MySQL InnoDB 中，主键索引就是聚簇索引：
  B+ 树的叶子节点 直接存储行数据（整行数据），
  同时按主键顺序物理存放在磁盘上。
- 因为数据本身按索引顺序排列，所以范围查询效率更高、IO 更少。
- 一个表最多只能有一个聚簇索引（因为数据只能按一个顺序存储）。
✅ 简单理解：聚簇索引把索引和真实数据合并在一起了，找一次索引就直接定位到数据，不需要再去查表。

非聚簇索引（Non‑Clustered Index）
- 非聚簇索引是 索引和数据分开存放 的结构。
- 在 MySQL InnoDB 中，非主键列创建的索引（辅助索引/二级索引）就是非聚簇索引：
  B+ 树的叶子节点 只存储索引键＋对应的主键值（或指向行的指针），
  查到这个主键值后，需再去聚簇索引里根据主键查一趟数据（这个过程叫 “回表”）。
- 一个表可以有 多个非聚簇索引。
✅ 简单理解：非聚簇索引是“辅助索引”，不存行数据，查完索引后需要再去聚簇索引（主键）拿数据。

5、什么是回表？什么是覆盖索引？（口述版）
- 回表：当查询用的是一个普通的二级索引（不是主键索引），数据库找到了匹配的主键，再根据主键回到聚簇索引把整行数据取出来，这一步就叫回表。
假设有这么一张表：
CREATE TABLE user (
  id INT PRIMARY KEY,
  name VARCHAR(20),
  age INT,
  INDEX(name)
);

- 你查询：SELECT id FROM user WHERE name = 'Tom';
因为 id 在索引里（在二级索引叶子节点存有主键值），所以 不用去主表取数据 ❗ — 不回表。
- 你查询：SELECT age FROM user WHERE name = 'Tom';
age 不在 name 索引里，只在主表数据里。
所以先通过二级索引找到记录对应的主键值，然后再 回到主表去取 age 值 — 这就是 回表。
📌 所以 回表 = 索引查到主键 → 再根据主键到表里取完整数据。

- 覆盖索引：一次查询需要的字段都在同一个索引里（通常是二级索引里就包含了 select 的列），直接从索引返回结果，不需要回表。
举个例子继续上面的表：
SELECT id, name
FROM user
WHERE name = 'Tom';

- 这个查询需要的数据是 id 和 name。
如果你有一个索引 (name, id)（顺序放一起）：
索引里已经有 name 和 id。查询时数据库就可以只靠这个索引就拿到结果，不需要去主表找别的字段 → 覆盖索引。

6、SQL 调优经验（极简记忆版）
核心口诀
索引优化：高选最左，覆盖少回表  
SQL 写法：避函数，少 %like，拆 OR，限返回  
执行计划：看 type，查 rows，避 filesort  
事务锁：短事务，合理隔离，避死锁  
表结构：分库分表，合理类型，用分区  
其他：缓存热点，读写分离，预计算  

关键要点（一句话总结）：
1. 索引：选区分度高的字段，联合索引按最左前缀，用覆盖索引减少回表。
2. SQL：索引列别用函数，少用 `%like`，OR 拆成 UNION，只查必要字段。
3. 执行计划：type 越左越好（const > eq_ref > ref > range > index > ALL），rows 越少越好，避开 filesort/temporary。
4. 事务：缩小事务范围，用读已提交，避免长事务，统一锁顺序。
5. 表结构：大表分库分表（如超过1000万行）时，选合适数据类型（INT 存数字），按时间分区，定期清理历史数据。
   垂直分库分表（基于“列”的拆分），水平分库分表（基于“行”的拆分）
6. 技巧：Redis 缓存热点，主从读写分离，统计数据预计算。

7、MySQL 中的锁有哪些（面试口述版）
按 MySQL/InnoDB 常见的说法，你可以从 3 个层面讲：
1）按锁粒度：
- 表锁：锁整张表，开销小但并发差。MyIsam就是通过表锁。
- 行锁：锁某些行，并发好；InnoDB 的行锁是基于索引实现的。InnoDB就主要以行锁为主。

2）按读写：
- 共享锁（S 锁）：读锁，多个事务可以同时读。
- 排他锁（X 锁）：写锁，写时别人不能读/写。

3）InnoDB 为了解决幻读的锁（RR 隔离级别常见）：
- 间隙锁（Gap Lock）：间锁住 两个索引值之间没有实际记录的空间（比如 id=1 和 id=5 之间的 gap）。只对 id=1 和 id=5 两条锁住，
                     其它事务仍可能往 (1,5) 插入新记录，这就可能导致 A 再次查询时结果突然多出来新行 → 幻读。为了阻止这种情况，
                     InnoDB 通过间隙锁 会锁住 (1,5) 这个区间的“间隙”，让别人不能插入新记录。这样下一次相同查询就不会多出幻影记录。
- 临锁键（Next-Key Lock）：记录锁（即最基本的行锁） + 间隙锁，既锁记录也锁间隙。在间隙锁的基础上，锁住了左开右闭区间的一个区间 (1,5],用行锁保证边界值不被修改。

补充：
- 意向锁：表级的“声明”，告诉 InnoDB 该事务打算稍后对表的某些行加哪种锁（S 或 X）。
- MDL（元数据锁）：对表结构加锁，任何对表结构的修改（如 ALTER TABLE/DROP TABLE）都需要 MDL 锁。这样可以避免 DDL（数据定义） 语句和 DML（数据修改）同时冲突。

一句话总结：表锁/行锁；S 锁/X 锁；Gap/Next-Key；再补意向锁和 MDL。

8、MySQL 主从复制怎么实现的？（结构化答法）
MySQL 主从复制就是：主库把变更写到 binlog，从库拉取到 relay log，再在从库重放执行。

① 前提条件
- 主库开启 binlog（所有数据修改都会写 binlog）
- 从库能连主库，并有权限读取 binlog

② 核心组件
- binlog（主库）：记录所有数据变更事件（DML + DDL），是复制的“源头日志”
- relay log（从库）：从库把拉到的 binlog 先落到本地，给后续重放用
- 复制线程：负责“拉取”和“执行”

③ 复制过程（更好理解版）
记法：主库“记账”（binlog），从库“搬运+落盘”（relay log），再“照着做”（重放）。

1）主库提交事务，把变更写进 binlog（binlog 在主库）
2）从库 IO 线程连主库发起请求：我要从哪个位置开始拉 binlog
3）主库开 binlog dump 线程，把 binlog 事件流发给从库（这是对“拉取请求”的响应）
4）从库 IO 线程收到后，写到本地 relay log（relay log 在从库）
5）从库 SQL 线程读 relay log 并执行，把数据追平，与主库保持一致

补一句：IO 线程负责拉取，SQL 线程负责执行，执行慢不会影响继续拉取。

relay log 的价值就是： 先把主库的变可靠更地落到从库本地，再慢慢按顺序执行 。防止主库故障了，更改也丢失了。

| 分类      | 主要作用       | 典型语句                      |
| ------- | ---------- | --------------------------------- |
| **DDL** | 定义/修改 表/结构 | `CREATE`, `ALTER`, `DROP`   |
| **DML** | 操作表中的数据    | `INSERT`, `UPDATE`, `DELETE` |
| **DQL** | 查询数据       | `SELECT`                        |
| **DCL** | 控制权限       | `GRANT`, `REVOKE`                 |
| **TCL** | 事务管理       | `COMMIT`, `ROLLBACK`, `SAVEPOINT` |


9、Elasticsearch 倒排索引（Inverted Index）
正排索引：更像“文档ID → 词”
倒排索引里：是“词 → 一堆文档ID”
Elasticsearch 的查询逻辑主要分四步：
1） 分词 ：分词器把查询语句拆成词项（比如“Java开发教程”拆成“Java”、“开发”、“教程”）。
2） 查倒排索引 ：分别找每个词项的倒排列表，取交集得到命中的文档。
3） 相关性评分 ：按TF-IDF计算每个文档的得分（词频越高、越稀有越相关），然后排序。
4） 返回结果 ：返回排序后的文档，可能带高亮。

五、Redis
1、缓存穿透/击穿/雪崩（怎么区分 + 怎么解决）
口诀：穿透查不到；击穿热点过期；雪崩一片失效。

- 缓存穿透：请求的数据“DB 里本来就不存在”（例如用大量随机 id 查用户/商品），所以缓存永远 miss，每次都打到 DB。
  解决：布隆过滤器（先挡掉明显不存在的 id）；缓存空值/空对象（短 TTL）；接口层参数校验 + 限流。

- 缓存击穿：某个“热点 key”（例如爆款商品详情）过期的瞬间，并发很高，大家同时 miss，一起打到 DB。
  解决：互斥锁（只让一个线程查询数据库并写回缓存）；逻辑过期（先返回旧值，后台异步查询数据库刷新）；热点预热。

- 缓存雪崩：大量 key 在同一时间过期，或 Redis 整体不可用，导致大量请求同时打到 DB。
  解决：TTL 就是缓存的过期时间，避免“同一时刻集体过期”；多级缓存（Redis + 本地缓存）；
        限流/熔断/降级保护 DB；Redis 高可用（主从/哨兵/集群）。

2、Redis 持久化策略（面试口语版）
Redis 持久化主要两种：RDB 和 AOF。
- RDB：定时把内存快照保存到磁盘（dump.rdb）。优点是文件小、恢复快；缺点是可能丢“最后一次快照之后”的数据。
- AOF：把写命令按顺序追加到日志文件（appendonly.aof）。优点是更安全、丢数据少；缺点是文件更大、恢复更慢（需要重放日志）。

常见组合：RDB + AOF 同时开，兼顾恢复速度和数据安全。
一句话总结：RDB 是“拍照”，AOF 是“记账”。

3、Redis 主从 / 哨兵 / 集群（面试口语版）
- 主从复制：1 主多从，主负责写，从负责读；用于读写分离和数据备份。缺点是主挂了需要人工切换。
- 哨兵 Sentinel：在主从基础上加“看门人”，监控主节点；主挂了会自动选一个从节点提升为主，实现自动故障转移。
- 集群 Cluster：多主多从，数据按槽位（16384 slots）分片到不同主节点；既能水平扩容容量/性能能，也高可用（主挂了从顶上）。

一句话总结：主从=读写分离；哨兵=主从的自动切；主集群=分片扩容 + 高可用。

4、Redis + RabbitMQ 菜单/门店信息预热与异步更新（面试口述版）
1）目标：用 Redis 扛高频读，把压力从 DB 挪到缓存。
2）预热：系统启动或定时任务把菜单/门店热点数据提前写入 Redis，查询优先读 Redis。
3）更新：数据变更时先写数据库，再发一条更新事件到 RabbitMQ。
4）异步刷新：消费者收到消息后更新或删除对应 Redis key（常见删缓存，下次读回源重建），保证最终一致。
5）补充：配合 TTL；必要时加互斥锁/逻辑过期；消息 MQ 做幂等（按业务ID去重）。

5、Redis 为什么快？（面试口述版）
- 内存读写：数据主要在内存里，避免磁盘 IO。
- 单线程事件循环：核心命令单线程执行，避免多线程锁竞争（配合 IO 多路复用处理大量连接）。
- 高效数据结构：String/Hash/List/Set/ZSet 都有针对性实现，常见操作接近 O(1)。
- 网络开销低：支持 pipeline 批量发送命令，减少 RTT。

6、Redis 有哪些数据结构？使用场景？（面试口述版）
- String：缓存对象（JSON）、计数器（incr）、分布式锁 value。
- Hash：存对象字段（user:1 -> {name,age}），适合部分字段更新。
- List：消息队列/时间线（BLPOP/BRPOP 做阻塞队列）。
- Set：去重、共同好友、抽奖。
- ZSet：排行榜（score 排序），延迟队列（score=时间戳）。
- HyperLogLog：做近似去重计数（UV 统计），空间占用固定但有一定误差，适合大规模“有多少不同用户/不同 IP”这类统计。
- Bitmap：用 bit 位表示某个 ID 是否出现过，适合签到（某天哪些用户打卡）、活跃用户标记、简单的布尔状态统计。
- Geo：存地理位置信息（经纬度），配合 GEO* 命令做附近的人/店铺、距离计算、范围查询等。

7、缓存和数据库如何保证一致性？（面试口述版）
- 先说结论：**最常用的是 Cache Aside（旁路缓存）策略，写时“先更新 DB，再删缓存”**。

- 读流程（Cache Aside）：
  1）先查缓存，命中就直接返回；
  2）没命中再查数据库，把结果回填到缓存里；
  3）以后再读都先走缓存。

- 写流程（关键一句话）：
  - **先更新数据库，再删除缓存对应的 key，而不是先改缓存。**
  - 原因：在高并发下，如果你是“更新缓存”，很可能把旧值写回缓存；而“删缓存”可以保证后续读一定会回源 DB、重新塞入最新值。

- 常见细节与补偿：
  - 删除缓存失败：需要重试机制（如通过 MQ 记录变更、后台补偿任务重删 key），避免长期读到脏数据。
  - 热点 key：可以配合互斥锁/逻辑过期，避免缓存失效瞬间大量请求打爆 DB（缓存击穿问题）。
  - 更强一致性需求：可以采用“延迟双删”策略：**删缓存 → 写 DB → 延迟一小段时间再删一次缓存**，降低并发时短暂不一致的窗口。

8、Redis 如何保证原子性？（面试口述版）
- 单条命令原子：Redis 单线程执行命令，单条命令天然原子。
- 多条命令原子：
  - Lua 脚本：EVAL 执行，脚本整体原子。
  - 事务 MULTI/EXEC：保证顺序执行，但不回滚；WATCH 可做乐观锁。

9、Redis 分布式锁怎么实现？（面试口述版）
- 最核心思路：**用 Redis 里的一条 key，当成“谁拿到锁”的标记**，结合过期时间 + 唯一标识，来保证“互斥 + 不误删 + 不死锁”。

- 加锁（拿锁）
  - 命令：SET lock_key requestId NX PX ttl
    - NX：key 不存在才设置，保证互斥。
    - PX ttl：设置过期时间，防止服务挂了后锁永远不释放。
    - requestId：用 UUID 或线程/实例唯一标识，区分“这把锁是谁加的”。
  - 返回 OK 代表加锁成功；返回 null 说明已经有人持有锁，可以重试或直接失败。

- 解锁（释放锁）
  - 不能直接 DEL lock_key，否则可能把别人的锁删掉（比如自己的锁已经过期，别的实例又加上了新锁）。
  - 正确做法：用 Lua 脚本先比对 value，再决定删不删（保证原子性）：
    - if redis.call('get', key) == requestId then redis.call('del', key) end
  - 这样只有“锁还是我加的”那个人，才能成功释放锁。

- 注意点/高级用法
  - 续期（看门狗）：如果业务执行时间可能超过 ttl，可以配合定时续期（比如 Redisson 的看门狗机制），避免锁被提前自动释放。
  - 业务超时控制：即使有续期，也要在应用层设置最大执行时间，防止锁长期占用。
  - 主从复制延迟问题：在主从切换、复制延迟严重的情况下，简单的单节点锁会有少量极端不安全场景；安全性要求非常高时，可以考虑 RedLock 或直接用 ZooKeeper/etcd 这种强一致性组件实现分布式锁。

2、如何优化 VAE/编码链路的？
1）瓶颈：VAE 的 encode/decode + 视频编码链路里，CPU↔GPU 来回拷贝和中间格式转换太多，导致延迟和抖动明显。
2）背景：VAE = Encoder + Decoder；MuseTalk 尽量在 latent space 做口型相关生成/修复，比像素空间更适合实时。
   补充：VAE（Variational AutoEncoder，变分自编码器）可以理解成“压缩+还原”的模型。
   - Encoder：把图片/特征压缩成一个更小的 latent 表示（潜变量）。
   - Decoder：再把 latent 还原回图片/特征。
   它的好处是：在 latent 空间做生成/编辑，计算量更小、更稳定，更适合实时场景。
3）思路：关键路径尽量留在 GPU 上跑 + 减少不必要的转换和同步。
4）做法：
   4.1）VAE encode/decode 放 GPU 推理，尽量用 GPU tensor 直通，减少 CPU↔GPU 拷贝。
   4.2）推理用 FP16，配合小 batch / 流水线并行，让 GPU 不空转，降低单帧耗时。
   .43）编码链路减少中间格式转换和拷贝，尽量让渲染输出更直接对接编码器，避免反复转码。
   4.4）整体拆成异步流水线（推理/后处理/编码/发送），避免单点阻塞全链路。
5）结果：帧率更稳定（30FPS），端到端延迟下降，抖动明显变小。


九、场景题（系统设计口述版）
1、设计登录系统/登录页面（面试讲解稿：1-2 分钟口述版）
我会从“功能实现、安全防范、高并发优化”三个层次来设计。

首先是基本功能：
- 核心逻辑：支持多标识（手机/邮箱/用户名）+ 密码登录。
- 会话管理：后端验证通过后，发放 Token（如 JWT）或创建 Session，前端存入 Cookie 或 LocalStorage，后续请求带上标识。
补充知识点：什么是 JWT？
- Token：就是一个身份令牌（字符串），证明登录过。
- JWT（JSON Web Token）：是一种特殊的 Token，它把用户信息（如 userId）直接加密在字符串里。
- 优点：服务器不需要存 Session（无状态），直接解密就能知道你是谁，非常适合分布式架构。
- 安全：靠服务器的“密钥”签名，别人改不了内容。

其次是关键细节（点到为止）：
- 传输与存储安全：全站 HTTPS 必选；密码严禁明文，注册和登录比对必须使用加盐哈希（如 Bcrypt）。
- 交互校验：前端做防抖和格式校验，后端做统一错误提示（不暴露是账号错还是密码错），防止撞库。
补充知识点：什么是加盐哈希（Bcrypt）？
- 用途：专门用于【数据库存储密码】，不是加密 Token。
- 哈希（Hash）：把密码变成不可逆的乱码，即使数据库泄露也看不到原密码。
- 加盐（Salt）：每个用户分配一个随机字符串（盐），即使两人密码相同，存的乱码也不同。
- Bcrypt：后端最常用的加密库，它会自动处理加盐，且计算速度故意设计得很慢，防止黑客暴力破解。
补充知识点：前端防抖（Debounce）
- 含义：你尽管点，我只执行最后一次。
- 原理：设置一个计时器，如果用户在计时结束前又点了，就清空重连，直到用户停下超过设定时间才发请求。
- 目的：防止用户因为网络慢或着急连点，导致向后端发送一堆重复的请求。

最后是体现技术功底的优化（针对高并发和稳定性）：
- 性能优化：在数据库 identifier 字段（用户名/手机号）建唯一索引；将会话数据（Token/Session）存入 Redis，利用内存读写应对高频的登录校验请求。
- 流量限流与防爆破：接入 Redis 计数器做接口限流，针对同 IP 或同账号频繁失败的情况进行强制验证码拦截或临时封禁。
- 扩展性：如果考虑到分布式环境，我会采用 JWT 或 Redis 共享 Session，保证水平扩展时登录状态的一致性。

2、RabbitMQ 消息积压如何处理？
当 RabbitMQ 消息出现积压时，我会从三个主要方面进行处理：
提升消费者处理能力：
    *   横向扩容：最直接有效的方式是增加消费者实例数量，让更多消费者并行处理消息，提高整体吞吐量。
    *   优化消费逻辑：对消费者业务逻辑进行优化，如将复杂耗时操作改为异步处理（快速 ACK），或采用批量处理机制，以提高单个消费者的处理效率。
    *   合理限流 (QoS)：通过 `basic.qos` 控制消费者一次性拉取的消息数量，避免单个消费者因处理过多消息而崩溃，保持处理的稳定性。
优化 RabbitMQ 队列配置：
    *   限制队列长度：通过 `x-max-length` (消息数) 或 `x-max-length-bytes` (总大小) 配置，避免队列无限膨胀，超出限制后可选择丢弃最老消息或发送到死信队列。
    *   消息 TTL (Time To Live)：为消息或队列设置过期时间，让超期未消费的消息自动清除或进入死信队列，避免无效消息长时间占用资源。
    *   死信队列 (DLQ)：配合 TTL 或消息拒绝机制，将过期、被拒或超长的消息路由到死信队列（DLQ），进行统一管理、分析和后续处理（如告警、重试），避免其影响主业务队列。
    *   优先级队列：针对核心业务消息，设置高优先级，确保在积压时能优先被处理，保障业务核心流程。
系统级流量控制与保护：
    *   消息分级：根据消息重要性，将不同等级的消息投递到不同的队列，甚至独立的 MQ 集群，确保核心业务通道的畅通和优先级。
    *   上游限流：在消息生产方进行流量控制（如令牌桶、漏桶算法），防止突发或过高的流量直接冲垮 MQ 或下游服务，从源头控制风险。
    *   下游熔断降级：当消息处理的下游服务出现故障或处理能力明显下降时，消费者可以暂停消费，或将消息路由到降级处理逻辑（如持久化到数据库稍后重试），避免故障蔓延导致整个系统雪崩。
    *   监控与报警：建立完善的监控体系，实时监测队列长度、消费者状态、消息生产消费速率等关键指标，并配置及时报警，以便在积压初期迅速发现问题并介入处理。
