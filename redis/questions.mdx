## 五、Redis

### 1、缓存穿透/击穿/雪崩（怎么区分 + 怎么解决）

口诀：穿透查不到；击穿热点过期；雪崩一片失效。

- **缓存穿透**：请求的数据“DB 里本来就不存在”（例如用大量随机 id 查用户/商品），所以缓存永远 miss，每次都打到 DB。  
  解决：布隆过滤器（先挡掉明显不存在的 id）；缓存空值/空对象（短 TTL）；接口层参数校验 + 限流。

- **缓存击穿**：某个“热点 key”（例如爆款商品详情）过期的瞬间，并发很高，大家同时 miss，一起打到 DB。  
  解决：互斥锁（只让一个线程查询数据库并写回缓存）；逻辑过期（先返回旧值，后台异步查询数据库刷新）；热点预热。

- **缓存雪崩**：大量 key 在同一时间过期，或 Redis 整体不可用，导致大量请求同时打到 DB。  
  解决：TTL 打散，避免“同一时刻集体过期”；多级缓存（Redis + 本地缓存）；限流/熔断/降级保护 DB；Redis 高可用（主从/哨兵/集群）。

**为什么会有这三种区分？设计上在解决什么问题？**  
本质是**“谁在扛流量、什么时机失效、影响面多大”**的差别。穿透是“请求本身无意义、却一直打穿到 DB”，所以要**在入口挡住**（布隆、空值、校验）；击穿是“单点热点在过期瞬间被集体打穿”，所以要**把“同一时刻大量回源”收束成一次**（互斥、逻辑过期）；雪崩是“大面积失效或整机不可用”，所以要**避免同时失效 + 保护 DB 不被冲垮**（TTL 打散、多级缓存、限流、高可用）。  
生产里设计缓存时，会同时考虑：**命中率**（穿透会拉低）、**失效策略**（击穿/雪崩和 TTL、淘汰策略有关）、**一致性**（先更 DB 再删缓存才能少脏读）。三者不是孤立的“三个名词”，而是同一套缓存体系在不同失效场景下的表现，答的时候能说出“为什么这么设计、取舍在哪”，会更像有实战经验。

### 2、Redis 持久化策略（面试口语版）

Redis 持久化主要两种：RDB 和 AOF。  
- RDB：定时把内存快照保存到磁盘（dump.rdb）。优点是文件小、恢复快；缺点是可能丢“最后一次快照之后”的数据。  
- AOF：把写命令按顺序追加到日志文件（appendonly.aof）。优点是更安全、丢数据少；缺点是文件更大、恢复更慢（需要重放日志）。

常见组合：RDB + AOF 同时开，兼顾恢复速度和数据安全。  
一句话总结：RDB 是“拍照”，AOF 是“记账”。

**Redis 能否实现数据不丢失？**  
**不能绝对保证。** 可以做到“尽量少丢”，但无论持久化还是主从，都仍有丢失窗口。面试一句话：**Redis 不保证 0 丢失，持久化 + 主从 = 尽量少丢 + 高可用。**

**1）持久化（单机留底）**  
- RDB：丢“上次快照之后”的写入。  
- AOF：写的是“文件”不是立刻“磁盘”，有内存缓冲区和 fsync 策略（everysec 最多丢约 1 秒；always 最稳但性能差）；单机/单盘坏了照样全丢。  
→ 持久化只能尽量少丢，不能消除单点。

**2）主从/集群（多机副本）**  
- **主库写、从库读**：客户端只写主库，从库从主库**同步**数据，主挂了从顶上有副本。主库把复制流发给每个从库，每个从库各自有一份。  
- **主从也不是 0 丢失**：复制是**异步**的，主先响应用户“写成功”再异步同步。主在“刚写完、还没同步到从”时挂了，这段数据从库没有，就会丢。  
→ 主从 = 多一份副本 + 故障可切换，仍可能丢“最后一点”。

**RDB 与 AOF 优缺点对比（面试可答）：**

| 维度 | RDB | AOF |
|------|-----|-----|
| **原理** | 内存快照，二进制文件 | 写命令追加日志 |
| **优点** | 文件小、恢复快、对性能影响小（可子进程备份） | 丢数据少、可读可重放、误操作可回放修复 |
| **缺点** | 两次快照间可能丢数据、大库 fork 可能卡顿 | 文件大、恢复慢、写入性能依赖 fsync 策略 |
| **适用** | 可接受分钟级丢失、需要快速恢复 | 对数据安全要求高、可接受更大文件和恢复时间 |

**为什么同时提供 RDB 和 AOF？为什么不全用 AOF？**  
因为**使用场景和代价不同**。RDB 是“某一时刻的完整快照”，恢复时直接加载即可，**恢复快、文件紧凑**，适合做冷备、主从全量同步；但两次快照之间的写入会丢，所以是“可接受丢一段”的场景。AOF 是“按写命令记账”，理论上可以重放成最新状态，**丢数据少、可读可修复**；但文件会越来越大、恢复要逐条重放，**恢复慢、占空间**。若只做 AOF，大库重启和主从全量会非常慢；若只做 RDB，丢数据窗口大。所以设计成**可组合**：RDB 做定期快照 + AOF 做增量记录，重启时若存在 AOF 优先用 AOF 重放（数据更新），否则用 RDB 快速恢复。这样既照顾“少丢数据”，又照顾“恢复时间”，是典型的**权衡设计**。

**总结**：单机靠持久化少丢；多机靠主从多一份副本。两者叠加（RDB+AOF + 主从）可以尽量少丢，但**不能承诺绝对不丢**。若业务强一致、零丢失，需从设计上避免把 Redis 当唯一真相源（如关键数据落库、Redis 做缓存）。

### 2.1、Redis 的过期键删除策略？内存满了怎么办？（面试口述版）

**过期键删除策略（两种配合）：**

- **惰性删除**：访问某个 key 时再检查是否过期，过期就删。优点是不占额外 CPU；缺点是已过期但长期没被访问的 key 会一直占内存。
- **定期删除**：Redis 定期（如每 100ms）从设置了过期时间的 key 里随机抽一批检查，过期就删。不是全量扫，而是抽样 + 限制时间，避免卡死。
- **实际效果**：两者一起用——平时靠惰性删除，定期删除兜底，减少“过期不删”的堆积。若过期 key 很多且很少被访问，仍可能短暂占内存，所以不要无脑给大量 key 设很长 TTL。

**内存满了怎么办？（淘汰策略）**

当 `maxmemory` 被设置且内存达到上限时，Redis 会按 **maxmemory-policy** 淘汰 key，而不是直接 OOM：

| 策略 | 含义 |
|------|------|
| **noeviction** | 不淘汰，写操作直接报错（默认） |
| **allkeys-lru** | 全体 key 中最近最少用的淘汰（**最常用**） |
| **allkeys-lfu** | 全体 key 中访问频率最低的淘汰（4.0+） |
| **allkeys-random** | 全体 key 随机淘汰 |
| **volatile-lru** | 只在**设了过期时间**的 key 里 LRU 淘汰 |
| **volatile-lfu** | 只在设了过期时间的 key 里 LFU 淘汰 |
| **volatile-random** | 只在设了过期时间的 key 里随机淘汰 |
| **volatile-ttl** | 只在设了过期时间的 key 里优先淘汰 TTL 更短的 |

**为什么过期键用“惰性 + 定期”而不是纯定时扫全表？**  
若每次过期检查都扫全表，key 多了会**长时间阻塞**主线程，违背“单线程要快”的设计。惰性删除是“用到再删”，**零额外成本**，但只删被访问到的 key，长期不被访问的过期 key 会一直占着内存。定期删除是“每隔一段时间抽一批带过期时间的 key 检查”，**限制每次执行时间**，既减轻堆积又不会卡死。两者配合：大部分过期 key 在访问时被惰性删掉，少数“冷”的过期 key 由定期删除兜底。所以这是**用两种策略分担成本、避免单点卡顿**的设计。

**为什么内存淘汰要分 volatile-* 和 allkeys-*？**  
因为 Redis 里有的 key **没设过期时间**，可能是业务上“长期有效”的数据（如配置、分布式锁的 key），若内存满了就无差别淘汰，可能把不该删的删掉。**volatile-*** 只从“设了过期时间的 key”里淘汰，没设 TTL 的 key 不会被动删，适合“缓存和常驻数据混用”的场景；**allkeys-*** 从全体 key 里淘汰，适合“全当缓存、都可以丢”的场景。noeviction 则是“宁可不写也不删”，适合把 Redis 当存储且不能丢数据的场景。所以淘汰策略的设计是在**“谁可以丢、谁不能丢”**之间做区分。

**面试一句话**：过期键 = 惰性删除 + 定期删除；内存满 = 按 maxmemory-policy 淘汰（常用 allkeys-lru），noeviction 则写报错。

### 3、Redis 主从 / 哨兵 / 集群（面试口语版）

- **主从复制**：1 主多从，主负责写，从负责读；用于读写分离和数据备份。缺点是主挂了需要人工切换。  
- **哨兵 Sentinel**：在主从基础上加“看门人”，监控主节点；主挂了会自动选一个从节点提升为主，实现自动故障转移。  
- **集群 Cluster**：多主多从，数据按槽位（16384 slots）分片到不同主节点；既能水平扩容容量/性能，也高可用（主挂了从顶上）。

一句话总结：主从=读写分离；哨兵=主从的自动切主；集群=分片扩容 + 高可用。

**主从复制原理（面试口述版）**

- **全量同步**：从节点第一次连主，或主判断无法做增量时，主会生成当前数据的 RDB 快照发给从，从加载 RDB 后再接收这期间的写命令，完成“追上”。  
- **增量同步**：主会维护一个**复制积压缓冲区**（replication buffer），把写命令发给从；从断线重连后，若主 runid 一致且从的 offset 还在主缓冲区范围内，主只发“从 offset 之后”的写命令，否则走全量。  
- **runid + offset**：主有唯一 runid；主从各自维护复制偏移量 offset。从上报自己的 runid 和 offset，主据此判断做全量还是增量（psync）。  

**为什么是“全量 + 增量”而不是一直全量？**  
全量同步要主节点生成 RDB 并传给从，**耗时长、占带宽**，从要清空再加载，期间几乎不可用。若每次断线重连都全量，网络抖动或从重启就会反复全量，成本太高。所以设计成：**首次或主判断“断线太久、缓冲区里的写命令已经没了”时**才全量，建立一致基线；之后只要从的 offset 还在主的复制积压缓冲区里，就只发**增量写命令**，从只需重放这一段，**秒级恢复**。runid 用来识别“主是否换过”（主重启或故障转移后 runid 会变），变了就必须全量；offset 用来判断“从落后多少、还能不能只补增量”。这是典型的**用状态（runid+offset）换时间**的设计。

**面试一句话**：首次或断线太久走全量（RDB + 期间写命令）；能续上则走增量（主发 offset 之后的写命令）；靠 runid + offset 判断。

**哨兵机制解决了什么问题？**

- **解决的问题**：主节点挂了之后**自动发现 + 自动选主 + 通知客户端新主**，不需要人工切主、改配置。  
- **哨兵做什么**：① **监控**：持续检查主、从是否存活；② **通知**：主客观下线时通知管理员或其他组件；③ **自动故障转移**：主客观下线后，在从里选一个提升为新主（规则一般包括优先级、offset、runid）；④ **配置提供**：客户端通过哨兵问“当前主是谁”，拿到新主地址再连。  

**为什么需要独立的“哨兵”而不是让从节点自己选主？**  
若没有哨兵，主挂后要么人工改配置、改客户端连接地址，要么在客户端或应用层写“发现主”的逻辑，**耦合高、容易不一致**。哨兵是**独立进程、多数派部署**，只负责“谁活着、谁当主、当前主是谁”，不参与数据读写。这样**故障发现、选主、通知**都集中在一套组件里，客户端只需问哨兵“主是谁”即可，主变了也不用改业务代码。设计上是**把“高可用决策”从业务和存储里拆出来**，职责清晰，也便于做客观下线、选举一致性（多数哨兵同意才认为主挂、才执行切换）。
- **面试一句话**：哨兵 = 监控主从 + 主挂了自动选主 + 客户端通过哨兵发现新主，解决主从“主挂需人工切”的问题。

**集群模式如何实现数据分片？**

- **槽位 slot**：Redis 集群把 key 空间分成 **16384 个槽**（0～16383）。每个主节点负责一段连续槽（如 0～5460、5461～10922）。  
- **分片算法**：对 key 做 **CRC16(key) % 16384** 得到 slot，该 key 落在负责这个 slot 的节点上。  
- **访问流程**：客户端连任意节点发请求；若 key 不在本节点负责的 slot，节点返回 **MOVED slot 目标节点**，客户端再连目标节点重试；若正在迁移则可能返回 **ASK**，客户端先 ASKING 再请求目标节点。  

**为什么是 16384 个 slot 而不是更多？**  
CRC16 结果是 16 位，理论上有 65536 个值。若用 65536 个 slot，集群心跳里要带 8K 字节的 slot 位图（每 bit 表示一个 slot 是否在本节点），心跳包会很大；用 16384 只要 2K，**心跳和元数据更轻量**。同时 16384 已经足够做数据分片和扩容时的迁移粒度，再大收益有限、反而增加心跳和迁移成本。所以是**心跳大小、迁移粒度、哈希分布**之间的折中。

**为什么分 MOVED 和 ASK 两种重定向？**  
**MOVED** 表示“这个 slot **已经永久归属**目标节点”，客户端应**更新本地 slot 映射**，以后直接连目标节点。**ASK** 表示“这个 slot 正在**迁移中**，这次请求的 key 可能已经被迁到目标节点，但 slot 归属还没改”，客户端只**这一次**去目标节点查，不要改本地映射。这样设计是因为：迁移是渐进式的，一部分 key 已迁走、一部分还在源节点，若一律用 MOVED 客户端会过早切到目标节点，没迁走的 key 就查不到了；用 ASK 让客户端“按请求临时跳转”，迁移完成后才用 MOVED 更新映射，**保证迁移过程中读写都能落到正确的节点**。

**面试一句话**：16384 个 slot，CRC16(key)%16384 定槽，每节点负责一段槽；访问错节点会 MOVED/ASK 重定向。

### 4、Redis + RabbitMQ 菜单/门店信息预热与异步更新（面试口述版）

1）目标：用 Redis 扛高频读，把压力从 DB 挪到缓存。  
2）预热：系统启动或定时任务把菜单/门店热点数据提前写入 Redis，查询优先读 Redis。  
3）更新：数据变更时先写数据库，再发一条更新事件到 RabbitMQ。  
4）异步刷新：消费者收到消息后更新或删除对应 Redis key（常见删缓存，下次读回源重建），保证最终一致。  
5）补充：配合 TTL；必要时加互斥锁/逻辑过期；消息 MQ 做幂等（按业务 ID 去重）。

### 5、Redis 为什么快？（面试口述版）

- **内存读写**：数据主要存在内存里，读写都是纳秒/微秒级，避免磁盘 IO 的毫秒级延迟。持久化（RDB/AOF）是异步或按策略做的，不阻塞主流程，所以“快”主要体现在命令执行路径上。
- **单线程事件循环**：6.0 以前命令执行是单线程的，没有锁竞争、上下文切换和竞态，代码路径简单、可预测。网络层用 **IO 多路复用**（Linux 上多是 epoll）监听大量连接，一个线程就能处理成百上千个连接，把就绪的请求交给单线程顺序执行命令，所以既省 CPU 又避免多线程带来的复杂度。
- **高效数据结构**：每种类型都有针对性的底层实现（如 SDS、跳表、压缩列表、整数集合等），而不是简单用通用哈希表。String 的 get/set、Hash 的 hget/hset、Set 的 add/ismember、ZSet 的 zadd/zrange 等常见操作时间复杂度低，很多是 O(1) 或 O(log N)，减少单次命令的 CPU 开销。
- **网络与协议**：基于 TCP、自定义的 **RESP 协议**简单、解析快；支持 **Pipeline** 批量发命令，客户端可以连续发多条命令再一次性读回结果，把多次 RTT 合并成一次，在高延迟或批量操作场景下明显降低延迟。
- **补充**：Redis 6.0 起支持多线程，但多线程只用于**网络 IO**（读写 socket），命令执行仍是单线程，所以“单线程执行命令、无锁”的优势还在，只是网络处理能力更强了。

**为什么敢用单线程？为什么瓶颈往往不在 CPU？**  
Redis 的数据在**内存**里，单次命令的 CPU 开销主要是哈希查找、简单计算、序列化，**量级是微秒级**；真正的耗时往往在**网络 RTT、大 value 的序列化/拷贝**。所以“一个线程顺序执行命令”不会成为瓶颈，反而避免了多线程的锁、上下文切换、竞态，**实现简单、行为可预期、bug 少**。设计取舍是：用单线程换**简单和稳定**，用 IO 多路复用换**高并发连接**，用内存和数据结构换**单次命令速度**。这也是为什么大 key、慢命令在生产环境特别敏感——单线程下一条慢命令会拖住后面所有请求。

**Redis 6.0 为什么引入多线程？一定比单线程快吗？（面试口述版）**

- **为什么引入**：在高并发、大 value 场景下，**网络 IO**（从 socket 读请求、把结果写回 socket）成为瓶颈，单线程既要执行命令又要处理大量网络读写，CPU 可能还没打满，但延迟已经上去。多线程把**网络读写**交给多个 IO 线程并行做，命令解析和命令执行仍由主线程单线程做，这样在不破坏“单线程执行、无锁”的前提下，提升网络吞吐、降低延迟。  
- **多线程用在哪**：只用于**网络 IO**（读请求进内核缓冲区、写响应出内核缓冲区），**命令执行仍是主线程单线程**，所以 Lua、事务、单条命令的原子性都不受影响。  
- **一定比单线程快吗**：**不一定**。在 **CPU 不是瓶颈、value 较小、连接数不高** 时，多线程带来的收益有限，甚至可能因线程切换、锁竞争略慢；在 **高并发、大 value、网络 IO 明显成为瓶颈** 时，多线程能明显提升吞吐、降低 P99 延迟。所以 6.0 多线程是“按需开启、主要解决网络 IO 瓶颈”，不是无条件更快。

**为什么 6.0 只把“网络 IO”多线程化，而不把“命令执行”也多线程？**  
命令执行多线程会动到**全局状态**：同一个 DB、同一个 key、Lua 脚本、事务 MULTI/EXEC 都依赖“顺序执行”的语义。若多线程并发执行命令，就要引入大量锁和并发控制，**复杂度高、易出 bug**，而且 Redis 的瓶颈往往不在 CPU 算力而在网络和内存访问。把**网络读写**拆出去用多线程做，既不破坏“命令单线程、天然原子”的模型，又能把“读请求进内核、写响应出内核”这种 IO 密集部分并行化，**改动小、收益明确**。所以这是“在不动核心模型的前提下，只把瓶颈部分并行化”的保守设计。

**面试一句话**：6.0 多线程只做网络 IO，命令执行仍单线程；高并发大 value 时更能体现优势，小 value、低并发时不一定更快。

### 6、Redis 支持哪些数据类型？特点和适用场景？（面试口述版）

| 类型 | 特点 | 适用场景 |
|------|------|----------|
| **String** | 简单 key-value，可过期，可存二进制；最基础 | 缓存对象（JSON）、计数器（incr）、分布式锁 value、Session |
| **Hash** | 字段级读写，不必整存整取；适合“对象多字段” | 存对象字段（user:1 的 name、age），部分字段更新 |
| **List** | 有序、可重复，头尾操作 O(1)；支持阻塞取 | 消息队列（BLPOP/BRPOP）、时间线、最新 N 条 |
| **Set** | 无序、不重复，交并差集 O(1) | 去重、共同好友、抽奖、标签 |
| **ZSet** | 有序、按 score 排序，范围查询 | 排行榜、延迟队列（score=时间戳） |
| **HyperLogLog** | 近似去重计数，固定空间、有误差 | UV/独立 IP 等“大概有多少不同”的统计 |
| **Bitmap** | 按 bit 存，省空间 | 签到、活跃标记、布尔状态 |
| **Geo** | 经纬度 + 距离/范围计算 | 附近的人/店铺、范围查询 |

**为什么 Redis 要提供这么多数据结构，而不是只做 key-value？**  
因为**业务场景差异很大**：有的只要“有没有、是多少”（String、计数器）；有的要“按字段更新、别整存整取”（Hash）；有的要“顺序、阻塞取”（List 做队列）；有的要“去重、交并集”（Set）；有的要“按分数排序、范围查”（ZSet）。若只有 String，就要在应用层自己拼 JSON、自己解析、自己实现排序和去重，**既占带宽又占 CPU**。Redis 在服务端提供这些结构，**语义清晰、操作原子、实现高效**（如 ZSet 用跳表+哈希，范围查和单点查都快），应用层只发一条命令就行。设计哲学是：**把“常用抽象”做成原语**，减少网络往返和客户端复杂度，同时便于做内存和算法层面的优化。所以选型时要按“是否要顺序、是否去重、是否要范围、是否要阻塞”来选，而不是什么都塞 String。

**面试一句话**：String/Hash/List/Set/ZSet 最常用；HyperLogLog/Bitmap/Geo 做统计和地理；按“是否要顺序、是否去重、是否要范围”选类型。

**项目中一般用 Redis 做什么？为什么选 Redis 而不是其他缓存？（面试口述版）**

- **一般用来做什么**：① **缓存**：热点数据（如商品详情、菜单、配置）减轻 DB 压力；② **分布式锁**：多实例间互斥；③ **限流**：用计数/滑动窗口/令牌桶；④ **简单消息队列**：List 的 BLPOP、或 Stream；⑤ **排行榜/去重/签到**：ZSet、Set、Bitmap；⑥ **会话/临时状态**：如登录态、验证码，设 TTL 自动过期。  
- **为什么选 Redis 而不是别的**：① **内存存储**，读写快、延迟低；② **丰富数据结构**，不只 key-value，还有 Hash/List/Set/ZSet 等，业务建模方便；③ **支持持久化**（RDB/AOF）和主从/集群，可接受一定程度的可靠性与高可用；④ **单线程执行命令**（6.0 前）无锁、可预期，易运维；⑤ **生态成熟**，客户端、集群、哨兵、分布式锁（如 Redisson）等方案多。和 Memcached 比，Redis 支持更多类型和持久化；和本地缓存比，Redis 是进程间/机器间共享、可做分布式锁和限流。  
- **面试一句话**：项目里常用 Redis 做缓存、分布式锁、限流、简单队列、排行榜等；选它是因为内存快、数据结构多、有持久化和高可用、生态好。

### 7、缓存和数据库如何保证一致性？（面试口述版）

- 先说结论：**最常用的是 Cache Aside（旁路缓存）策略，写时“先更新 DB，再删缓存”**。

- 读流程（Cache Aside）：  
  1）先查缓存，命中就直接返回；  
  2）没命中再查数据库，把结果回填到缓存里；  
  3）以后再读都先走缓存。

- 写流程（关键一句话）：  
  - **先更新数据库，再删除缓存对应的 key，而不是先改缓存。**  
  - 原因：在高并发下，如果你是“更新缓存”，很可能把旧值写回缓存；而“删缓存”可以保证后续读一定会回源 DB、重新塞入最新值。

**为什么写时是“删缓存”而不是“更新缓存”？**  
在高并发下，若采用“先更新 DB，再更新缓存”，可能出现：请求 A 更新 DB 为 1，请求 B 更新 DB 为 2，但 B 的“写缓存”先于 A 执行，最后缓存里是 1（旧值），这就是**写竞争导致脏读**。若采用“先删缓存，再更新 DB”，下次读会 miss，回源 DB 再写回缓存，拿到的一定是 DB 的最新值；唯一的小概率问题是“读请求在删缓存和 DB 更新完成之间进来，把旧值又写回缓存”，所以才有**延迟双删**（写完 DB 后再删一次缓存）来缩小这个窗口。**删缓存**的语义是“作废这份副本，以 DB 为准”，比“试图把缓存改成新值”更简单、更不容易在并发下出错，所以业界普遍采用“先更新 DB，再删缓存”而不是“更新缓存”。

- 常见细节与补偿：  
  - 删除缓存失败：需要重试机制（如通过 MQ 记录变更、后台补偿任务重删 key），避免长期读到脏数据。  
  - 热点 key：可以配合互斥锁/逻辑过期，避免缓存失效瞬间大量请求打爆 DB（缓存击穿问题）。  
  - 更强一致性需求：可以采用“延迟双删”策略：**删缓存 → 写 DB → 延迟一小段时间再删一次缓存**，降低并发时短暂不一致的窗口。  
- **项目中如何做（面试可答）**：我们采用 **Cache Aside**，写时先更新 DB 再删缓存；删缓存失败时通过 MQ 发变更事件或后台补偿任务重删 key；热点 key 配合互斥锁或逻辑过期防击穿。**设计上的取舍**是：不强求“读一定是最新写”的强一致，接受**短暂不一致窗口**，换的是**高并发下的性能和简单实现**；若业务真的要求“写后读必见”，就要考虑读主库、或更重的方案（如分布式事务），成本和复杂度会高很多，所以大多数场景用“最终一致 + 补偿”就够。

### 8、Redis 如何保证原子性？（面试口述版）

- 单条命令原子：Redis 单线程执行命令，单条命令天然原子。  
- 多条命令原子：  
  - Lua 脚本：EVAL 执行，脚本整体原子。  
  - 事务 MULTI/EXEC：保证顺序执行，但不回滚；WATCH 可做乐观锁。

### 9、Redis 分布式锁怎么实现？（面试口述版）

- 最核心思路：**用 Redis 里的一条 key，当成“谁拿到锁”的标记**，结合过期时间 + 唯一标识，来保证“互斥 + 不误删 + 不死锁”。

- **加锁（拿锁）**  
  - 命令：SET lock_key requestId NX PX ttl  
    - NX：key 不存在才设置，保证互斥。  
    - PX ttl：设置过期时间，防止服务挂了后锁永远不释放。  
    - requestId：用 UUID 或线程/实例唯一标识，区分“这把锁是谁加的”。  
  - 返回 OK 代表加锁成功；返回 null 说明已经有人持有锁，可以重试或直接失败。

- **解锁（释放锁）**  
  - 不能直接 DEL lock_key，否则可能把别人的锁删掉（比如自己的锁已经过期，别的实例又加上了新锁）。  
  - 正确做法：用 Lua 脚本先比对 value，再决定删不删（保证原子性）：  
    - if redis.call('get', key) == requestId then redis.call('del', key) end  
  - 这样只有“锁还是我加的”那个人，才能成功释放锁。

- **注意点/高级用法**  
  - 续期（看门狗）：如果业务执行时间可能超过 ttl，可以配合定时续期（比如 Redisson 的看门狗机制），避免锁被提前自动释放。  
  - 业务超时控制：即使有续期，也要在应用层设置最大执行时间，防止锁长期占用。  
  - 主从复制延迟问题：在主从切换、复制延迟严重的情况下，简单的单节点锁会有少量极端不安全场景；安全性要求非常高时，可以考虑 RedLock 或直接用 ZooKeeper/etcd 这种强一致性组件实现分布式锁。

**为什么是 NX + 过期时间 + requestId？设计上分别解决什么？**  
**NX**（key 不存在才 set）保证**互斥**：只有一个客户端能 set 成功，等价于“抢到锁”。**过期时间（PX/EX）**保证**不死锁**：若持锁方宕机或异常没释放，key 到点自动过期，别人还能再抢，否则锁会永远占着。**requestId（value 存唯一标识）**保证**不误删**：释放时只有“当前持有者”才能删 key；若不用 value 校验，可能出现：A 的锁过期了，B 抢到锁，A 事后又执行 DEL，把 B 的锁删了。所以“加锁带 requestId、解锁先比对再 DEL（且用 Lua 保证原子）”是**互斥、自动释放、安全释放**三个目标一起满足的设计。生产里还会加续期（看门狗），防止业务执行时间超过 TTL 导致锁被提前释放。

### 9.1、如何使用 Redis 实现一个简单的消息队列？（面试口述版）

- **List + BLPOP/BRPOP（阻塞队列）**：生产者用 **LPUSH** 往列表头塞消息，消费者用 **BRPOP key 0** 从列表尾阻塞取（0 表示一直等）。多消费者时每个都会阻塞在同一个 key 上，谁先到谁取走一条，实现**多消费者竞争消费**。优点：实现简单、无需额外组件。缺点：消息只被一个消费者取走，无广播；**没有 ack**，消费者取走后若崩溃，消息就丢了；没有重复消费控制；列表无限增长时需自己控制或用 LTRIM。  
- **Stream（5.0+）**：支持消费组、每条消息有 id、可 ack、可 pending 列表，更接近“正规”消息队列。用 **XADD** 生产，**XREADGROUP** 消费，**XACK** 确认。适合需要“至少一次”、消费组、可回溯的场景，但实现和运维比 List 复杂。  
- **面试可答**：简单场景用 List + BLPOP，实现快；要 ack、消费组、少丢消息用 Stream。都要注意 Redis 持久化（AOF）和内存，消息堆积时要有上限或 TTL；强一致性、不丢消息还是用 RabbitMQ/Kafka 等更合适。

**为什么用 List 做队列时通常选 BLPOP 而不是轮询 LPOP？**  
轮询 LPOP 会**空转**：队列空时一直发 LPOP，占连接、占 Redis、占网络，延迟还高。BLPOP 是**阻塞式**：队列空时连接挂起，有数据时服务端主动推送，**既不空转又能及时唤醒**，单连接即可实现“有活就干、没活就等”。所以 List + BLPOP 的设计是“用阻塞换低延迟和低资源占用”。代价是：没有 ack，消费者取走后若崩溃消息就丢；没有消费组，一条消息只能被一个消费者取走。若业务能接受“至多一次、竞争消费”，用 List 就够；若要“至少一次、消费组、回溯”，就上 Stream 或专业 MQ。

**面试一句话**：简单队列用 List + BLPOP；要 ack/消费组用 Stream；真不丢、强一致用专业 MQ。

### 10、大 Key 与热 Key（面试口述版）

**1）大 Key 是什么？有什么问题？怎么处理？**

- **是什么**：某个 key 的 value 特别大（比如一个 String 几 MB，或者一个 Hash/List/ZSet 里塞了非常多元素），占内存、占带宽。  
- **问题**：Redis 单线程执行命令，一条大 key 的 `GET` / `HGETALL` 要序列化和传输一大块数据，这段时间里其他命令都要排队，很容易把后面的请求“堵”住；一次传几 MB 的 value 也会占据大量带宽；主从复制、集群迁移时，大 key 要整块搬，拖慢复制和扩容；如果大 key 集中在一台机器上，还会造成内存和流量分布不均。  
- **处理思路（拆、压、换、删）**：  
  - **拆**：把一个大 key 拆成多个小 key，让单次读写只碰一小块数据。比如大 List 改成 `list:0`、`list:1`… 分段；或按时间拆 `order:2026-03`、`order:2026-04`；或按范围拆 `user:0-9999`、`user:10000-19999`。这样既降低单次命令耗时，也方便迁移和删除。  
  - **压**：value 是 JSON/文本时可以 gzip/snappy 压缩后再存，减少网络传输和内存占用；但要注意压缩/解压有 CPU 成本，通常只对“体积大、读写不那么频繁”的 key 做。  
  - **换**：别把所有内容塞进一个超大的 String/List。能做“字段级更新/按需取字段”的就用 Hash；大 Hash 也可以按字段分桶（如按 userId%N 分多个 Hash）；或者把大对象拆成多个 key（避免一次 `GET` 把全部内容拖出来）。原则是：**让查询“只取需要的部分”，减少一次性回包体积。**  
  - **删**：大 key 删的时候不要直接 `DEL`（会在主线程里释放大量内存，容易卡住）。优先用 **`UNLINK` 异步删除**；Hash/List/ZSet 这种也可以用 `SCAN` 找 key，或用 `HSCAN/SSCAN/ZSCAN` 分批删字段/元素，做到“分批处理、每批很小”，避免长时间阻塞。

**2）热 Key 是什么？有什么问题？怎么处理？**

- **是什么**：访问量特别集中的 key，比如爆款商品详情、热门榜单，所有请求都在打这一两个 key。  
- **问题**：所有请求集中打到同一个 key/同一个分片上，这台节点的 CPU、带宽、连接数会率先打满，形成热点倾斜，严重时拖垮单节点，影响整体服务。  
- **处理思路**：  
  - **本地缓存**：在应用里对热 key 加一层短期本地缓存（如 Caffeine），把“1 秒内被反复读很多次”的请求挡在应用内存里。常见做法是给本地缓存一个很短的 TTL（比如 100ms～1s），并设置最大容量，避免把 JVM 内存撑爆；同时要考虑多实例：每台机器都有一份本地缓存，所以能把热点读压力按实例数分散掉。  
  - **多副本**：把同一逻辑 key 做成多个物理 key（如 `key_1`、`key_2`、`key_3`…），**写的时候同步写多份**，读的时候**随机挑一个**或按 hash 选一个，这样请求会落到不同 slot/不同节点，避免单点打满。注意点：多副本会增加写放大和一致性复杂度，所以通常只对“极少数超级热点 key”使用。  
  - **读分散**：在主从场景下可以把读请求尽量打到从节点（或在集群里让读分布到不同分片），把压力从单点扩散。要注意主从延迟：对“写后立刻读必须是最新”的场景仍需要读主库/主节点或做读写黏连，否则可能读到旧值。  
  - **限流/降级**：极端热点时，哪怕做了缓存和读分散也可能扛不住，需要在入口做保护：对该 key 的读做限流（超了直接返回“稍后重试”或兜底值），或降级为“返回旧值/默认值/不返回该模块”。目的不是“所有请求都成功”，而是“系统别被打挂、核心链路能活下来”。

**面试一句话**：大 key = value 过大，容易堵住单线程、拖慢复制，拆压换删+UNLINK；热 key = 访问集中在少数 key/节点，容易打爆单点，用本地缓存、多副本、读分散 + 限流来削峰。

### 11、Redis 怎么限流？有哪些常见算法？（面试口述版）

**思路：** 把“当前请求量/时间窗口”放 Redis，请求前查是否超限，超就拒。

**四种算法（好背版）：**

| 算法 | 一句话 | 缺点 |
|------|--------|------|
| **固定窗口** | 按 1 分钟/1 小时一个窗口计数，INCR + EXPIRE，超了就限。 | 边界突刺（两窗口交界处可能 2 倍流量）。 |
| **滑动窗口** | 按“当前往前推一个窗口”内的请求数算，用 ZSet/List 存时间戳，去掉窗口外的再数。 | 占内存、要删过期。 |
| **漏桶** | 请求进桶，桶按固定速率“漏水”处理，桶满就拒。 | 输出匀速，不能突发。 |
| **令牌桶** | 桶里按速率加令牌，请求来就扣一个，扣到就过，扣不到就限。 | 实现稍复杂，但能突发，最常用。 |

**背诵口诀：** 固定窗口——按窗 INCR，简单但有突刺；滑动窗口——按“当前往前”数，更准但费内存；漏桶——进水匀速漏，桶满就拒；令牌桶——按速率发牌，有牌就过，**最常用**。  

**分布式限流为什么用 Lua？**  
限流逻辑一般是“先读当前计数/令牌，再判断是否超限，再决定是否 +1 或扣减”，这是**多步操作**。若用多条 Redis 命令，在高并发下会出现：多个请求同时读到“还剩 1 个令牌”，都判断通过，都去扣减，实际就超限了。**Lua 脚本在 Redis 里是原子执行**的，读-判断-写 在脚本里一次完成，中间不会被其他请求插队，所以能准确限流。用 Redis 做分布式限流时，把“查计数、比较、INCR/扣减”写进一段 Lua，是**用原子性换正确性**的常见做法。
